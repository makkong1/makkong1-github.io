# 리팩토링 세대 지도 (Generation Map) — 2026-07

> **왜 이 문서가 필요한가**
> Petory는 같은 문제를 **리팩토링 위에 리팩토링**으로 여러 번 고쳤다. 그래서 기존 문서·포트폴리오·핵심성과 수치가 **중간 세대**를 붙잡고 있는데, 실제 코드는 **최신 세대**로 넘어가 있어서 설명할 때 어지럽다.
>
> 이 문서의 원칙은 하나다: **현재 코드 = 유일 기준.** 각 사례마다
>
> - **[현재 코드가 실제 하는 것]** = 지금 면접에서 말할 것
> - **[옛 세대 → 무시]** = 문서에만 남은 이전 구현·수치 (참고만, 현재형으로 말하지 말 것)
> - **[면접 한 줄]** = 현재 기준으로 방어 가능한 한 문장
>
> 확인 방법: 실제 소스 코드 + git 커밋 + 마이그레이션 파일. (작성 2026-07-06)

**결론: 실제로 "리팩토링 위에 리팩토링"으로 엉킨(다세대) 곳은 검색(위치·모임) 뿐이다.** 나머지는 문서 수치가 현재 코드와 맞으므로 그대로 써도 된다. 위치·모임 검색만 "현재는 공간 인덱스"로 말하고 옛 bounding box 수치는 버리면 어지러움이 사라진다.

⚠️ **단, 검색은 도메인마다 다르다.** 위치·모임은 공간 인덱스로 넘어갔지만 **케어(Care) 근처 검색은 아직 SQL Haversine + bounding box가 현재(그리고 유일) 세대**다 — 위 표 Gen 0의 "Java 인메모리 Haversine"과 혼동하지 말 것(케어는 DB에서 SQL Haversine을 돌린다). 그래서 검색을 "전부 공간 인덱스"로 뭉뚱그리면 케어에서 어긋난다. 자세한 건 §3 끝의 케어 노트 참고.

## 1. N+1 조회 최적화 (Board / MissingPet / Chat)

### 현재 코드가 실제 하는 것

세 도메인 모두 **동일한 패턴**으로 최종 정착:

1. 목록을 한 번에 조회 → ID 리스트 추출
2. 연관 데이터(첨부파일 수, 댓글 수, 반응 수, 참여자, 최신 메시지 등)를 `IN (:ids)` **배치 쿼리**로 한 방에 조회
3. 결과를 `Map<ID, 값>`**으로 만들어 메모리에서 매핑** → 루프 안에서 추가 쿼리 없음

- **Board**: `BoardPopularityService.fetchBoardCountsInParallel` — 좋아요/댓글/조회수 3개 배치 쿼리를 `CompletableFuture`로 **병렬 실행** 후 `Map<Long, BoardCounts>`로 통합. (`board/service/BoardPopularityService.java:192`)
- **MissingPet**: `attachmentFileService.getAttachmentsBatch` + `getCommentCountsBatch` → `Map<Long, ...>` 매핑. (`board/service/MissingPetBoardService.java:116`)
- **Chat**: `getMyConversations` — 참여자·전체참여자·최신메시지를 각각 `findParticipantsByConversationIdxs...` / `findLatestMessagesByConversationIdxs` 배치로 조회 후 Map 매핑. (`chat/service/ConversationService.java:72`)

### 옛 세대 → 무시

- **없음.** 이 세 사례는 배치+Map이 최종 세대이고, 수치도 현재 코드와 일치한다.
- 단 **Chat 라벨 주의**: 옛 문서가 "로그인 시 N+1 21→4"라고 적었는데 이건 **로그인/인증 코드가 아니라** `getMyConversations`**(로그인 직후 프론트가 부르는 채팅방 목록) 쿼리**다. 라벨만 고치면 됨.

### 유효 수치 (현재 코드 기준으로 말해도 됨)

- Board 목록: **301개 → 3개 (약 99%↓)**
- MissingPet 목록: **105개 → 3개**
- Chat 채팅방 목록(`getMyConversations`): **21개 → 4개 (약 81%↓)**

### 면접 한 줄

> "목록 조회에서 항목마다 연관 데이터를 개별 쿼리하던 걸, ID를 모아 `IN` 절 배치 쿼리 한 번으로 가져오고 `Map`으로 매핑하도록 바꿔 N+1을 제거했습니다. Board는 3개 카운트 쿼리를 `CompletableFuture`로 병렬화까지 했습니다."

## 2. 동시성 제어 (원자적 UPDATE / 비관적 락)

### 현재 코드가 실제 하는 것

두 가지 전략을 **문제 성격에 따라 나눠** 쓴다.

**(a) 조건부 원자적 UPDATE** — 카운터류. 락 없이 DB가 조건+증감을 원자적으로 처리.

- 모임 참여 인원 증가: `incrementParticipantsIfAvailable` — `SET current = current + 1 WHERE current < max AND status = RECRUITING`. 정원 초과를 **UPDATE 조건**으로 막고, 갱신 행 수(`int`)로 성공/실패 판정. (`meetup/repository/SpringDataJpaMeetupRepository.java:116`)
- 경고 횟수 증가: `incrementWarningCount` — `SET warningCount = warningCount + 1`. (`user/repository/SpringDataJpaUsersRepository.java:87`)

**(b) 비관적 락 (**`PESSIMISTIC_WRITE`**)** — 잔액·에스크로처럼 읽고→검증하고→쓰는 흐름.

- 펫코인 에스크로: `findByCareRequestForUpdate`, `findByIdForUpdate`. (`payment/repository/SpringDataJpaPetCoinEscrowRepository.java:33`)
- 사용자 코인 차감: `findByIdForUpdate`. (`user/repository/SpringDataJpaUsersRepository.java:96`)
- 모임 단건 락 조회: `findByIdWithLock`. (`meetup/repository/SpringDataJpaMeetupRepository.java:112`)

### 옛 세대 → 무시 (단, 흔적 하나 존재)

- 참여 **취소** 로직에 내부 세대 흔적이 남아 있다. 코드 주석: `// [FIX] 참가 취소 시 원자적 감소 — 기존 read-modify-write(Math.max)는 동시 취소 시 카운트 불일치 위험`. 즉 **예전엔** `Math.max(0, current-1)`**로 읽고-계산-저장**했다가, 지금은 `decrementParticipantsIfPositive`(`SET current = current - 1 WHERE current > 0`) **원자적 감소**로 바뀜. (`SpringDataJpaMeetupRepository.java:129`)
- 면접에선 이 흔적을 **오히려 좋은 소재**로 쓸 수 있다: "증가만 원자화했다가, 동시 취소에서도 같은 문제가 생겨 감소도 원자적 UPDATE로 통일했습니다."

### 면접 한 줄

> "단순 카운터(모임 인원·경고 횟수)는 락 대신 `WHERE 조건 + 증감`을 한 문장에 담은 원자적 UPDATE로 처리하고, 갱신 행 수로 성공을 판정합니다. 반면 잔액·에스크로처럼 _읽고 검증 후 써야 하는 건_ `PESSIMISTIC_WRITE` _비관적 락(_`findByIdForUpdate`)으로 Race Condition을 막습니다."

## 3. 검색 — 위치(Location) / 모임(Meetup) ★ 유일한 다세대

### 현재 코드가 실제 하는 것

둘 다 **공간 인덱스 기반**으로 최종 정착.

**반경 검색** (`Location.findByRadius` / `Meetup.findNearbyMeetupIds`):

1. `ST_Within(location/geo_point, ST_GeomFromText('POLYGON(...)', 4326))` — 위경도 ± 반경으로 만든 bounding polygon 안 후보를 **공간 인덱스**로 좁힘
2. `ST_Distance_Sphere(..., point) <= :radius * 1000` — 미터 단위 **정밀 반경 필터**
3. `ORDER BY 거리 LIMIT` — 가까운 순

- Location: `SpringDataJpaLocationServiceRepository.java:72`
- Meetup: `SpringDataJpaMeetupRepository.java` `findNearbyMeetupIds`, 마이그레이션 `meetup-spatial-location.sql`, 커밋 `6e0a29c7`

**키워드 검색**:

- Location: `MATCH(name, description, category1, category2, category3) AGAINST(...)` **FULLTEXT** (`:21`, `:47`)
- Meetup: `MATCH(title, description) AGAINST(...)` FULLTEXT + `location LIKE` (`findAllForAdminWithKeyword`), 마이그레이션 `meetup-fulltext-ngram-index.sql`, 커밋 `f5f45b78` ("LIKE 양쪽 와일드카드 풀스캔 → FULLTEXT ngram")

**지역명 검색** (Location): `sigungu = :sigungu` + `USE INDEX (idx_locationservice_sigungu_deleted_rating)` 강제 인덱스. (`:107`)

### 옛 세대 → 무시 (여기가 어지러움의 근원)

| 세대           | 옛 구현                                                                  | 옛 수치 (현재 코드엔 안 붙음)              |
| -------------- | ------------------------------------------------------------------------ | ------------------------------------------ |
| Gen 0          | 전체 로드 후 Java Haversine **인메모리** 필터                            | —                                          |
| Gen 1          | `lat/lng BETWEEN` bounding box + **B-tree** 인덱스 `idx_meetup_location` | 모임 스캔 **2,958→117행, 486→273ms, 96%↓** |
| Gen 1 (키워드) | `LIKE '%kw%'` 양쪽 와일드카드 풀스캔                                     | —                                          |

- 핵심성과 #4 "인메모리 → DB Bounding Box"와 `nearby-meetups/*` 문서, **96% / 2958→117** 수치는 전부 **Gen 1** 측정치다.
- 현재는 "bounding box 개념"은 남았지만 **구현이 B-tree BETWEEN → 공간 인덱스** `ST_Within`**으로 바뀌었고** `ST_Distance_Sphere` **정밀 필터가 추가**됐다. 그래서 옛 수치를 현재형으로 말하면 코드와 안 맞는다.

### 면접 한 줄 (현재 기준)

> **반경**: "전체 로드 후 인메모리로 거리 계산하던 걸, 공간 컬럼(`geo_point`/`location`, SRID 4326)에 공간 인덱스를 걸고 `ST_Within`으로 bounding polygon 후보를 좁힌 뒤 `ST_Distance_Sphere`로 정밀 반경 필터 + 거리순 LIMIT 하도록 바꿨습니다."
> **키워드**: "`LIKE '%kw%'` 풀스캔을 `FULLTEXT`(ngram) 인덱스 `MATCH ... AGAINST`로 교체했고, 지역명은 `USE INDEX`로 커버링 인덱스를 강제했습니다."
> (수치를 꼭 대야 하면: "초기 개선 단계에선 bounding box만으로도 스캔을 2,958→117행으로 줄였고, 이후 공간 인덱스로 재구현했습니다"처럼 **세대를 명시**해서 말할 것.)

### ⚠️ 케어(Care)는 예외 — Haversine이 현재 세대 (미이전)

위 위치·모임과 달리 **케어 요청 근처 검색은 공간 인덱스로 이전되지 않았다.** `SpringDataJpaCareRequestRepository.findNearbyCareRequests`는 현재도 **네이티브 SQL Haversine**(`6371 * acos(cos·cos·cos + sin·sin)`) + `lat/lng BETWEEN` bounding box로 반경을 계산한다. 위 표 Gen 0의 "Java **인메모리** Haversine"과는 다르다 — 케어는 **DB에서 SQL Haversine을 돌리는 현재(그리고 유일) 세대**다. (`care/repository/SpringDataJpaCareRequestRepository.java` `findNearbyCareRequests`)

- **정직한 프레이밍**: 검색을 "전부 공간 인덱스"로 말하면 케어에서 어긋난다. → "위치·모임 주변 검색은 공간 인덱스(`ST_Within` + `ST_Distance_Sphere`)로, 케어 요청 검색은 Haversine + bounding box로 — **도메인 특성에 맞춰 검색 전략을 선택 적용**했습니다."
- **개선 여지(면접에서 물으면)**: "케어도 `geo_point` + 공간 인덱스로 통일할 수 있으나 아직 미이전 상태입니다. 위치·모임을 먼저 이전하며 검증한 패턴이라 케어 적용은 남은 과제입니다."

## 4. 보안 / 인가

### 현재 코드가 실제 하는 것

- **메서드 레벨 인가**: `@PreAuthorize` + Role 계층 (USER < SERVICE_PROVIDER < ADMIN < MASTER).
- **채팅방 접근 통제**: `ChatMessageService.requireActiveParticipant(conversationIdx, userId)` — 메시지 조회/전송/읽음처리 등 진입점마다 **활성 참여자인지 재검증**. (`chat/service/ChatMessageService.java:44`, 호출부 116/149/166/198)
- **인증**: JWT Access(기본 15분) + Refresh(1일, DB 저장) → `JwtAuthenticationFilter` → `SecurityConfig`.

### 옛 세대 → 무시

- 없음. 현재 구조가 최신이며 문서와 일치.

### 면접 한 줄

> "인가는 `@PreAuthorize` 메서드 레벨로 하고, 특히 채팅은 URL만 알면 접근 가능한 IDOR를 막기 위해 각 메시지 API 진입점에서 `requireActiveParticipant`로 '이 방의 활성 참여자인가'를 매번 검증합니다."

## 5. 순환의존 제거 (도메인 의존성 리팩토링)

### 현재 코드가 실제 하는 것

- 도메인을 **3계층**으로 정리: A(집계: admin/activity/profile/statistics) ← B(핵심 도메인) ← C(공용 leaf: file/notification/common).
- 결합 제거는 **이벤트 기반**: `ApplicationEventPublisher`로 이벤트 발행 + `@TransactionalEventListener(AFTER_COMMIT)` + `@Transactional(REQUIRES_NEW)` 리스너로 수신.
  - 예: 에스크로 지급 시 `statisticsService.recordPayment(...)` 직접 호출 → `PaymentRecordedEvent` 발행으로 교체. `payment → statistics` 의존성 제거 + **통계 실패가 코인 지급을 롤백하던 잠재 버그**까지 해소.
- 결과: `user`가 leaf가 되고, `statistics`가 순수 sink가 되며, 3-노드 순환(payment↔statistics 포함)이 끊김. (근거: `docs/analysis/dependency-graph/before-*` vs `after-step1-4-*`)

### 옛 세대 → 무시

- 없음. 이게 **가장 최근** 리팩토링이라 코드 = 문서 = 최신. (설계·실행 로그: `docs/analysis/domain-dependency-refactoring-2026-07.md`)
- 남은(의도적으로 수용한) 순환: `care ↔ payment`(에스크로 집계), `chat ↔ meetup`(Step 5 DIP 보류).

### 면접 한 줄

> "도메인 간 양방향 의존과 순환을 없애려고, 결제→통계 같은 부가 처리를 직접 호출 대신 `ApplicationEvent`로 발행하고 `@TransactionalEventListener(AFTER_COMMIT) + REQUIRES_NEW`로 비동기·독립 트랜잭션에서 처리했습니다. 덕분에 결합이 끊기고, 통계 실패가 본 거래를 롤백하던 문제도 사라졌습니다."

## 2. 패스 A — 검사행순 (풀스캔 / 비싼 COUNT)

| 쿼리 | 횟수 | 반환 | 검사 | 회당검사 | 인덱스미사용 | 총ms |
|---|---|---|---|---|---|---|
| 🔴 `SELECT COUNT(b1_0.idx) FROM board b1_0 ...` | 3 | 3 | **180,003** | **60,001** | 3 | 74 |
| 🔴 `SELECT b1_0.idx, b1_0.title, ...` (깊은 페이지) | 1 | **0** | **100,000** | 100,000 | 0 | **115** |
| 🟡 `SELECT COUNT(b1_0.idx) ...` (카테고리) | 1 | 1 | 20,000 | 20,000 | 0 | 21 |
| 🟡 `SELECT COUNT(*) FROM board b INNER JOIN users u ...` (검색) | 1 | 1 | 16,666 | 16,666 | 0 | 26 |
| 🟡 `SELECT b.*, MATCH(b.title, b.content) ...` (FULLTEXT) | 1 | 20 | 8,373 | 8,373 | 0 | 34 |

문제는 **그 옆에 붙어 나가는 COUNT** 와 **깊은 페이지** 다.

## 3. 패스 B — 호출횟수순 (N+1)

| 쿼리 | 횟수 | 회당검사 | 판정 |
|---|---|---|---|
| `SELECT u1_0.idx, u1_0.birth_date, ...` (users) | 9 | 1 | ✅ **N+1 아님** — curl 9회 = 인증 필터의 사용자 조회 9회. 요청당 1회다 |
| `SELECT af1_0...` (첨부 배치) | 6 | 0 | ✅ 목록 엔드포인트당 1회 (배치) |
| `SELECT br1_0...` (리액션 배치) | 5 | 29 | ✅ 목록 엔드포인트당 1회 (배치) |

> ## ✅ **board 도메인에 N+1 은 없다.**
> 목록 20건을 조회해도 리액션·첨부 쿼리는 **각 1회**다 (`IN (...)` 배치).
> projection + 배치 사후주입이 의도대로 동작하고 있다.
>
> `u1_0` 9회는 **JWT 인증 필터**가 요청마다 사용자를 조회하는 것이다. 호출 9회 = 조회 9회로 정확히 일치한다.
> 목록 크기(20)에 비례하지 않으므로 N+1 이 아니다. **패스 B 의 분모를 적어둔 덕에 즉시 판별됐다.**

## 5. 🔴 발견 1 (신규) — 깊은 페이지에서 목록 SELECT 가 폭발한다

격리 측정으로 확정했다 (해당 엔드포인트 **단독** 호출):

| 호출 | 목록 SELECT 반환 | 목록 SELECT 검사 | 시간 |
|---|---|---|---|
| `page=0` | 20행 | **40행** | 1ms |
| `page=2500` | **0행** | **100,000행** | **129ms** |

**EXPLAIN ANALYZE:**

```
-> Limit/Offset: 20/50000 row(s)  (actual time=133..133 rows=0 loops=1)
    -> Nested loop inner join       (actual time=0.629..132 rows=50000 loops=1)
        -> Index lookup on b using idx_board_deleted_created (is_deleted=false)
                                    (actual time=0.602..58.4 rows=50000 loops=1)
        -> Single-row index lookup on u using PRIMARY (idx=b.user_idx)
                                    (actual time=0.00114..0.00117 rows=1 loops=50000)
                                                                        └─ 5만 번 반복
```

**원인은 인덱스가 아니라 `OFFSET` 그 자체다.**
인덱스(`idx_board_deleted_created`)는 **정상적으로 탄다.** 그런데 `OFFSET 50000` 이므로 MySQL 은
**5만 행을 전부 만들어낸 다음 그걸 통째로 버린다.** users PK 룩업도 **5만 번(loops=50000)** 돈다.
그러고 반환은 0행이다. board 가 50,000행이라 offset 50,000 은 끝을 넘어선다.

> 페이지가 깊어질수록 선형으로 나빠진다. **인덱스를 아무리 잘 만들어도 OFFSET 페이징은 못 고친다.**

**처방 후보**
1. **키셋(커서) 페이징** — `WHERE b.created_at < :lastCreatedAt ORDER BY b.created_at DESC LIMIT 20`. OFFSET 이 사라져 깊이와 무관하게 일정하다. 단 **UI 가 "N페이지로 점프"를 포기**해야 한다 (무한스크롤/더보기).
2. **지연 조인(deferred join)** — 커버링 인덱스로 `idx` 만 먼저 20개 뽑고 그 뒤에 조인. OFFSET 은 남지만 5만 번의 users 룩업이 사라진다. **UI 를 안 바꿔도 된다.**
3. 페이지 상한 (예: 100페이지까지만) — 미봉책.

> 실제로 사용자가 2500페이지를 볼 일은 없다. 다만 **크롤러·봇이 그렇게 한다.**

## 6. 🔴 발견 2 (재확인) — 자동생성 COUNT 가 users 를 풀스캔한다

`findBoardListItems` 에 `countQuery` 가 없어서 Hibernate 가 COUNT 를 만든다. **호출 3회에 180,003행 검사 / 74ms.**

**EXPLAIN ANALYZE — 드라이빙 테이블이 `users` 다:**

```
-> Aggregate: count(b.idx)                                  (actual time=22.4..22.4 rows=1)
    -> Nested loop inner join                               (actual rows=50000 loops=1)
        -> Filter: (u.status='ACTIVE' and u.is_deleted=false)
            -> Table scan on u                              (actual rows=10001 loops=1)   ← 풀스캔
        -> Covering index lookup on b using idx_board_user_deleted_created
                                                            (actual rows=5 loops=10001)
```

검사 60,001행 = **users 10,001행 풀스캔** + board 쪽 커버링 인덱스 룩업 50,000행.
board 쪽은 이미 커버링 인덱스로 최적이다. **낭비는 users 를 전부 훑는 데서 나온다.**

**주의 — 이 JOIN 은 그냥 걷어낼 수 없다.** `u.is_deleted`/`u.status` 가 **결과 건수에 영향을 준다**
(탈퇴·정지 회원의 글은 목록에서 빠지므로 총건수에서도 빠져야 한다). `countQuery` 를 `SELECT COUNT(*) FROM board`
로 단순화하면 **총건수가 틀린다.**

**처방 후보**
1. **`PageableExecutionUtils.getPage()`** — 마지막/단일 페이지면 COUNT 자체를 생략. `JpaUsersAdapter` 가 이미 쓰는 패턴. 다만 board 는 2,500 페이지라 **1페이지에서는 여전히 COUNT 가 나간다.**
2. **총건수 캐싱** (Redis, TTL 수 분) — 게시글 총건수는 몇 분 틀려도 된다. **효과가 가장 크다.**
3. **`Slice` 전환** — COUNT 를 아예 없앤다. UI 변경 필요.
4. 작성자 상태를 `board` 로 비정규화 — JOIN 제거 가능하나 정합성 유지 비용이 크다. **권하지 않는다.**

> 5번의 깊은 페이지 처방으로 **키셋 페이징**을 고르면 COUNT 문제도 **같이 사라진다** (`Slice` 이므로).
> 두 발견을 한 번에 해결하는 유일한 선택지다.

## 7. 🟡 발견 3 — FULLTEXT 검색의 COUNT (16,666행)

`searchByKeywordWithPaging` 은 `countQuery` 를 **명시해놨는데도** 16,666행을 검사한다.
`INNER JOIN users` 를 그대로 물고 가기 때문이다 (발견 2 와 같은 구조).
본문 SELECT(8,373행 / 34ms)보다 **COUNT 가 2배 비싸다.**

우선순위는 낮다 — 검색은 목록만큼 자주 호출되지 않는다.

## 8. 🐛 부수 발견 (성능 아님)

| 문제 | 근거 |
|---|---|
| **필수 파라미터 누락이 500 을 낸다 (400 이어야 함)** | `GET /api/boards/my-posts` (userId 없이) → `MissingServletRequestParameterException` → **HTTP 500**. `GlobalExceptionHandler` 가 이 예외를 처리하지 않아 일반 500 으로 떨어진다 |
| **`/my-posts?userId=` 가 쿼리 파라미터로 대상을 받는다** | `@PreAuthorize("isAuthenticated()")` 뿐이라 **인증된 사용자가 userId 만 바꾸면 남의 글 목록을 조회**할 수 있는 구조. 게시글 자체는 공개라 피해는 작지만, 인증 주체(`SecurityContext`)에서 꺼내는 게 맞다 |

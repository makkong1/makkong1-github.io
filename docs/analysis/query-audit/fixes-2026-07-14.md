## 1. ① care 검색이 죽어 있던 것 — FULLTEXT 인덱스 추가

### 무슨 일이었나

```
GET /api/care-requests/search?keyword=산책   →   HTTP 500

java.sql.SQLException: Can't find FULLTEXT index matching the column list
```

쿼리는 `MATCH(cr.title, cr.description) AGAINST(...)` 를 쓰는데, `**carerequest` 에 FULLTEXT 인덱스가 없었다.**

> **MySQL 은 FULLTEXT 인덱스 없이 `MATCH ... AGAINST` 를 실행하지 못한다.**
> 느린 게 아니라 **에러**다. 데이터가 0건이든 100만 건이든 **항상 500** 이다.
> 즉 이건 성능 문제가 아니라 **기능이 통째로 죽어 있던 것**이다.

`board` 는 같은 형태의 인덱스(`idx_board_title_content`)를 처음부터 갖고 있었다. **care 만 빠져 있었다.**

### 고친 것 — `V2__care_search_fulltext_index.sql`

```sql
ALTER TABLE carerequest
    ADD FULLTEXT INDEX idx_carerequest_title_desc (title, description);
```

### A/B/A 로 인과를 증명했다 (원칙 5)

"고쳤더니 됐다" 는 증거가 아니다. **빼면 다시 깨지는지**까지 봐야 한다.


| 단계    | 상태      | 결과                      |
| ----- | ------- | ----------------------- |
| **A** | 인덱스 있음  | **HTTP 200**            |
| **B** | 인덱스 제거  | 🔴 **HTTP 500** ← 문제 재현 |
| **A** | 인덱스 재적용 | **HTTP 200**            |


**인덱스가 원인이었음이 확정됐다.**

### 덤 — 인덱스 하나로 엔드포인트 두 개가 살아났다

공개 API(`/api/care-requests/search`)와 관리자 API(`/api/admin/care-requests?q=`)가
**같은 `MATCH()` 쿼리**를 쓰고 있었다. 둘 다 500 이었고, **둘 다 함께 200 이 됐다.**

## 2. ② admin care 의 N+1 — fetch join 추가

### 무슨 일이었나

`/api/admin/care-requests` 는 목록 20건을 보는 데 **쿼리 66개**를 날렸다.
쿼리 수가 **결과 건수에 1:1 로 비례**했다 (= N+1 의 정의).

원인은 **같은 테이블에 fetch join 이 있는 쿼리와 없는 쿼리가 공존**한 것이다.

```java
// 공개 API — JOIN FETCH 있음  ✅
"SELECT cr FROM CareRequest cr JOIN FETCH cr.user u LEFT JOIN FETCH cr.pet WHERE ..."

// 관리자 API — JOIN FETCH 없음  🔴  ← 공개 쪽만 고치고 여기는 빠뜨렸다
"SELECT r FROM CareRequest r WHERE (:status IS NULL OR ...) ORDER BY r.createdAt DESC"
```

### 고친 것 — `SpringDataJpaCareRequestRepository.findAllForAdmin`

```java
@Query(value = "SELECT r FROM CareRequest r JOIN FETCH r.user LEFT JOIN FETCH r.pet WHERE ...",
       countQuery = "SELECT COUNT(r) FROM CareRequest r WHERE ...")   // ← 필수
```

> ⚠️ `**Page<>` + `JOIN FETCH` 는 `countQuery` 를 반드시 명시해야 한다.**
> 안 그러면 Hibernate 가 **fetch join 을 물고 COUNT 를 만든다** — 감사에서 찾은 "자동생성 COUNT" 함정과 같은 것이다.

### 결과 — 비례성이 사라졌다


| `size` | 수정 전 | 수정 후  |
| ------ | ---- | ----- |
| 10     | 36   | **7** |
| 20     | 66   | **7** |
| 40     | 127  | **8** |


**수정 전에는 결과 수에 비례해 늘었고, 수정 후에는 늘지 않는다.**
size 를 4배로 키워도 쿼리가 7 → 8 로 거의 그대로다. **N+1 이 사라졌다는 증거는 "줄었다" 가 아니라 "비례하지 않는다" 이다.**

## 3. ③ pets 무제한 반환 — 페이징 추가 (그리고 내가 만든 버그 2개)

### 무슨 일이었나

```
GET /api/pets/type/DOG   →   강아지 7,667마리 전부 · 쿼리 155개 · 331ms
```

페이징도 상한도 없었다. 컨트롤러 → 서비스 → 어댑터 → Spring Data 4계층에 `Pageable` 을 태웠다.

### 🔴 그런데 수정하다가 내가 **새 N+1 을 만들었다**

처음엔 이렇게 썼다.

```java
return petRepository.findByPetTypeAndIsDeletedFalse(type, pageable)
        .map(petConverter::toDTO);        // ← 여기가 함정
```

측정해보니 20건에 쿼리가 **24개**였다. 첨부(`af1_0`) 조회가 **행마다 1번씩** 나갔다.

`**PetConverter` 의 javadoc 이 이미 경고하고 있었다:**

```java
/**
 * Entity → DTO 변환 (단일 객체용, File 개별 조회)
 * [주의] 리스트 변환 시에는 toDTOList() 사용 권장 (배치 조회로 N+1 방지)
 */
public PetDTO toDTO(Pet pet) { ... }
```

`Page.map()` 은 **단건 변환기를 행마다 호출**한다. 나는 그 경고를 읽지 않고 N+1 을 도로 만들었다.

**고친 것:**

```java
Page<Pet> page = petRepository.findByPetTypeAndIsDeletedFalse(type, pageable);
// Page.map(petConverter::toDTO) 를 쓰면 안 된다 — 단건 toDTO 는 펫마다 첨부를 조회해 N+1 이 난다.
return new PageImpl<>(petConverter.toDTOList(page.getContent()), pageable, page.getTotalElements());
```

`toDTOList` 는 첨부를 **배치(`IN`)로 한 번에** 가져온다. 쿼리 24 → **5** 로 떨어졌다.

> **교훈: `Page.map(converter::toDTO)` 는 N+1 의 지름길이다.**
> 이 프로젝트의 컨버터들은 `toDTO`(단건, 개별 조회)와 `toDTOList`(배치)가 쌍으로 있다. **목록에는 반드시 `toDTOList`.**
> 같은 실수가 `AdminCareAndMeetupFacade` 에도 있어서 함께 고쳤다.

### 🔴 그리고 **페이징을 붙이자 없던 COUNT 가 생겼다**

`List` → `Page<>` 로 바꾸면 **COUNT 쿼리가 따라온다.** 그 COUNT 가 이랬다.

```
pets 테이블은 12,000행인데   COUNT 가 19,667행을 검사한다
```

**EXPLAIN 을 보니 인덱스 머지(Intersect)였다:**

```
-> Filter: (is_deleted = 0) and (pet_type = 'DOG')
    -> Intersect rows sorted by row ID                          ← 두 인덱스를 교집합
        -> Index range scan using idx_pets_deleted (is_deleted=0)   → 12,000행
        -> Index range scan using idx_pets_type (pet_type='DOG')    →  7,667행
                                                        합계 = 19,667행
```

`(pet_type, is_deleted)` 복합 인덱스가 없어서 MySQL 이 **두 인덱스를 각각 읽고 교집합**을 구했다.

**고친 것 — `V3__pets_type_deleted_index.sql`**

```sql
ALTER TABLE pets ADD INDEX idx_pets_type_deleted (pet_type, is_deleted);
```

**A/B/A 로 확인했다:** 복합 인덱스를 빼면 **인덱스 머지가 부활**하고, 넣으면 사라진다.

> **막다른 길 하나 — 히스토그램은 안 먹혔다 (기록으로 남긴다).**
> 이 저장소는 `BoardListQueryPlanMaintainer` 에서 히스토그램으로 선택도 오판을 고친 전례가 있어서 먼저 시도했다.
> `ANALYZE TABLE pets UPDATE HISTOGRAM ON pet_type, is_deleted` 를 걸어도 **계획이 바뀌지 않았다.**
> **MySQL 은 인덱스가 있는 컬럼에는 히스토그램을 쓰지 않는다.** 그래서 히스토그램은 걷어내고 복합 인덱스만 남겼다.
> (안 쓰이는 변경은 남기지 않는다.)

### 최종


|           | 수정 전          | 수정 후                                |
| --------- | ------------- | ----------------------------------- |
| 반환 건수     | **7,667**     | **20** (`totalElements` 는 7,667 유지) |
| 쿼리 수      | **155**       | **5**                               |
| COUNT 검사행 | — (COUNT 없었음) | **7,667** (머지였다면 19,667)            |
| 시간        | 331ms         | **37ms**                            |


`size` 를 10/20/40 으로 바꿔도 쿼리는 **5개 그대로**다. 결과 수와 무관해졌다.

### ⑥ care 주변검색 — 🔴 **B-tree 로 시도했다가 실패했고, SPATIAL 로 다시 갔다**

**처음엔 이렇게 했다 (틀렸다):**

```sql
ALTER TABLE carerequest ADD INDEX idx_carerequest_geo (is_deleted, latitude, longitude);
```

**측정해보니 아무 효과가 없었다.** 3,261행 → 3,141행. 여전히 `Table scan on cr` 3,000행.

**왜 안 되는가 — 두 가지 이유가 겹쳤다:**

1. `**is_deleted` 는 선택도가 0 이다.** 전 행이 `0` 이라 인덱스 선두 컬럼으로서 아무것도 걸러내지 못한다.
2. `**longitude` 는 범위 조건으로 쓰이지 못한다.** B-tree 복합 인덱스는 **범위 조건을 선두에서 하나만**
 쓸 수 있다. `latitude BETWEEN ...` 이 범위를 잡아먹으면 뒤의 `longitude BETWEEN ...` 은 인덱스로 못 걸른다.

> **지리 검색은 B-tree 로 안 된다. 이건 SPATIAL 인덱스가 존재하는 이유 그 자체다.**

**제대로 고친 것 — meetup·locationservice 가 이미 쓰는 방식을 그대로 따랐다:**

```sql
-- SPATIAL 인덱스는 NOT NULL 컬럼에만 걸린다. 그런데 기존 행이 있는 테이블에 곧바로
-- NOT NULL POINT 컬럼을 추가하면 실패한다(Error 1138: Invalid use of NULL value).
-- → NULL 허용 추가 → 값 채움 → NOT NULL 로 변경 → SPATIAL 인덱스 순서로 간다.
ALTER TABLE carerequest ADD COLUMN geo_point POINT SRID 4326 NULL;
UPDATE carerequest SET geo_point = ST_GeomFromText(CONCAT('POINT(', latitude, ' ', longitude, ')'), 4326);
ALTER TABLE carerequest MODIFY COLUMN geo_point POINT NOT NULL SRID 4326;
ALTER TABLE carerequest ADD SPATIAL INDEX idx_carerequest_geo_point_spatial (geo_point);

CREATE TRIGGER trg_carerequest_set_geo_point_insert BEFORE INSERT ON carerequest ...  -- lat/lng → geo_point
CREATE TRIGGER trg_carerequest_set_geo_point_update BEFORE UPDATE ON carerequest ...
```

**트리거로 채우기 때문에 엔티티에 필드를 추가하지 않아도 되고, `ddl-auto=validate` 도 그대로 통과한다.**
(`meetup` 과 `locationservice` 가 정확히 이 패턴이다.)

쿼리도 `BETWEEN` → `ST_Within` + `ST_Distance_Sphere` 로 바꿨다. **인덱스만 만들고 쿼리를 안 바꾸면 무용지물이다.**


|            | 수정 전                             | 수정 후                                                       |
| ---------- | -------------------------------- | ---------------------------------------------------------- |
| 계획         | `Table scan on cr`               | `Index range scan using idx_carerequest_geo_point_spatial` |
| 검사행        | **3,000**                        | **208**                                                    |
| **선택도 추정** | 예상 3.77행 / 실제 783행 (**208배 오판**) | 예상 190행 / 실제 208행 (**거의 정확**)                              |


**A/B/A:** 인덱스 제거 → `Table scan` 3,000행 복귀 → 재적용 → `Index range scan` 208행. **인과 확정.**

## 4-2. ✅ CI 회귀 테스트 (원칙 6 — 2단계 검증)

**"고치면 테스트를 붙인다" 로는 부족하다.** 원칙 6 은 2단계를 요구한다:
**① 수정 전 상태에서 테스트가 실제로 빨간불이 되는지 먼저 확인** → ② 수정 후 초록불이 되는지 확인.
①을 건너뛰면 아무것도 검증하지 않는 테스트가 초록불만 켜고 있게 된다.


| 테스트                                   | 무엇을 지키나                           | ① 회귀 재현 시                                   |
| ------------------------------------- | --------------------------------- | ------------------------------------------- |
| `CareSearchFulltextIndexTest` (2)     | FULLTEXT 인덱스 존재 + 검색이 예외 없이 실행    | 인덱스 제거 → **FAILED**                         |
| `QueryCountScalingRegressionTest` (3) | **쿼리 수가 결과 건수에 비례하지 않는다**         | N+1 복원 → **FAILED** (size 5→40 에서 쿼리 15→85) |
| `IndexUsageRegressionTest` (3)        | 인덱스 사용 + filesort 없음 + SPATIAL 존재 | 인덱스 3개 제거 → **3개 모두 FAILED**                |


### 왜 "쿼리 수 상한" 이 아니라 "비례하지 않음" 인가

쿼리 수 상한(예: "20개 미만")만 걸면 **결과가 10건일 때는 통과하고 100건일 때만 터지는 테스트**가 된다.
N+1 의 정의는 "쿼리 수가 결과 건수에 비례한다" 이므로, **size 를 바꿔 두 번 호출하고 증가분을 본다.**

### 🔴 테스트를 짜다가 계기판이 두 번 고장났다 (기록으로 남긴다)

**하나. `getQueryExecutionCount()` 는 지연로딩을 세지 않는다.**
처음엔 이 지표로 쿼리를 셌다. 그랬더니 `findAllForAdmin` 의 `JOIN FETCH` 를 제거해도 **테스트가 통과했다.**
Hibernate 의 `getQueryExecutionCount()` 는 **JPQL/네이티브 쿼리만 세고, 지연로딩 엔티티 로드(PK 단건 조회)는 세지 않는다.**
그래서 pet 지연로딩 N+1 이 통째로 안 보였다.

> 하마터면 **"JOIN FETCH 는 불필요하다" 는 잘못된 결론**을 내리고 되돌릴 뻔했다.
> 실 API 로 재보니 `JOIN FETCH` 없이는 27/47/88 쿼리로 여전히 비례했다.
> `**getPrepareStatementCount()**` 로 바꾸니 그제야 테스트가 잡아냈다 (size 5→40 에서 15→85).

**둘. Gradle 이 테스트를 안 돌리고 이전 결과를 재생했다.**
인덱스를 지우고 테스트를 돌렸는데 통과하길래 파보니 `Task :test UP-TO-DATE` 였다.
**DB 상태는 Gradle 의 입력이 아니므로**, 코드가 안 바뀌면 테스트를 아예 실행하지 않는다.
→ DB 를 바꿔가며 검증할 때는 **반드시 `cleanTest`** 를 붙여야 한다.

**셋. 테스트는 개발 DB 가 아니라 `petory_test` 를 쓴다.**
`build.gradle` 이 테스트 DB 를 분리해놨다(`jdbc:mysql://localhost:3306/petory_test`).
개발 DB(`petory`)에서 인덱스를 지워봐야 테스트엔 아무 영향이 없다. `**petory_test` 에서 지워야 ①단계가 성립한다.**

> **세 번 다 "테스트가 초록불인데 실제로는 아무것도 검증하지 않는" 상태였다.**
> 이 감사의 원칙 6(①단계 먼저)이 없었다면 셋 다 놓쳤을 것이다.

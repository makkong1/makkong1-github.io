# meetup 도메인 쿼리 감사 — 실측 결과

> **한 줄 요약: N+1 처럼 보였지만 N+1 이 아니었다. 진짜 원인은 "페이징이 없다"는 것이다.**

---

## 1. 패스 B 가 잡은 것 — 쿼리가 결과 수에 비례한다

**curl 9회**를 던졌는데 digest 는 이렇게 나왔다.

| 쿼리 | 횟수 | 검사 |
|---|---|---|
| `SELECT u1_0...` (users) | **62** | 2,657 |
| `SELECT p1_0...` (participants) | **17** | 3,645 |

curl 9회에 users 62회. **패스 A(검사행순)로는 절대 안 보였을 신호다** — 개별 쿼리가 값싸기 때문이다.
**패스 B 를 만든 이유가 정확히 이것이다.**

---

## 2. 격리 — 범인은 검색과 주변이다

| 엔드포인트 | 반환 건수 | participants 쿼리 | users 쿼리 | 시간 |
|---|---|---|---|---|
| 목록 (page=0, size=20) | 20 | ~1 | 2 | 88ms |
| 주최자별 | 2 | 1 | 1 | 14ms |
| 참가자 목록 | 6 | 1 | 1 | 12ms |
| 🔴 **주변** | **~200** | **4** | **16** | 98ms |
| 🔴 **검색** | **500** | **10** | **40** | **247ms** |

**검색 1회 = 쿼리 51개.**

---

## 3. ⚠️ 정정 — 이건 고전적 N+1 이 아니다

처음엔 N+1 로 보였다. 하지만 숫자를 맞춰보면 **`@BatchSize` 는 정상 작동하고 있다.**

```java
// Meetup.java
@OneToMany(mappedBy = "meetup", cascade = CascadeType.ALL)
@BatchSize(size = 50)                    // ← 동작한다
private List<MeetupParticipants> participants;
```

```
검색 결과 500건  ÷  BatchSize 50  =  participants 쿼리 10회   ✅ 정확히 일치
```

N+1 이었다면 **500회**가 나왔어야 한다. 10회가 나왔다. **배칭은 제대로 되고 있다.**

> **그러면 무엇이 문제인가 — 결과가 500건이라는 것 자체다.**
> 배칭을 아무리 잘해도 500건을 반환하면 51개 쿼리는 나간다.
> **N+1 을 고치려 들었다면 이미 잘 되어 있는 코드를 건드릴 뻔했다.**

---

## 4. 🔴 진짜 원인 — 검색과 주변에 페이징이 없다

**검색: 페이징 파라미터가 아예 없다.**

```java
@GetMapping("/search")
public ResponseEntity<Map<String, Object>> searchMeetupsByKeyword(
        @RequestParam(value = "keyword") String keyword) {     // ← page/size 없음
    List<MeetupDTO> meetups = meetupService.searchMeetupsByKeyword(keyword);
```

```java
// MeetupService
public static final int MAX_LIST_SIZE = 500;
return converter.toDTOList(meetups.size() > MAX_LIST_SIZE
        ? meetups.subList(0, MAX_LIST_SIZE) : meetups);        // ← DB 에서 다 가져와 메모리에서 자른다
```

**두 번 잘못됐다.**
1. 페이징이 없어 **최대 500건**을 한 응답에 담는다.
2. `subList` 로 **메모리에서** 자른다 — DB 는 이미 전부 읽어온 뒤다.

**주변: `maxResults` 기본값이 500 이다.**

```java
@RequestParam(value = "maxResults", defaultValue = "500") int maxResults
```

프론트가 값을 안 주면 **500건**이 나간다. 지도에 500개를 찍을 리 없다.

**처방**
1. 검색에 `Pageable` 을 붙인다 (`page`/`size`, 기본 20). board 검색은 이미 이렇게 되어 있다.
2. 주변의 `maxResults` 기본값을 **500 → 20~50** 으로 낮춘다. 한 줄 수정이고 효과가 즉시 난다.
3. `MAX_LIST_SIZE` 500 은 **DB LIMIT 으로 내린다** (`subList` 제거).

## 5. ✅ 공간 인덱스는 제대로 쓰고 있다 (care 와 정반대)

```sql
WHERE ST_Within(m.geo_point, ST_GeomFromText(CONCAT('POLYGON((...))'), 4326))
  AND ST_Distance_Sphere(m.geo_point, ...) <= (:radius * 1000)
ORDER BY ST_Distance_Sphere(...) ASC
```

`meetup` 인덱스 8개 — `idx_meetup_geo_point_spatial`(SPATIAL), `idx_meetup_location`(lat,lng),
`idx_meetup_date_status`, `idx_meetup_title_description`(FULLTEXT) 등.

**`care` 는 이게 하나도 없어서 주변 검색이 풀스캔이고 검색은 HTTP 500 이다** ([care-2026-07-14.md](care-2026-07-14.md)).
**같은 프로젝트인데 두 도메인의 인덱스 성숙도가 완전히 다르다.**

목록 쿼리도 `idx_meetup_status` 로 **인덱스 레인지 스캔**을 탄다 (20행만 실제 검사).

## 6. 🟡 남은 것 — 자동생성 COUNT

| 쿼리 | 횟수 | 검사 | 인덱스미사용 |
|---|---|---|---|
| `SELECT COUNT(m1_0.idx) FROM meetup ...` | 1 | **5,000** | **1** |

`findAllNotDeleted(Pageable)` 의 COUNT 가 meetup 전체(5,000행)를 훑는다. board 와 같은 구조다.

> **숫자를 맞춰봐야 한다 — 500 ÷ 50 = 10.**

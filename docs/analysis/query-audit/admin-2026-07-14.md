# admin 도메인 (+statistics) 쿼리 감사 — 실측 결과

> **여기서 프로젝트 최초의 진짜 N+1 이 나왔다.** 앞선 10개 도메인에서는 0건이었다.

**측정:** 컨트롤러 10개 / 엔드포인트 16개 실호출. ADMIN 계정(`seed_user_1000`). 스케줄러 OFF. Redis FLUSHDB.

---

## 1. 🔴 발견 1 — `/api/admin/care-requests` 는 진짜 N+1 이다

앞선 도메인들에서 N+1 처럼 보였던 신호는 전부 **배칭이 정상 동작하는 것**이었다 (`@BatchSize`).
**이번엔 다르다.**

### 비례성 증명 (A/B/A — 원칙 5)

| `size` | pets 쿼리 | 첨부 쿼리 | 백신 쿼리 | 총 쿼리 |
|---|---|---|---|---|
| 10 | **10** | **10** | **10** | 36 |
| 20 | **20** | **20** | **20** | 66 |
| 40 | **40** | **40** | **40** | 127 |

**결과 건수에 1:1 로 정확히 비례한다.** 회당 검사행은 **1행(PK 룩업)**.
배칭이었다면 `size 40` 에서도 쿼리 1개(`IN (...)`)여야 한다. **N+1 이 맞다.**

목록 20건을 보는 데 **60개의 추가 쿼리**가 나간다.

### 원인 — 같은 테이블에 fetch join 이 있는 쿼리와 없는 쿼리가 공존한다

```java
// 공개 API — JOIN FETCH 있음 → N+1 없음  ✅
@Query("SELECT cr FROM CareRequest cr JOIN FETCH cr.user u LEFT JOIN FETCH cr.pet WHERE ...")
Page<CareRequest> findAllActiveRequestsWithPaging(...);

// 관리자 API — JOIN FETCH 없음 → N+1  🔴
@Query("SELECT r FROM CareRequest r WHERE "
     + "(:status IS NULL OR CAST(r.status AS string) = :status) AND "
     + "(:deleted IS NULL OR r.isDeleted = :deleted) ORDER BY r.createdAt DESC")
Page<CareRequest> findAllForAdmin(...);
```

**공개 쪽은 고쳤는데 관리자 쪽은 안 고쳤다.** `admin board` 는 `JOIN FETCH b.user` 가 있어서 멀쩡하다
(실측: 20건에 쿼리 1개씩). **care 만 빠졌다.**

**처방:** `findAllForAdmin` 에 `JOIN FETCH r.user LEFT JOIN FETCH r.pet` 을 추가한다.
단, `Page<>` + `JOIN FETCH` 조합은 `countQuery` 를 명시해야 한다 (안 그러면 Hibernate 가 fetch join 을 물고 COUNT 를 만든다).

> 첨부·백신까지 없애려면 `@BatchSize` 를 `pets.vaccinations` 와 첨부 조회에도 붙이거나,
> board 처럼 **배치 사후주입**으로 바꾼다.

## 2. 🔴 발견 2 — care 검색은 **admin 에서도** HTTP 500

```
GET /api/admin/care-requests?q=산책   →   500
```

`searchWithPaging` 도 `MATCH(cr.title, cr.description)` 를 쓴다. **FULLTEXT 인덱스가 없으니 여기서도 터진다.**

**care 검색 기능은 공개·관리자 양쪽 모두 죽어 있다.** ([care-2026-07-14.md](care-2026-07-14.md) §2)
인덱스 하나 추가로 **두 엔드포인트가 같이 살아난다.**

## 3. 🔴 발견 3 — admin 사용자 목록: `users` 풀스캔 + filesort

| 쿼리 | 횟수 | 반환 | 검사 | 정렬스캔 | 인덱스미사용 |
|---|---|---|---|---|---|
| `SELECT u1_0.idx, u1_0.id, ...` (QueryDSL) | 2 | 20 | **10,021** | **1** | **1** |
| `SELECT COUNT(u1_0.idx) FROM users` | 1 | 1 | **10,001** | 0 | **1** |

**EXPLAIN ANALYZE:**

```
-> Limit: 20 row(s)                                    (actual time=15.3..15.3 rows=20)
    -> Sort: u.created_at DESC, limit input to 20 row(s) per chunk    ← filesort
        -> Table scan on u                             (actual rows=10001 loops=1)   ← 풀스캔
```

**`users` 에 `created_at` 인덱스가 없다.**

```
users 인덱스: PRIMARY(idx) · id · email · username · uk_users_nickname
                                                   └─ 전부 단일 컬럼 유니크. 정렬용 인덱스 0개
```

20명을 보려고 **1만 명을 전부 읽고 전부 정렬한다.** board 는 `idx_board_deleted_created` 로 40행만 읽는다.

**처방:** `ALTER TABLE users ADD INDEX idx_users_created_at (created_at);`
(관리자 목록의 필터가 `role`/`status` 이므로 `(status, created_at)` 복합도 검토)

> 이건 `00-plan.md` §0.1 에서 **집계에서 누락됐다고 지적했던 그 `Page<>`** 다
> (`UsersService:64` 의 상속 `findAll(pageable)` + QueryDSL admin 목록 2개).
> **누락을 바로잡은 덕에 실제로 문제를 찾았다.**

## 5. ✅ statistics — 측정 완료. **가장 깨끗한 도메인이다**

`@PreAuthorize("hasRole('MASTER')")` 라 시드 ADMIN 으로는 403 이었다.
**사용자 승인을 받고** 시드 계정을 임시 MASTER 로 승격해 측정한 뒤 **즉시 원복했다**
(사용자 실계정 `dud123` 은 건드리지 않았다).

| 엔드포인트 | 결과 | 최대 검사행 |
|---|---|---|
| `/daily` | 200 · 54ms | **1행** |
| `/weekly` | 200 · 19ms | **0행** |
| `/monthly` | 200 · 12ms | **0행** |
| `/summary` | 200 | **1행** |

**Daily Summary Pattern 이 의도대로 동작한다.** 실시간 집계 쿼리를 날리지 않고
사전 집계된 `DailyStatistics` / 주간 / 월간 테이블을 **단건 조회**한다.

```java
public TodaySnapshotResponse getTodaySnapshot() {
    DailyStatistics snapshot = dailyStatisticsRepository.findByStatDate(today)   // ← 단건
            .orElse(DailyStatistics.builder().statDate(today).build());
    return TodaySnapshotResponse.from(snapshot);
}
```

### 덤 — `00-plan.md` §1-1(캐시 규칙)이 실측으로 검증됐다

`/summary` 는 `@Cacheable("todayStats")` 다. Redis 를 비우고 3회 연속 호출했다.

| 회차 | 시간 | **DB 쿼리 수** |
|---|---|---|
| 1회차 | 49ms | **2개** |
| 2회차 | 22ms | **1개** |
| 3회차 | 8.7ms | **1개** |

**2회차부터 통계 쿼리가 사라진다.** 남은 1개는 JWT 인증 필터의 사용자 조회로, 캐시 대상이 아니다.

> **캐시를 안 비우고 3회 측정 최소값을 썼다면 "통계 쿼리 없음" 이라는 잘못된 결론이 나왔을 것이다.**
> 원칙 §1-1 이 실제로 값을 했다.

> ⬜ **자정 배치(`StatisticsScheduler`)는 별개이고 여전히 미측정이다** (cron).
> 집계를 *만드는* 쪽이라 읽기보다 훨씬 무거울 수 있다 — 후속 과제.

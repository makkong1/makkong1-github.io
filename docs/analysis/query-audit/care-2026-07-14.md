# care 도메인 쿼리 감사 — 실측 결과

> **한 줄 요약: board 는 인덱스 6개로 잘 튜닝돼 있는데, care 는 인덱스가 사실상 없다.**
> 그리고 검색 엔드포인트는 **아예 죽어 있다 (HTTP 500).**

---

## 1. 측정 조건

`00-plan.md` §2 절차 동일 (앱 8081 · 스케줄러 OFF · digest 4096 · 호출 직전 TRUNCATE).
**엔드포인트 10개 × 각 1회.** 데이터: carerequest 3,000 / users 10,000 / pets 12,000.

```
목록                GET /api/care-requests?page=0&size=20                  200   93ms
목록(상태필터)      GET /api/care-requests?status=OPEN&page=0&size=20      200
주변검색            GET /api/care-requests/nearby?lat=..&lng=..&radius=5   200   17ms
상세                GET /api/care-requests/1                               200   13ms
내 요청             GET /api/care-requests/my-requests                     200    9ms
🔴 검색             GET /api/care-requests/search?keyword=산책             500    8ms  ← 죽어 있음
댓글 목록           GET /api/care-requests/1/comments                      200    9ms
리뷰(받은/쓴/평균)  GET /api/care-reviews/...                              200
```

## 2. 🔴 발견 1 (치명) — 검색 엔드포인트가 HTTP 500. FULLTEXT 인덱스가 없다

```
GET /api/care-requests/search?keyword=산책  →  500

java.sql.SQLException: Can't find FULLTEXT index matching the column list
  SELECT cr.* FROM carerequest cr INNER JOIN users u ...
  AND MATCH(cr.title, cr.description) AGAINST(? IN NATURAL LANGUAGE MODE)
```

쿼리는 `MATCH(cr.title, cr.description)` 을 쓰는데 **`carerequest` 에 FULLTEXT 인덱스가 없다.**
MySQL 은 FULLTEXT 인덱스 없이는 `MATCH ... AGAINST` 를 **아예 실행하지 못한다** (느린 게 아니라 에러다).

**이건 성능 문제가 아니다. 기능이 죽어 있는 것이다.** 데이터가 몇 건이든 항상 500 이다.

> **성능 감사를 하다가 기능 버그를 찾았다. API 를 실제로 호출했기 때문에 발견됐다.**
> SQL 만 손으로 던졌다면 — 어제처럼 — 절대 못 봤을 것이다.

**처방:** `board` 에는 이미 있는 것과 같은 FULLTEXT 인덱스를 추가한다 (Flyway 새 버전 파일).

```sql
-- board 의 기존 인덱스: idx_board_title_content (title, content) FULLTEXT
ALTER TABLE carerequest ADD FULLTEXT INDEX idx_carerequest_title_desc (title, description);
```

## 3. 🔴 발견 2 — `carerequest` 에 인덱스가 사실상 없다

**board 와 비교하면 방치 상태가 명확하다.**

| board (6개) | carerequest (3개) |
|---|---|
| `idx_board_deleted_created` (is_deleted, created_at) | `PRIMARY` (idx) |
| `idx_board_category_deleted_created` (category, is_deleted, created_at) | `user_idx` (user_idx) ← FK |
| `idx_board_user_deleted_created` (user_idx, is_deleted, created_at) | `fk_carerequest_pet` (pet_idx) ← FK |
| `idx_board_created_at_desc` (created_at) | |
| `idx_board_title_content` (title, content) FULLTEXT | |
| `idx_board_status` (status) | |

**carerequest 의 3개는 전부 PK 아니면 FK 다. 조회를 위해 만든 인덱스가 하나도 없다.**

### 목록 SELECT — 풀스캔 + filesort

```
-> Limit: 20 row(s)                              (actual time=4.53..4.63 rows=20)
    -> Nested loop inner join
        -> Sort: cr.created_at DESC              (actual rows=20)            ← filesort
            -> Filter: (cr.is_deleted = 0)       (actual rows=3000)
                -> Table scan on cr              (actual rows=3000 loops=1)  ← 풀스캔
```

20행을 얻으려고 **3,000행을 전부 읽고 전부 정렬한다.**
board 는 같은 일을 `idx_board_deleted_created` 로 **40행만 읽고** 해낸다 (정렬 불필요).

**처방:** board 와 동일한 형태의 인덱스를 추가한다.

```sql
ALTER TABLE carerequest ADD INDEX idx_carerequest_deleted_created (is_deleted, created_at);
ALTER TABLE carerequest ADD INDEX idx_carerequest_status_deleted_created (status, is_deleted, created_at);
```

## 4. 🔴 발견 3 — 주변 검색: 풀스캔 + 선택도 **208배** 오판

```
-> Filter: (cr.is_deleted=0 and cr.latitude between .. and cr.longitude between ..)
           (cost=330 rows=3.77)  (actual time=0.0971..5.78 rows=783 loops=1)
                  └─ 예상 3.77행                        └─ 실제 783행   = 208배 오판
    -> Table scan on cr  (actual rows=3000 loops=1)
```

**두 가지가 동시에 잘못됐다.**

1. **인덱스가 없어서 3,000행을 전부 읽는다.** `latitude`/`longitude` 에 인덱스가 없다 — B-tree 도, 공간 인덱스도.
2. **옵티마이저가 선택도를 208배 오판한다.** `lat BETWEEN` 과 `lng BETWEEN` 을 **독립 조건으로 곱해서**
   3.77행이 남을 거라 예상하지만, 실제로는 783행이 남는다. 위도와 경도는 **강하게 상관**되어 있기 때문이다
   (서울 안의 점은 위도도 경도도 같이 서울 범위다).

지금은 carerequest 가 3,000행이라 5.78ms 로 버티지만, **행이 늘면 선형으로 나빠진다.**

**처방:** `POINT` 컬럼 + `SPATIAL INDEX`. `location` 도메인은 **이미 이렇게 하고 있고 잘 동작한다**
(baseline §2 에서 `ST_Within` 이 공간 인덱스를 타는 것 확인). **care 만 안 되어 있다.**

## 5. 🟡 발견 4 — COUNT 가 인덱스를 못 쓴다

| 쿼리 | 횟수 | 검사 | 인덱스미사용 |
|---|---|---|---|
| `SELECT COUNT(cr1_0.idx) FROM carerequest ...` | 1 | **6,000** | **1** |

carerequest 3,000행인데 6,000행을 검사한다 (users JOIN). board 와 동일한 **자동생성 COUNT** 구조다.
발견 2의 인덱스를 추가하면 같이 개선된다.

## 6. ✅ N+1 없음 / ✅ 쓰기 과잉 락 없음

**패스 B (호출횟수순):**

| 쿼리 | 횟수 | 회당검사 | 판정 |
|---|---|---|---|
| `SELECT u1_0...` (users) | 10 | 1 | ✅ curl 10회 = 인증 필터 10회. **N+1 아님** |
| `SELECT a1_0...` (careapplication 배치) | 1 | 30 | ✅ 배치 1회 |

목록 20건에 대해 지원자(careapplication)·첨부 조회가 **각 1회**다. 배치가 동작한다.

**패스 C:** 쓰기 엔드포인트를 호출하지 않아 digest 가 비었다. **care 쓰기 경로는 미측정** — 후속 과제.

## 7. 🐛 감사 중 발견한 시드 데이터 버그 (수정함)

care 목록·상세가 **HTTP 400** 을 뱉어서 파보니 앱 버그가 아니라 시드 데이터 문제였다.

```
InvalidDataAccessApiUsageException: No enum constant PetGender.FEMALE
```

| | 값 |
|---|---|
| `PetGender` enum | `M`, `F`, `UNKNOWN` |
| 시드가 넣은 값 | `MALE`, `FEMALE` — **12,000행 전부** |

`pets` 를 로딩하는 **모든 엔드포인트**가 터진다. `pets.gender` 가 `varchar(10)` 이라 DB 는 순순히 받았고,
`ddl-auto=validate` 도 타입만 보므로 못 잡는다.

**수정:** `scripts/seed/seed-dev-data.sql` 의 두 곳(156·167행)을 `M`/`F` 로 고치고, 기존 12,000행도 UPDATE 했다.

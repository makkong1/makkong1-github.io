# Petory 포트폴리오 PDF 구성안

## 방향

PDF는 단순 프로젝트 소개서보다 **문제 해결 역량을 증명하는 기술 포트폴리오**로 구성하는 것이 좋다.

Petory는 기능 범위가 넓기 때문에 모든 도메인을 동일 비중으로 설명하면 산만해진다. 대신 다음 메시지를 중심에 둔다.

> Spring Boot, JPA, MySQL 기반 프로젝트에서 실제 사용자 흐름을 기준으로 병목과 동시성 문제를 재현하고, 쿼리 수·실행 시간·메모리 사용량으로 개선 결과를 검증한 백엔드 프로젝트

권장 분량은 **13장**이다. 10장으로 줄일 수도 있고 15장까지 늘릴 수도 있지만, 현재 문서와 프로젝트 자료 기준으로는 13장이 가장 균형이 좋다.

## 4. Architecture

### 목적

프로젝트가 어떤 구조로 설계됐는지 보여준다.

### 핵심 메시지

React 클라이언트, Spring Boot 백엔드, MySQL, Redis, 외부 API, FastAPI NLP 서버가 역할별로 분리되어 있다.

### 넣을 내용

```text
React SPA / Capacitor
  -> Spring Security / JWT
  -> Domain Services
  -> MySQL / Redis
  -> Naver Map, OAuth2, SMTP, FCM, FastAPI NLP
```

### 강조할 점

- Spring Security 기반 인증/인가
- Controller-Service-Repository 레이어 분리
- 도메인별 패키지 구조
- Redis 캐시, TTL, NLP dedup 활용
- NLP 서버를 Spring 뒤에 두어 프론트 직접 호출 방지

### 추천 문구

> 전체 구조는 React 클라이언트가 Spring Boot API를 호출하고, Spring Security를 통과한 요청이 도메인 서비스로 전달되는 방식입니다. 데이터 저장은 MySQL, 캐시와 TTL성 데이터는 Redis가 담당하며, NLP 분석은 FastAPI 서버를 별도 프로세스로 분리했습니다.

## 6. Case 1. Care 요청 목록 N+1

### 목적

가장 임팩트가 큰 성능 개선 사례를 보여준다.

### 핵심 메시지

DTO 변환 과정에서 연관 컬렉션 lazy load가 반복되며 대량 쿼리가 발생했고, 이를 배치 조회 관점으로 해결했다.

### 문제

- 펫케어 요청 목록 조회 시 `CareRequest.applications` 접근으로 N+1 발생
- 페이징 쿼리에서 applications fetch가 빠짐
- Converter에서 `applicationCount`, `applications` 변환 중 lazy load 반복

### 해결

- 필요한 연관 데이터 조회 전략 재정리
- `@BatchSize` 또는 IN 절 기반 배치 조회
- DTO 변환 시 이미 조회한 데이터를 메모리에서 매핑

### 결과

| 지표 | Before | After | 개선 |
|---|---:|---:|---:|
| 쿼리 수 | 2400개 | 4-5개 | 99.8% 감소 |
| 실행 시간 | 1084ms | 66ms | 94% 개선 |
| 메모리 | 21MB | 6MB | 71% 감소 |

### 추천 문구

> 목록 조회 API에서 DTO 변환 중 연관 컬렉션이 반복 lazy load되며 쿼리가 폭증했습니다. 조회 경로와 변환 책임을 분리하고 배치 조회를 적용해 쿼리 수를 2400개에서 4-5개로 줄였습니다.

## 7. Case 2. Board 목록 조회 최적화

### 목적

N+1 해결 패턴이 한 도메인에만 우연히 적용된 것이 아니라, 다른 도메인에도 확장됐음을 보여준다.

### 핵심 메시지

게시글 목록 조회에서 반복 조회되던 작성자, 반응, 댓글/첨부 정보를 배치 조회와 Map 매핑으로 해결했다.

### 문제

- 게시글 목록 조회 시 반응 정보, 댓글 수, 첨부 정보가 게시글마다 반복 조회
- 관리자 API에서 전체 게시글 로드 후 메모리 필터링

### 해결

- 게시글 ID 목록 추출
- IN 절 배치 조회
- GROUP BY count
- Map 기반 DTO 조립
- 관리자 API는 DB 레벨 필터링과 페이징으로 전환

### 결과

| 지표 | Before | After | 개선 |
|---|---:|---:|---:|
| 쿼리 수 | 301개 | 3개 | 99% 감소 |
| 실행 시간 | 745ms | 30ms | 24.8배 개선 |
| 메모리 | 22.5MB | 2MB | 91% 감소 |

### 추천 문구

> Board 도메인에서는 게시글 목록 조회 시 필요한 부가 정보를 게시글마다 개별 조회하던 구조를 ID 기반 배치 조회로 바꿨습니다. 그 결과 100개 게시글 기준 쿼리 수를 301개에서 3개로 줄였습니다.

## 8. Case 3. Login/Chat 목록 N+1

### 목적

단일 API가 아니라 실제 사용자 흐름 기준으로 병목을 분석했다는 점을 보여준다.

### 핵심 메시지

로그인 직후 채팅방 목록 구성 과정에서 방마다 참여자와 메시지를 조회하던 구조를 배치 조회로 개선했다.

### 문제

- 로그인 후 채팅방 목록 조회
- 각 채팅방마다 참여자 정보 개별 조회
- 각 채팅방마다 메시지 조회
- 최신 메시지만 필요한데 전체 메시지 접근 가능성 존재

### 해결

1. 채팅방 목록 조회
2. conversation ID 목록 추출
3. 현재 사용자 참여자 정보 배치 조회
4. 전체 참여자 정보 배치 조회
5. 최신 메시지만 배치 조회
6. Map으로 DTO 조립

### 결과

| 지표 | Before | After | 개선 |
|---|---:|---:|---:|
| 쿼리 수 | 21개 | 4개 | 80.95% 감소 |
| 실행 시간 | 305ms | 55ms | 81.97% 개선 |
| 메모리 | 0.58MB | 0.13MB | 77.24% 감소 |

### 추천 문구

> 로그인 직후 사용자가 실제로 마주치는 채팅방 목록 흐름을 기준으로 성능을 측정했습니다. 채팅방 수에 비례하던 참여자·메시지 조회를 배치 조회로 바꿔 쿼리 수를 4개로 고정했습니다.

## 9. Concurrency

### 목적

성능뿐 아니라 데이터 정합성 문제도 다뤘음을 보여준다.

### 핵심 메시지

동시성 문제는 하나의 방식으로 해결하지 않고, 문제 성격에 따라 다른 전략을 적용했다.

### 넣을 내용

| 영역 | 문제 | 해결 |
|---|---|---|
| Meetup | 최대 인원 초과 | 비관적 락 또는 원자적 UPDATE |
| Board | 좋아요/싫어요 중복 | DB Unique 제약조건 |
| Chat | unread count Lost Update | 원자적 증가 쿼리 |
| User | 닉네임 중복, 경고 횟수 | Unique 제약조건, 원자적 UPDATE |
| Report | 중복 신고 | DB Unique 제약조건 |
| Payment/Care | 거래 확정, 지급/환불 중복 | 트랜잭션과 row lock |

### 추천 문구

> 동시성 문제는 모두 락으로 해결하지 않았습니다. 중복 데이터는 DB Unique 제약조건으로 막고, 카운터성 데이터는 원자적 UPDATE로 처리했으며, 상태 전이가 중요한 거래 흐름은 트랜잭션과 row lock으로 보호했습니다.

## 10. Location Optimization

### 목적

JPA N+1 외의 쿼리 최적화 경험을 보여준다.

### 핵심 메시지

전체 데이터를 메모리에 로드하던 위치 기반 조회를 DB 필터링과 Bounding Box로 개선했다.

### 문제

- `findAllNotDeleted()`로 전체 meetup 로드
- Java에서 거리 계산, 필터링, 정렬 수행
- 데이터 증가 시 O(n) 메모리·시간 증가

### 해결

- DB에서 반경 조건 처리
- 위도/경도 Bounding Box 조건 적용
- 위치 인덱스 활용
- Haversine 계산은 후보군 축소 후 수행

### 결과

| 지표 | Before | After | 개선 |
|---|---:|---:|---:|
| 전체 실행 시간 | 486ms | 273ms | 43.8% 감소 |
| DB 쿼리 시간 | 241ms | 143ms | 40.7% 감소 |
| 메모리 | 1.48MB | 0.21MB | 85.8% 감소 |
| 스캔 행 수 | 2958개 | 117개 | 96% 감소 |

### 추천 문구

> 근처 모임 조회는 처음에 전체 데이터를 메모리에 올린 뒤 Java에서 거리 계산과 필터링을 수행했습니다. 이를 DB 조건으로 옮기고 Bounding Box를 추가해 스캔 행 수를 2958개에서 117개로 줄였습니다.

## 11. Recommendation / NLP

### 목적

외부 서버 연동과 장애 격리 설계 경험을 보여준다.

### 핵심 메시지

NLP는 핵심 트랜잭션에서 분리하고, 실패해도 원 요청을 막지 않도록 설계했다.

### 구조

```text
Board / Care / Location event
  -> Spring Event
  -> petIntentExecutor
  -> FastAPI NLP
  -> user_pet_intent_signal
  -> Location 추천 카드
```

### 강조할 설계

- 게시글/케어 이벤트는 `AFTER_COMMIT` 이후 분석
- Location 검색은 반복 호출 가능성이 높아 Redis dedup 적용
- NLP 전용 bounded executor 사용
- 큐 포화 시 부가 기능 작업 discard
- Python 서버 timeout 시 원 액션 영향 없음
- 즉시 추천은 NLP 실패 시 keyword fallback

### 추천 문구

> Recommendation은 진단 서버가 아니라 Location 카테고리 검색을 돕는 signal 생성 기능입니다. NLP 호출은 전용 executor에서 비동기로 수행하고, 실패하더라도 게시글 작성·케어 요청·검색 같은 원 요청에는 영향을 주지 않도록 설계했습니다.

## 전체 톤

문장은 화려한 소개보다 **문제 - 원인 - 해결 - 결과** 흐름으로 짧게 가져간다.

각 성능 사례 페이지는 아래 구조를 반복하면 좋다.

```text
1. 문제 상황
2. 원인
3. 해결 방식
4. Before/After 수치
5. 이 사례에서 얻은 패턴
```

이렇게 반복하면 PDF 전체가 일관되고, 면접관이 빠르게 훑어도 어떤 문제를 어떤 방식으로 해결했는지 바로 보인다.

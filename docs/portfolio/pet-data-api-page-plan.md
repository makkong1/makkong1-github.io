# pet-data-api 포트폴리오 페이지 계획 (개정판)

코드/`README`/Petory Recommendation 연동을 기준으로 점검한 내용을 반영했다. 추가로 아래 전제가 있다.

- **pet-data-api 레포 전체를 “내가 원래부터 바닥부터 짠 Python 서버”라고 적지 않는다.** 멀티스택 연동 과제 속에서 참고하거나 들어와 있는 레포라면 솔직하게 범위를 나눈다.
- **Python/FastAPI 쪽 구현 디테일**은 깊게 쓰려 하지 않는다. 읽어본 결과를 **한 줄로 요약하고 GitHub 링크로 넘기는** 형태가 맞다.

---

## 1. 페이지가 말해야 할 한 가지

**“Petory 메인 앱 밖에 두는 데이터·추천 파이프라인이 왜 필요한지, 그리고 내가 Petory 쪽에서 어떤 경계·계약을 맞췄는지”**다.

Python 서버를 하나 더 올렸다는 사실 자체가 중심이 아니다.

---

## 2. 나의 역할을 어떻게 쓸지 (정직한 스펙트럼)

아래 중 **사실에 맞는 범위만** 골라 쓴다. 과장하지 않는다.

| 범위 | 예시 문장 방향 |
|------|----------------|
| **Petory 측 (내가 직접 만질 수 있는 영역)** | Spring BFF(`RecommendService`, `PetDataApiClient`), 프론트 추천 카드·이벤트, `X-Request-Id`/`request_id` 세션, 이중 타임아웃, `user_ref` 익명화, 트렌드 시계열 프록시 등 |
| **연동·검증** | 엔드포인트·요청/응답 스키마 맞춤, 문서·페이지와 실제 플로우 불일치 점검, 계약 틀어짐 수정 제안 또는 반영(예: `age_months` 정합) |
| **Python 레포** | “전체 코드를 내가 작성했다”가 아니라면 → **역할 분리**(수집/서빙, 플래그별 파이프)와 **링크 근거**만 짧게. 세부 라이브러리 나열 금지 수준 |

핵심 문장 초안 예:

> “추천 점수·외부 데이터 수집 로직은 pet-data-api에 두고, Petory에서는 인증된 사용자 위치와 화면 맥락만 조립해 넘기는 BFF 역할까지 맞췄다. Python 레포 안쪽 구현은 제한적으로만 보완했다” / “실제 수정은 레포 소유 관계와 협업 범위에 따라 다르니, 포트폴리오에서는 **내가 수정·검증한 부분만** 명시한다.”

---

## 3. 한 줄 정의 (페이지 Hero)

**pet-data-api**는 Petory의 **위치·트렌드·외부 데이터**를 쓴 추천/시설 신호를 담당하는 **분리 저장소·파이프라인 쪽 레포**(또는 연동 서버)이고,

**Petory 페이지에서 강조할 정의**는 이쪽이다.

> **“메인 서비스 DB·배포 단위와 수집/추천 실험 단위를 나누기 위한 확장 레이어”**

“완제품 SaaS 하나 더” 정의는 쓰지 않는다.

---

## 4. 기술 흐름 — 코드 검증 결과를 반영한 정확한 말하기

플랜·포트폴리오 카피와 다이어그램에 아래가 **실제 코드와 일치**한다고 가정하고 쓴다. (예전처럼 “본 `/recommend`에서 항상 Ollama”처럼 쓰면 틀림.)

### 4.1 추천 응답에서 LLM이 어디 있는가

- **기본 `POST /recommend`**: `include_copy=false`가 기본 — **규칙 기반 카피** 등만으로 응답을 짧게 유지하는 방향이다. Petory BFF도 이 필드를 보내지 않아 **항상 그 기본 경로에 맞춰진 상태**와 같다.
- **LLM(또는 실패 시 규칙 폴백)**: **`POST /recommend/copy`** 등 분리 호출 또는 opt-in 플래그 쪽 개념으로 설명한다.
- 따라서 페이지 다이어그램에서는 **「본 호출」(빠름)과 「카피 호출」(느림, 선택)** 을 노드 두 개로 나누는 것이 맞다.

### 4.2 Petory 연동 플로 (BFF 관점 요약)

1. FE → **`GET /api/recommend`(lat, lng, context)** → JWT로 사용자 식별.
2. BE → 펫 정보는 **저장 프로필에서만** 채워 `POST {pet-data-api}/recommend` (헤더 `X-API-Key`, **`X-Request-Id`**).
3. 주요 고정값 예: 반경 km, top_n 등은 **프론트가 아니라 서버에서** 맞춤(조작 줄이기).
4. 카피: 본 응답 후 **`POST /api/recommend/copy`** → 같은 `request_id`로 로그 세션 매칭.
5. 타임아웃: **본 호출 클라와 카피 전용 클라를 분리**해, 긴 카피가 빠른 본 추천을 막지 않게 함.
6. 이벤트: **`POST .../events`** Petory에서는 202 즉시, 실패해도 사용자 막지 않음; 필요 시 **`user_ref`** 익명 식별.
7. 시계열: **`GET …/trends/{category}/timeseries`** 같은 경로를 BFF에서 프록시.

### 4.3 pet-data-api 쪽 파이프라인 두 갈래 (설명 깊이는 얕게)

- **플래그·컨텍스트별 “확장 파이프”**(멘션·외부 장소 보강·다중 신호 랭킹 등)와 **더 단순한 경로**가 코드상 분기된다. 활성 여부는 **환경 변수** 등에 따른다.
- **설명 깊게 가지 말 것**: “있고, 플래그에 따라 분기된다. 상세는 README” 정도로 끈다.

### 4.4 수집 레이어

- **스케줄**: README 기준 크론(로컬 시각, 타임존 미고정)·`max_instances=1` 같은 **설계 포인트**만.
- 배지상 **Redis PostgreSQL 버전**은 레포 규격이며, 실제 장비는 각자 다를 수 있음을 암시해도 무방하다.

### 4.5 계약·데이터 소비 시 주의 (페이지 한계 설명용)

- **시설 레코드에 DB 정수 PK(`id`)가 응답에 항상 없을 수 있다.** 노출 이벤트는 **`source_id` 중심**으로도 동작하게 설계된 구간이 있음을 짧게 쓸 수 있다.
- 펫 정보는 Java에서 **`type`/`breed`/`age_months`** 등으로 보내고, API 스키마가 이를 받아 신호에 쓰이도록 맞추는 것이 **계약 검증 포인트**다(예: 개월 단위 노령 휴리스틱).

## 5. 페이지 포지셔닝 (피해야 할 것 / 살리는 것)

**살리기**

- Petory 확장·**데이터 파이프 검증**.
- 책임 분리(수집 vs 서빙).
- 내가 한 일이 **어디 레이어까지인지**를 분리해서 쓴다.

**피하기**

- “내가 단독으로 Python 마이크로서비스를 완공했다”.
- 본 호출 하나에 항상 LLM이 들어간다는 식의 **플로 차트**.
- “Python 잘함” 증명용 라이브러리 나열.
- 배포까지 완성된 별도 상용 서비스인 것처럼 읽히는 헤더.

## 6. 핵심 메시지 (반복 노출 후보 문장 — 개정 반영)

1. 메인 서비스와 수집/추천 실험의 **실패 영역과 지연 시간을 분리**하려 했다.
2. 추천은 **단일 신호가 아니라** 시설·트렌드·환류 이벤트 등 **조합**으로 설계돼 있다(내가 깊게 짠 게 아니면 “레포 설계 의도가 그렇다”로 표현).
3. LLM 긴 호출과 짧은 본 호출을 **타임아웃·경로 레벨에서 분리**한 것은 Petory 통합 과정에서 명확한 구조 선택이다 (여기까지가 **내 역할 강점**으로 쓰기 좋다).
4. 현재 단계는 “운영 완성 하나 더”보다 **가능성·계약 검증**에 가깝다고 맞춘다.

## 7. 섹션 구성 제안 (구현 순서 무관 요약)

| 섹션 | 목표 |
|------|------|
| Hero | 서브타이틀: Petory **데이터·추천 보강 / 파이프라인 검증**. 한 줄에 **본인 역할 범위** 짧게. |

| 아키텍처 | 수집 상단, 서빙 하단. **본 `/recommend` vs `/recommend/copy`** 분리 표기. |

| 범위·한계 | 플래그 의존, 소스 품질, 이벤트 식별자 이중 체계 등 **정직한 한계**. |
| 얻은 점 | API vs 배치, 입력 신호·폴백, 계약 정합을 맞출 때의 학습. Python 자랑이 아님. |
| 링크 | pet-data-api `README`, Petory `PetDataApiClient`/`RecommendService` 등. |

## 8. 포트폴리오 / 이력서용 문구 후보

### 짧은 소개 (Python 자신감 낮을 때)

> Petory에서 추천·외부 시설 데이터를 쓰기 위해 **본 서버와 분리된 데이터 API**와 연동했다. 인증·위치·맥락 조립과 **짧은 본 호출 / 긴 카피 호출 분리**, 추천 세션 추적, 이벤트 익명화까지 BFF 쪽을 맞췄다. Python 레포는 수집·서빙 구조와 계약을 읽고 필요한 부분만 정합을 맞췄다.

### 이력서 한 줄 각도

> 위치 기반 추천: 외부 데이터 API 연동 설계(BFF·타임아웃·계약)·품질 환류 이벤트.

## 9. 페이지 구현 시 체크리스트

- [ ] 페이지 경로가 **`/domains/recommendation/pet-data-api`** 이고, 홈에서는 **별도 프로젝트 카드 없이** Petory 아래 안내 또는 Petory 페이지 링크로만 들어오게 했다 (`/pet-data-api` 등 구 주소는 리다이렉트).
- [ ] 다이어그램에 **본 추천에 LLM이 항상 붙어 있지 않다**는 걸 표현했다.
- [ ] “내가 전부 구현했다”가 아니라면 **내 기여 레이어**를 문장 한 번으로 명시했다.
- [ ] Python 세부 코드 대신 **README·Git 링크**로 넘긴다.
- [ ] Petory Recommendation 도메인 페이지와 **같은 스토리**(이중 타임아웃, copy, events)인지 교차 검토했다.

---

## 10. 결론

이 문서와 페이지는 **pet-data-api를 새 메인 프로젝트로 포장하기 위한 것이 아니라**, Petory를 확장할 때 필요한 **데이터·추천 경계**와 내가 거기까지 맞춘 **통합 스토리**를 보여 주기 위한 계획이다. 파이썬 숙련도가 핵심 메시지가 되지 않게 쓴다.

## 개요

GitHub Pages를 활용하여 **Petory 프로젝트 소개 + 포트폴리오** 사이트를 구축합니다.

### 목표
- 프로젝트 전반적인 소개 및 아키텍처 설명
- 각 도메인별 상세 기능 설명
- GitHub 저장소 링크 제공
- MCP 파일(설계 문서) 링크 제공
- 성능 개선 및 트러블슈팅 경험 정리
- 포트폴리오 섹션 포함

## 사이트 구조

```
/
├── index.html (메인 페이지)
├── portfolio.html (포트폴리오 섹션)
├── domains/
│   ├── user.html (유저 도메인)
│   ├── board.html (게시판 도메인)
│   ├── care.html (펫케어 도메인)
│   ├── missing-pet.html (실종 신고 도메인)
│   ├── location.html (주변 서비스 도메인)
│   ├── meetup.html (모임 도메인)
│   └── chat.html (채팅 도메인)
├── performance/
│   └── improvements.html (성능 개선 & 트러블슈팅)
├── docs/
│   └── mcp-files.html (MCP 파일 링크 모음)
└── assets/
    ├── css/
    ├── js/
    └── images/
```

### 3. 도메인별 상세 페이지 (`domains/*.html`)

각 도메인 페이지는 동일한 구조를 가집니다:

#### 공통 구조
1. **도메인 소개**
   - 도메인 이름 및 역할
   - 주요 기능 설명

2. **핵심 기능**
   - 기능별 상세 설명
   - 사용자 시나리오
   - 스크린샷/다이어그램 (가능하면)

3. **기술 구현**
   - 주요 엔티티/모델
   - API 엔드포인트 요약
   - 비즈니스 로직 하이라이트

4. **아키텍처 다이어그램**
   - 도메인 간 관계
   - 데이터 흐름

5. **관련 문서 링크**
   - GitHub 저장소 링크
   - 관련 MCP 파일 링크
   - 상세 문서 링크

#### 도메인별 특화 내용

##### 유저 도메인 (`user.html`)
- 사용자 인증/인가 (JWT)
- 반려동물 관리
- 프로필 관리
- 소셜 로그인
- 사용자 제재 시스템

##### 게시판 도메인 (`board.html`)
- 커뮤니티 게시판
- 댓글 시스템
- 좋아요/싫어요 반응
- 인기글 스냅샷
- 실시간 조회수 업데이트

##### 펫케어 도메인 (`care.html`)
- 펫케어 요청 작성
- 지원 시스템
- 댓글 기반 채팅 시작
- 리뷰 시스템
- 상태 관리 (OPEN, IN_PROGRESS, COMPLETED)

##### 실종 신고 도메인 (`missing-pet.html`)
- 실종 동물 신고
- 위치 기반 검색
- 상태 관리
- 알림 시스템 연동

##### 주변 서비스 도메인 (`location.html`)
- 위치 기반 서비스 검색
- 병원, 카페 등 POI 정보
- 리뷰 시스템
- 지도 연동

##### 모임 도메인 (`meetup.html`)
- 오프라인 모임 생성
- 참여자 관리
- 최대 인원 제한
- 모임 상태 관리

##### 채팅 도메인 (`chat.html`)
- 실시간 채팅 시스템
- WebSocket 기반 통신
- 채팅방 타입별 관리
  - DIRECT (1:1 채팅)
  - CARE_REQUEST (펫케어 요청 채팅)
  - MISSING_PET (실종 신고 채팅)
  - MEETUP (모임 채팅)
- 메시지 읽음 처리
- 알림 연동

## 기술 스택 및 구현 방식

### 옵션 1: 정적 HTML + CSS + JavaScript (가장 간단)
- **장점**: 빠른 구현, GitHub Pages와 완벽 호환
- **단점**: 반복 코드 많음, 유지보수 어려움
- **추천도**: ⭐⭐⭐

### 옵션 2: Jekyll (GitHub Pages 기본 지원)
- **장점**: GitHub Pages 네이티브 지원, 템플릿 활용 가능
- **단점**: Ruby 기반, 학습 필요
- **추천도**: ⭐⭐⭐⭐

### 옵션 3: React + GitHub Pages (빌드 후 배포)
- **장점**: 현재 프로젝트와 동일한 스택, 컴포넌트 재사용
- **단점**: 빌드 과정 필요, 라우팅 설정 필요
- **추천도**: ⭐⭐⭐⭐⭐ (프로젝트와 일관성)

### 옵션 4: Next.js Static Export
- **장점**: SEO 최적화, 정적 생성 지원
- **단점**: 추가 설정 필요
- **추천도**: ⭐⭐⭐⭐

### 추천: React + GitHub Pages
- 현재 프로젝트와 동일한 기술 스택
- 컴포넌트 기반 개발로 유지보수 용이
- `gh-pages` 패키지로 간편한 배포

## 배포 전략

### 1. 저장소 구조
```
petory-portfolio/ (새 저장소 또는 기존 저장소의 gh-pages 브랜치)
├── package.json
├── src/
│   ├── components/
│   ├── pages/
│   ├── styles/
│   └── App.js
├── public/
└── build/ (빌드 결과물)
```

### 2. 배포 프로세스
1. React 프로젝트 생성
2. `gh-pages` 패키지 설치
3. `package.json`에 배포 스크립트 추가:
   ```json
   {
     "scripts": {
       "deploy": "npm run build && gh-pages -d build"
     }
   }
   ```
4. GitHub 저장소 Settings → Pages에서 `gh-pages` 브랜치 선택
5. `npm run deploy` 실행

### 3. 도메인 설정 (선택)
- `username.github.io/petory-portfolio` 또는
- 커스텀 도메인 연결 가능

# Statistics 도메인 예외처리 리팩토링

## 1. 현황 분석

### 1.1 예외 발생 위치 및 유형

| 서비스/컨트롤러 | 예외 메시지 | 현재 타입 | 권장 HTTP |
|----------------|-------------|-----------|-----------|
| **StatisticsService** | (없음) | - | - |
| **StatisticsScheduler** | (없음) | - | - |

### 1.2 분석 결과

- **Statistics 도메인**에서는 현재 `throw new`, `orElseThrow`, `catch` 패턴이 **발견되지 않음**
- 통계 서비스는 주로 조회·집계 위주로 동작하며, 리소스 미존재 시 빈 값 또는 0 반환 패턴 사용 가능성 높음
- 추후 통계 API 확장 시(예: 특정 기간/조건 필수, 권한 검증 등) 예외가 추가될 수 있음

---

## 2. 도입할 예외 클래스 (추후 필요 시)

### 2.1 예상 시나리오

| 시나리오 | 예외 클래스 | HTTP |
|----------|-------------|------|
| 통계 조회 권한 없음 | StatisticsForbiddenException | 403 |
| 필수 파라미터 누락 (기간, 타입 등) | StatisticsValidationException | 400 |
| 통계 데이터 미존재 (선택) | - | 404 또는 빈 응답 |

### 2.2 구조 (추후)

```
domain/statistics/exception/
├── StatisticsForbiddenException.java   # 403 - 통계 조회 권한 없음
└── StatisticsValidationException.java  # 400 - 기간/타입 등 파라미터 검증 실패
```

## 4. 참고

- 다른 도메인 예외처리 문서와 동일한 `ApiException` 베이스 상속 원칙 적용
- 통계는 읽기 전용이므로, 404 대신 빈 집계 결과(`{ "count": 0, "items": [] }`) 반환을 우선 고려

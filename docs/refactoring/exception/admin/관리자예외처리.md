# Admin 도메인 예외처리 리팩토링

## 1. 현황 분석

### 1.1 예외 발생 위치 및 유형

| 서비스/컨트롤러 | 예외 메시지 | 현재 타입 | 권장 HTTP |
|----------------|-------------|-----------|-----------|
| **AdminUserManagementController** | ADMIN 역할만 지정할 수 있습니다 | IllegalArgumentException | 400 |
| | 이미 존재하는 사용자명입니다 | IllegalArgumentException | 409 |
| | 이미 존재하는 아이디입니다 | IllegalArgumentException | 409 |
| | 이미 존재하는 이메일입니다 | IllegalArgumentException | 409 |
| | 비밀번호는 필수입니다 | IllegalArgumentException | 400 |
| | 사용자를 찾을 수 없습니다 | IllegalArgumentException (orElseThrow) | 404 |
| | MASTER 권한은 변경할 수 없습니다 | IllegalArgumentException | 403 |
| | 일반 사용자는 promote-to-admin 엔드포인트를 사용해주세요 | IllegalArgumentException | 400 |
| | ADMIN 역할만 지정할 수 있습니다. MASTER로 변경할 수 없습니다 | IllegalArgumentException | 400 |
| | 일반 사용자는 이 엔드포인트로 관리할 수 없습니다 | IllegalArgumentException | 403 |
| | 새 비밀번호는 필수입니다 | IllegalArgumentException | 400 |
| **AdminUserController** | 관리자 계정은 별도 엔드포인트를 사용해주세요 | IllegalArgumentException | 400 |
| | 관리자 역할 변경은 별도 엔드포인트를 사용해주세요 | IllegalArgumentException | 400 |
| | 관리자 계정 삭제는 별도 엔드포인트를 사용해주세요 | IllegalArgumentException | 400 |
| **AdminMissingPetController** | status is required | IllegalArgumentException | 400 |
| | catch (IllegalArgumentException e) 후 재throw | - | - |
| **AdminFileController** | catch (IllegalArgumentException e) → 빈 목록 또는 badRequest | - | - |
| **AdminMeetupController** | catch (IllegalArgumentException e) | - | - |
| **AdminLocationController** | catch (Exception e) | - | - |
| **AdminCareRequestController** | 인증되지 않은 사용자입니다 | RuntimeException | 401 |
| | 유효하지 않은 사용자 ID입니다 | RuntimeException | 401 |
| | 복구 기능은 아직 구현되지 않았습니다 | UnsupportedOperationException | 501 |
| **AdminReportController** | 관리자 식별자를 확인할 수 없습니다 | IllegalArgumentException | 401 |
| | catch (Exception e) 후 throw e | - | - |

### 1.2 문제점

1. **예외 타입 혼용**: RuntimeException, IllegalArgumentException, UnsupportedOperationException 혼재
2. **인증 예외**: AdminCareRequestController의 getCurrentUserId()에서 RuntimeException 사용 → 401에 적합한 예외 필요
3. **User 도메인 의존**: "사용자를 찾을 수 없습니다" → UserNotFoundException 재사용 가능
4. **중복 검증**: "이미 존재하는 사용자명/아이디/이메일" → DuplicateUserFieldException 재사용 가능
5. **Controller try-catch**: AdminFileController, AdminLocationController 등에서 예외를 잡아 수동 처리

## 2. 도입할 예외 클래스

### 2.1 구조

```
domain/admin/exception/
├── AdminUnauthorizedException.java      # 401 - 관리자 인증 실패 (principal 없음, 유효하지 않은 ID)
├── AdminForbiddenException.java        # 403 - MASTER 권한 변경 불가, 일반 사용자 엔드포인트 접근 불가
├── AdminValidationException.java       # 400 - 역할 검증, status 필수, 비밀번호 필수 등
├── AdminConflictException.java         # 409 - 중복 (ADMIN 생성 시 username/id/email 중복)
└── UnsupportedAdminOperationException.java # 501 - 복구 기능 미구현 등
```

### 2.2 User 도메인 예외 재사용

- **UserNotFoundException**: 사용자 조회 실패 시 (orElseThrow 대체)
- **DuplicateUserFieldException**: username/id/email 중복 시 (user 도메인 예외 import)

### 2.3 ApiException 베이스

- `global/exception/ApiException.java` 상속 (기존 유지)

## 4. AdminValidationException 정적 메서드 제안

```java
AdminValidationException.roleAdminOnly()           // ADMIN 역할만 지정할 수 있습니다
AdminValidationException.masterRoleNotChangeable() // MASTER 권한은 변경할 수 없습니다
AdminValidationException.usePromoteEndpoint()      // 일반 사용자는 promote-to-admin 엔드포인트를 사용해주세요
AdminValidationException.passwordRequired()        // 비밀번호는 필수입니다
AdminValidationException.newPasswordRequired()     // 새 비밀번호는 필수입니다
AdminValidationException.statusRequired()          // status is required
AdminValidationException.useAdminEndpoint()        // 관리자 계정은 별도 엔드포인트를 사용해주세요
AdminValidationException.adminIdentifierRequired() // 관리자 식별자를 확인할 수 없습니다
```

# Notification 도메인 예외처리 리팩토링

## 1. 현황 분석

### 1.1 예외 발생 위치 및 유형

| 서비스/컨트롤러 | 예외 메시지 | 현재 타입 | 권장 HTTP |
|----------------|-------------|-----------|-----------|
| **NotificationService** | User not found | IllegalArgumentException | 404 |
| | Notification not found | IllegalArgumentException | 404 |
| | 본인의 알림만 읽음 처리할 수 있습니다 | IllegalStateException | 403 |
| **NotificationController** | catch (Exception) - stream 초기 알림 개수 | - | 로그만 |
| **NotificationSseService** | catch (IOException) - 알림 전송 | - | SSE 연결 해제 |
| | catch (Exception) - emitter 완료 | - | 로그만 |

### 1.2 문제점
1. **예외 타입 혼용**: IllegalArgumentException vs IllegalStateException
2. **메시지 영문**: "User not found", "Notification not found"
3. **User 도메인 의존**: UserNotFoundException 재사용 필요
4. **SSE 특수성**: NotificationSseService는 IOException 등 catch 후 연결 해제 (SSE 스트림 특성상 예외 전파 어려움)

---

## 2. 도입할 예외 클래스

### 2.1 구조
```
domain/notification/exception/
├── NotificationNotFoundException.java  # 404 - 알림 미존재
└── NotificationForbiddenException.java # 403 - 본인의 알림만 읽음 처리 가능
```

### 2.2 User 도메인 예외 재사용
- **UserNotFoundException**: 사용자 조회 실패 시

### 2.3 ApiException 베이스
- `global/exception/ApiException.java` 상속 (기존 유지)

## 4. 예외 정적 메서드 제안

```java
NotificationNotFoundException // 알림을 찾을 수 없습니다
```

# Location 지도 결과 안정성 리팩토링

**작성일**: 2026-05-29  
**상태**: 구현 완료, 검증 완료  
**대상**: `UnifiedPetMapPage`, `unifiedMapApi`, `LocationServiceService`, `SpringDataJpaLocationServiceRepository`, `docs/domains/location.md`

---

## 배경

현재 주변서비스 지도는 `searchCenter`와 `mapViewportCenter`를 분리해, 지도 드래그만으로 location 검색이 바로 바뀌지 않도록 개선되어 있다.

다만 아직 다음 요인 때문에 사용자가 보기에는 "같은 지역을 검색했는데 결과가 달라지는" 흐름이 남아 있다.

1. location 조회 캐시 키에 `mapLevel`이 포함되어 있다.
2. location 조회 `size`가 줌 레벨에 따라 바뀐다.
3. `"이 지역 검색"` 표시 기준이 좌표 차이 `0.0001`도, 약 11m 수준이다.
4. 반경 검색은 DB에서 제한하지 않고 Java에서 `limit(maxResults)`를 적용한다.
5. 기본 거리 정렬은 현재 좌표의 미세한 변화에도 `LIMIT` 절단선이 달라질 수 있다.

---

## 현재 문제 구조

```text
사용자 위치 A: 37.4979, 127.0276
사용자 위치 B: 37.4981, 127.0278

두 좌표는 사용자가 체감하기에 같은 강남역 인근이지만,
거리순 정렬 기준점은 매번 달라진다.

ORDER BY ST_Distance_Sphere(현재 좌표)
LIMIT N

결과적으로 절단선 근처 장소가 요청마다 포함되거나 빠질 수 있다.
```

이 문제는 단순히 프론트에서 debounce를 늘리거나 캐시를 추가하는 방식으로는 해결되지 않는다. 조회 기준, 캐시 키, 반환 개수, 정렬 정책을 함께 정리해야 한다.

---

## 선택한 방향

### 1. location 조회 조건에서 `mapLevel` 분리

location 탭에서는 줌 레벨 변경만으로 API를 다시 호출하지 않는다.

- `useEffect` 의존성에서 location 조회에만 `mapLevel` 영향을 제거한다.
- location 캐시 키에서도 `mapLevel`을 제외한다.
- meetup/care는 기존처럼 `mapLevel` 기반 제한을 유지한다.

기대 효과:

- 같은 `searchCenter`, `radius`, `keyword`, `category`, `sort` 조합은 같은 캐시 키를 사용한다.
- 지도를 확대/축소했을 뿐인데 주변서비스 결과가 바뀌는 현상을 줄인다.

### 2. location `size`를 고정값으로 변경

location은 줌 레벨별 가변 `size` 대신 고정값을 사용한다.

```text
location size: 300
meetup size: mapLevel 기반 유지
care size: mapLevel 기반 유지
```

기대 효과:

- 같은 지역 검색에서 줌 레벨 때문에 후보군 크기가 바뀌지 않는다.
- 캐시 키와 API 응답이 더 예측 가능해진다.

주의점:

- 현재 `ZOOM_LIMIT_TABLE.location.default`는 500이다.
- 고정값 300은 최대 노출 후보 수를 줄이는 변경이므로, 실제 데이터 밀집 지역에서 체감 결과 수를 확인해야 한다.

### 3. `"이 지역 검색"` 기준을 거리 기반으로 변경

현재 `COORD_EPSILON = 0.0001`은 위도 기준 약 11m이다. 사용자가 체감하는 "같은 지역" 기준으로는 너무 작다.

변경안:

```text
threshold = max(300m, radius * 10%)
```

예시:

| 반경 | 버튼 표시 기준 |
|------|----------------|
| 1km | 300m 이상 이동 |
| 3km | 300m 이상 이동 |
| 5km | 500m 이상 이동 |
| 10km | 1km 이상 이동 |

기대 효과:

- 지도 중심이 아주 조금 움직였다는 이유로 "이 지역 검색"이 과도하게 뜨지 않는다.
- 사용자가 체감하는 지역 단위와 버튼 표시 기준이 가까워진다.

### 4. 반경 검색에 SQL `LIMIT` 적용

현재 반경 검색은 DB에서 전체 후보를 가져온 뒤 Java에서 `limit(maxResults)`를 적용한다.

변경안:

- `findByRadius`에 `:limit` 파라미터를 추가한다.
- native query 마지막에 `LIMIT :limit`을 적용한다.
- `maxResults`가 null 또는 0 이하인 경우 컨트롤러/서비스에서 기본 제한값을 정규화한다.

기대 효과:

- DB 결과 전송량과 애플리케이션 메모리 사용량을 줄인다.
- 밀집 지역에서 반경 검색 비용이 예측 가능해진다.

주의점:

- SQL `LIMIT`은 성능 개선이다.
- 결과 안정성은 정렬 정책까지 함께 바꿔야 개선된다.

### 5. `sort=stable` 추가

거리순은 현재 좌표가 조금만 달라져도 정렬 순서가 바뀐다. 지도 기본 검색에는 안정적인 후보군 정렬이 필요하다.

변경안:

```text
sort=stable   rating DESC, review_count DESC, idx ASC
sort=distance ST_Distance_Sphere ASC, rating DESC, idx ASC
sort=rating   rating DESC, distance ASC, idx ASC
sort=reviews  review_count DESC, distance ASC, idx ASC
sort=score    score DESC
```

`score`는 `LocationService.score` 컬럼 기준이다. 현재 스케줄러가 매일 자정 `0.5 * rating * log10(reviewCount + 1) + 0.2 * petFriendly`로 재계산한다.

프론트 정책:

- location 기본 정렬은 `stable`로 변경한다.
- 사용자가 명시적으로 거리순을 선택하면 `distance`를 사용한다.
- UI 라벨은 `stable`을 "추천순"으로 노출한다.

기대 효과:

- 같은 지역 검색에서 좌표 미세 변화로 후보군이 크게 바뀌는 현상을 줄인다.
- 거리순이 필요한 사용자는 기존 거리순을 계속 선택할 수 있다.

## 검토했지만 이번에는 선택하지 않는 대안

### 좌표 격자 스냅핑

좌표를 0.005도 또는 0.01도 단위로 반올림하면 같은 격자 안에서는 요청 파라미터가 안정된다.

이번에 선택하지 않는 이유:

- 격자 경계선을 넘는 순간 결과가 다시 크게 바뀔 수 있다.
- 거리순 `LIMIT` 절단선 문제는 남아 있다.
- 사용자가 실제로 보고 있는 지도 중심과 서버 검색 중심이 달라질 수 있다.

### 기존 마커 점진적 머지

새 결과로 마커를 전면 교체하지 않고 기존 마커와 병합하면 시각적 흔들림은 줄어든다.

이번에 선택하지 않는 이유:

- 서버 결과의 일관성 문제를 가린다.
- 오래된 마커가 화면에 남아 실제 검색 결과와 UI가 어긋날 수 있다.
- 선택/호버/AI 추천 매칭 상태까지 병합 규칙이 복잡해진다.

### 뷰포트 바운딩 박스 쿼리

지도 화면의 남서/북동 좌표로 검색하면 화면이 같을 때 결과가 가장 직관적으로 안정된다.

이번에 선택하지 않는 이유:

- API 파라미터와 백엔드 쿼리 구조 변경 폭이 크다.
- 현재 반경 필터 UI와 제품 언어가 이미 `radius` 중심이다.
- 장기 개선안으로는 유효하지만, 이번 안정화 범위에는 과하다.

## 적용 후 좋아진 점

### 1. 줌 변경으로 인한 불필요한 location 재조회 제거

기존에는 location 캐시 키와 조회 의존성에 `mapLevel`이 포함되어 있었다. 그래서 같은 `searchCenter`와 반경을 유지해도 지도를 확대/축소하면 다른 캐시 키가 만들어지고 API가 다시 호출될 수 있었다.

변경 후 location 조회는 `searchCenter`, `radius`, `keyword`, `category`, `sort`가 바뀔 때만 의미 있게 갱신된다. 줌 레벨은 meetup/care 제한 계산에는 남겨 두고, location 결과 후보군에는 영향을 주지 않도록 분리했다.

### 2. 같은 지역 검색의 후보군 크기 고정

기존 location `size`는 줌 레벨에 따라 30, 50, 100, 150, 250, 400, 500으로 달라졌다. 같은 지역을 보고 있어도 줌만 바꾸면 서버에서 잘라오는 후보군 크기가 달라져 결과가 흔들릴 수 있었다.

변경 후 location은 항상 `size=300`을 보낸다. 결과 개수 정책이 지도 줌과 분리되어 캐시와 사용자 경험이 더 예측 가능해졌다.

### 3. "이 지역 검색" 표시 기준이 사용자 체감 거리와 가까워짐

기존 기준은 위도/경도 차이 `0.0001`도로, 위도 기준 약 11m 수준이었다. 지도 조작 중 발생하는 아주 작은 중심 이동에도 "이 지역" 버튼이 뜰 수 있었다.

변경 후 Haversine 거리 기준을 사용하고, 기준값은 `max(300m, radius * 10%)`로 계산한다. 반경 5km에서는 약 500m 이상 움직였을 때만 새 지역으로 본다.

### 4. 반경 검색의 DB/애플리케이션 비용 감소

기존 반경 검색은 DB에서 후보를 모두 가져온 뒤 Java에서 `stream().limit(maxResults)`로 잘랐다. 밀집 지역에서는 실제로 쓰지 않을 행까지 네트워크와 힙에 올라올 수 있었다.

변경 후 `findByRadius` native query가 `LIMIT :limit`을 직접 받는다. DB가 필요한 개수만 반환하므로 전송량과 DTO 변환 비용이 줄어든다.

### 5. 기본 지도 검색 결과의 안정성 개선

기존 기본 정렬은 `distance`였기 때문에 중심 좌표가 조금만 변해도 절단선 근처 장소가 들어오거나 빠질 수 있었다.

변경 후 프론트 기본 정렬은 `stable`이고, UI 라벨은 "추천순"이다. `stable`은 `rating DESC, review_count DESC, idx ASC` 기준이라 같은 반경과 필터에서는 좌표 미세 변화에 덜 흔들린다. 사용자가 거리 기준이 필요하면 기존 `distance`를 선택할 수 있다.

## 완료 기준

1. location 탭에서 지도 줌 변경만으로 주변서비스 API가 재호출되지 않는다.
2. 같은 `searchCenter`, `radius`, `keyword`, `category`, `sort` 조합은 같은 location 캐시 키를 사용한다.
3. location 조회 `size`는 지도 줌 레벨과 무관하게 300으로 고정된다.
4. `"이 지역 검색"` 버튼은 Haversine 거리 기준으로 표시된다.
5. `sort=stable` 요청 시 좌표 미세 변화와 무관한 안정 정렬을 사용한다.
6. 반경 검색은 SQL 레벨에서 `LIMIT`을 적용한다.
7. `./gradlew compileJava`가 통과한다.
8. 프론트 빌드가 가능한 환경이면 `npm run build`가 통과한다.

# 주변 서비스 알고리즘 설계안

**작성일**: 2026-04-12  
**목적**: 기존 코드의 이슈를 바탕으로, 처음부터 설계한다면 어떻게 구성할지 정리한 설계 문서.  
엔티티(`LocationService`) 구조는 현행 유지를 전제로 한다.

---

## 설계 원칙

> **"필터는 DB에서, 정렬은 스코어로, 추천은 명확한 로직으로"**

- **「주변」서비스는 위치(또는 사용자가 고른 지역)를 1차 기준으로 한다.** 키워드·카테고리는 그 안에서 좁히는 축으로 둔다. (아래 §1.1)
- 카테고리·반려동물 등 필터 조건은 전부 SQL WHERE로 처리한다. Java 메모리 필터링은 사용하지 않는다.
- 정렬 기준을 단순 `rating DESC`가 아닌 복합 스코어로 설계한다.
- AI 추천은 결정론적 로직을 fallback으로 두고, LLM은 부가 기능에만 활용한다.

---

## 1. 검색 모드 설계

### 기존 방식의 문제

파라미터 조합으로 `if-else` 분기를 암묵적으로 결정한다. 어떤 모드로 동작하는지 호출부에서 불명확하다.

### 개선 방향: 검색 모드를 명시적으로 모델링

```java
public enum SearchMode {
    KEYWORD,  // 키워드 FULLTEXT
    RADIUS,   // 위도·경도·반경
    REGION    // 지역 계층 (sido > sigungu > eupmyeondong > roadName)
}

public class SearchRequest {
    SearchMode mode;        // 필수: 검색 모드
    String keyword;         // KEYWORD 모드 시
    Double latitude;        // RADIUS 모드 시
    Double longitude;       // RADIUS 모드 시
    Integer radiusMeters;   // RADIUS 모드 시, 기본 10,000m
    String sido;            // REGION 모드 시
    String sigungu;         // REGION 모드 시
    String eupmyeondong;    // REGION 모드 시
    String roadName;        // REGION 모드 시
    SearchFilter filter;    // 공통 필터
    CursorPagination page;  // 커서 기반 페이지네이션
}

public class SearchFilter {
    String category;        // category1/2/3 중 하나 매칭
    Boolean petFriendly;
    Boolean indoor;
    Boolean outdoor;
    String petSize;         // 입장 가능 동물 크기 필터
}
```

컨트롤러에서 파라미터를 받아 `SearchMode`를 명시적으로 결정하고, 서비스는 `SearchRequest` 하나만 받는다.

### 도메인 정책: 「주변」이면 위치가 먼저

제품명이 **주변 서비스**이면 사용자 기대는 보통 **”내 위치(또는 지도 중심) 기준으로 가까운 곳”**이다. 그런데 **키워드만 있을 때 전국 FULLTEXT**로 가면 “주변”과 어긋난다.

**현행(`LocationServiceService.searchLocationServices`)**: `keyword`가 있으면 **키워드 검색이 위도·경도·반경보다 우선**되어, 같은 요청에 좌표가 있어도 반경 한정이 적용되지 않는다.

**✅ 확정: B. 반경(또는 지역) 먼저, 키워드는 필터**

위치(반경 또는 지역 계층)를 1차 기준으로 후보를 제한한 뒤, 그 안에서 키워드로 좁힌다.  
위치 정보가 없을 때만 키워드 단독 FULLTEXT로 fallback한다.

```
[위치 있음 + 키워드 있음]  →  반경/지역 쿼리 WHERE에 name LIKE '%keyword%' 추가
[위치 있음 + 키워드 없음]  →  반경/지역 쿼리 (기존과 동일)
[위치 없음 + 키워드 있음]  →  FULLTEXT 단독 (fallback)
[위치 없음 + 키워드 없음]  →  전체 평점순 (기존과 동일)
```

**키워드 매칭을 LIKE로 쓰는 이유**: FULLTEXT는 독립 실행 시 인덱스 효율이 좋지만,  
반경/지역으로 이미 후보가 좁혀진 상황에서는 `name LIKE '%keyword%'` 가 더 단순하고 충분하다.  
후보 수가 많아 LIKE 성능이 문제가 되는 경우에는 `MATCH(...) AGAINST(...)` 조건을 AND로 추가하는 방식으로 전환한다.

## 2. 쿼리 설계

### 핵심 원칙

**모든 필터(카테고리, petFriendly, indoor 등)를 SQL WHERE에 포함한다.** DB 조회 후 Java에서 필터링하지 않는다.

### RADIUS 검색 쿼리

```sql
SELECT *,
       ST_Distance_Sphere(location, ST_GeomFromText('POINT(:lat :lng)', 4326)) AS dist
FROM locationservice
WHERE ST_Distance_Sphere(location, ST_GeomFromText('POINT(:lat :lng)', 4326)) <= :radius
  AND is_deleted = 0
  AND (:category IS NULL
       OR category3 = :category
       OR category2 = :category
       OR category1 = :category)
  AND (:petFriendly IS NULL OR pet_friendly = :petFriendly)
  AND (:indoor IS NULL OR indoor = :indoor)
  AND (:outdoor IS NULL OR outdoor = :outdoor)
ORDER BY score DESC, idx DESC   -- 복합 스코어 기준 (아래 3절 참고)
LIMIT :size
```

바운딩 박스(POLYGON) + 원형 이중 필터 구조는 제거한다.  
`ST_Distance_Sphere` 단일 조건으로 단순화하고, 공간 인덱스는 `location` 컬럼에만 유지한다.

> **바운딩 박스 제거 이유**: POLYGON 근사가 고위도에서 원을 완전히 포함하지 않을 위험이 있고, 필터가 두 단계로 분리되어 쿼리 복잡도가 높다. MySQL의 공간 인덱스는 `ST_Distance_Sphere`만으로도 활용 가능하다.

### REGION 검색 쿼리

가장 세밀한 단위(roadName > eupmyeondong > sigungu > sido)를 WHERE 조건으로 사용하고, 카테고리 필터를 동일 쿼리에 포함한다.

```sql
SELECT *
FROM locationservice
USE INDEX (idx_locationservice_eupmyeondong_deleted_rating)
WHERE eupmyeondong = :eupmyeondong
  AND is_deleted = 0
  AND (:category IS NULL
       OR category3 = :category
       OR category2 = :category
       OR category1 = :category)
  AND (:petFriendly IS NULL OR pet_friendly = :petFriendly)
ORDER BY score DESC, idx DESC
LIMIT :size
```

### KEYWORD 검색 쿼리

```sql
SELECT *
FROM locationservice
WHERE MATCH(name, description, category1, category2, category3)
      AGAINST(CONCAT(:keyword, '*') IN BOOLEAN MODE)
  AND is_deleted = 0
  AND (:category IS NULL
       OR category3 = :category
       OR category2 = :category
       OR category1 = :category)
  AND (:petFriendly IS NULL OR pet_friendly = :petFriendly)
ORDER BY score DESC, idx DESC
LIMIT :size
```

> **§1.1과의 연결**: 「주변」을 위치 우선으로 둘 경우, 위 쿼리는 **위치 정보가 없을 때의 키워드 전용** 예시로 두고, 위도·경도·반경이 있으면 `ST_Distance_Sphere(...) <= :radius`(및 필요 시 동일 필터)를 **AND**로 붙인 **키워드+반경 결합 쿼리**로 정의한다.

## 3. 정렬 스코어 설계

### 기존 방식의 문제

단순 `rating DESC`는 리뷰 0개인 평점 5.0 장소와 리뷰 500개인 평점 4.8 장소를 구분하지 못한다.

### 복합 스코어 공식

```
score = W1 * rating * log10(reviewCount + 1)
      + W2 * petFriendly_bonus
      + W3 * (1 - dist / maxDist)   // RADIUS 검색 시에만
      + W4 * recency_score           // lastUpdated 기준 최신성
```

| 가중치 | 값(초안) | 이유 |
|---|---|---|
| W1 (평점×신뢰도) | 0.5 | 핵심 품질 지표. 로그로 리뷰 수 폭발 방지 |
| W2 (반려동물 친화) | 0.2 | 플랫폼 특성상 반려동물 친화 장소 우대 |
| W3 (거리 근접도) | 0.2 | 가까울수록 우대. RADIUS 모드에만 적용 |
| W4 (최신성) | 0.1 | 오래된 데이터 자연 하락 |

**스코어 컬럼 운영 방법:**
- `locationservice.score` 컬럼 추가
- 배치 스케줄러(매일 자정)로 전체 재계산
- 리뷰 등록/삭제 시 해당 장소만 즉시 갱신 (이벤트 기반, 아래 5절 참고)

## 4. 페이지네이션 설계

### 기존 방식의 문제

`LIMIT 100 OFFSET 500` 방식은 매 요청마다 500건을 읽고 버린다. 데이터가 많을수록 뒤 페이지일수록 느려진다.

### 커서 기반 페이지네이션 (Keyset Pagination)

```java
public class CursorPagination {
    Double lastScore;   // 이전 페이지 마지막 항목의 score
    Long lastIdx;       // 이전 페이지 마지막 항목의 idx
    int size;           // 페이지 크기 (기본 20)
}
```

```sql
WHERE (score < :lastScore OR (score = :lastScore AND idx < :lastIdx))
ORDER BY score DESC, idx DESC
LIMIT :size
```

응답에 `nextCursor`를 포함하여 클라이언트가 다음 페이지를 요청할 때 전달한다.

```json
{
  "services": [...],
  "count": 20,
  "nextCursor": { "lastScore": 3.42, "lastIdx": 1028 }
}
```

## 5. 평점 갱신 설계

### 기존 원자적 UPDATE 쿼리 유지

```sql
UPDATE locationservice
SET rating = (
  SELECT avg_rating FROM (
    SELECT COALESCE(AVG(r.rating), 0.0) AS avg_rating
    FROM locationservicereview r
    WHERE r.service_idx = :serviceIdx
      AND (r.is_deleted IS NULL OR r.is_deleted = 0)
  ) t
)
WHERE idx = :serviceIdx
```

이 쿼리는 Lost Update 없이 원자적으로 평균을 갱신하므로 유지한다.

### 이벤트 기반으로 분리

리뷰 저장 트랜잭션과 평점 갱신을 분리하여, 평점 갱신 실패가 리뷰 저장을 롤백시키지 않도록 한다.

```java
// 리뷰 서비스
@Transactional
public void createReview(ReviewCreateRequest request) {
    locationServiceReviewRepository.save(review);
    // 트랜잭션 커밋 후 이벤트 발행
    eventPublisher.publishEvent(new ReviewCreatedEvent(review.getServiceIdx()));
}

// 평점 갱신 리스너
@TransactionalEventListener(phase = TransactionPhase.AFTER_COMMIT)
public void handleReviewCreated(ReviewCreatedEvent event) {
    locationServiceRepository.updateRatingByAvg(event.getServiceIdx());
    // score 컬럼도 함께 갱신
    locationServiceRepository.updateScoreByServiceIdx(event.getServiceIdx());
}
```

## 6. 데이터 적재 설계 (CSV 배치 임포트)

### 기존 방식의 문제

`saveBatch`가 `private @Transactional(REQUIRES_NEW)`로 선언되어 있어 Spring AOP 프록시가 작동하지 않는다. `REQUIRES_NEW`(배치 단위 독립 커밋) 의도가 실제로는 적용되지 않는다.

### 개선 방향

배치 저장 로직을 별도 빈으로 분리한다.

```java
@Service
public class LocationServiceBatchWriter {

    private final LocationServiceRepository locationServiceRepository;

    @Transactional(propagation = Propagation.REQUIRES_NEW, rollbackFor = Exception.class)
    public int saveBatch(List<LocationService> batch) {
        if (batch.isEmpty()) return 0;
        locationServiceRepository.saveAll(batch);
        return batch.size();
    }
}
```

`PublicDataLocationService`는 `LocationServiceBatchWriter`를 주입받아 호출한다. 이렇게 하면 AOP 프록시를 통해 `REQUIRES_NEW`가 실제로 적용된다.

배치 사이즈는 고정값 대신 설정값으로 외부화한다.

```properties
app.location.import.batch-size=1000
```

## 7. AI 추천 설계 재검토

### 기존 방식의 문제

Ollama LLM에 30개 후보를 넘겨 JSON으로 재순위화하는 방식은 다음 문제가 있다.
- 타임아웃(30초) 시 fallback 처리 필요
- JSON 파싱 실패 가능성 (코드펜스, 서두 문장 등)
- LLM 서버 의존성 — LLM이 내려가면 추천 기능 전체가 degraded 상태

### 개선 방향: 2단계 분리

**1단계 — 규칙 기반 스코어 정렬 (항상 동작)**

섹션 3의 복합 스코어를 사용한다. 사용자의 반려동물 프로필을 가져올 수 있다면 petSize 매칭 보너스를 추가한다.

```
사용자 반려동물 크기 = "소형"
→ petSize 컬럼에 "소형" 포함된 장소 score + 가중치
```

**2단계 — AI 개인화 (선택적, 비동기)**

LLM 활용 범위를 재정의한다.

| 용도 | 적합 | 이유 |
|---|---|---|
| 정렬 재순위화 | 부적합 | 결과가 비결정론적, 검증 어려움 |
| 장소 설명 요약/생성 | 적합 | 품질 향상, 실패해도 기존 description fallback |
| 사용자 리뷰 요약 | 적합 | 리뷰 다수일 때 핵심 키워드 추출 |

추천 엔드포인트는 1단계 스코어 정렬 결과를 기본 반환하고, 2단계 AI 결과는 비동기로 응답에 포함하거나 별도 엔드포인트로 분리한다.

## 전체 흐름 요약

```
클라이언트 요청
    ↓
SearchRequest 생성 (모드 명시적 결정, 필터 포함)
    ↓
단일 쿼리 (필터 + 공간/FULLTEXT + score 정렬 + 커서 페이지네이션)
    ↓
결과 반환 (nextCursor 포함)

--- 백그라운드 ---
리뷰 이벤트 → rating·score 갱신 (AFTER_COMMIT)
배치 스케줄러 → 전체 score 재계산 (매일 자정)
```

## 현행 코드 대비 핵심 변경점 요약

| 항목 | 현행 | 설계안 |
|---|---|---|
| 검색 모드 결정 | if-else 암묵적 분기 | SearchMode 명시적 열거 |
| 키워드 vs 위치 우선 | 키워드 있으면 **위치·반경 무시** | **위치(또는 지역)를 1차 기준**으로 두고 키워드·카테고리 결합(§1.1) |
| 카테고리 필터 위치 | Java 메모리 스트림 | SQL WHERE 조건 |
| 반경 쿼리 구조 | POLYGON bbox + ST_Distance_Sphere | ST_Distance_Sphere 단일 조건 |
| 정렬 기준 | rating DESC | 복합 스코어 (rating × log(reviewCount) + ...) |
| 페이지네이션 | LIMIT offset | 커서 기반 (Keyset) |
| 평점 갱신 | 직접 호출 | 이벤트 기반 AFTER_COMMIT |
| 배치 트랜잭션 | private @Transactional (AOP 미작동) | 별도 빈 public 메서드 |
| AI 추천 | LLM 정렬 재순위화 | 규칙 기반 fallback + AI는 설명/요약에 한정 |

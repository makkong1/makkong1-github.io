# 주변서비스 후속 쿼리 리팩토링

## 배경

- 반경 검색 기본 정렬은 `distance`로 전환했고, `rating`, `reviews` 정렬 옵션도 추가했다.
- 현재 `distance` / `rating` 정렬은 `locationservice` 단일 테이블 기준이라 비교적 단순하다.
- 반면 `reviews` 정렬은 후보 행마다 `locationservicereview`를 다시 세는 상관 서브쿼리라 데이터가 커질수록 비용이 빠르게 증가할 수 있다.(SpringDataJpaLocationServiceRepository.java (line 63) > findByRadius(...) > ORDER BY의 이 부분)
- 카테고리 필터도 `category1 OR category2 OR category3` 구조라 메인 검색 쿼리 가독성과 확장성이 떨어진다.

## 현재 상태

### 1. reviews 정렬

- 반경 검색 `ORDER BY`에 리뷰 수 집계 서브쿼리가 들어간다.
- 동작은 가능하지만, 후보 수만큼 리뷰 테이블을 반복 스캔할 수 있다.
- 실제 스키마와 soft delete 컬럼 정합성이 깨지면 기능이 바로 실패한다.

### 2. category OR 필터

- 메인 location 검색은 `category1`, `category2`, `category3`를 모두 OR로 비교한다.
- 공공데이터 계층 구조를 그대로 보존한 장점은 있지만, 검색용 대표 카테고리 관점에서는 비효율적이다.

## 문제

1. `reviews` 정렬은 상관 서브쿼리 비용이 커서 트래픽 증가 시 가장 먼저 병목이 될 가능성이 높다.
2. `reviews` 정렬은 스키마 드리프트에 취약하다.
3. `category1 OR category2 OR category3`는 메인 검색 쿼리를 길고 읽기 어렵게 만든다.
4. 카테고리 검색 UX는 단일 대표 카테고리처럼 보이는데, DB 조건은 3개 컬럼 분기라 계약이 불필요하게 복잡하다.

## 목표

- `reviews` 정렬 비용을 줄이고 스키마 의존성을 단순화한다.
- 메인 검색용 카테고리 조건을 대표 카테고리 기준으로 단순화한다.
- 현재 사용자 계약은 유지하면서 쿼리 가독성과 유지보수성을 높인다.

## 후속 방향

### 1. review_count 캐시 컬럼 적용

- `locationservice.review_count` 캐시 컬럼을 추가한다.
- 리뷰 생성/수정/삭제 시 평균 평점 갱신과 함께 리뷰 수를 원자적으로 반영한다.
- 집계 규칙은 현재 쿼리와 동일하게 `soft delete 제외`를 기본 계약으로 둔다.
- 즉 `review_count`는 `(is_deleted IS NULL OR is_deleted = 0)` 조건을 만족하는 리뷰만 센다.
- 이후 숨김 리뷰 같은 상태가 도입되면, 같은 집계 계약에 포함할지 별도 규칙으로 문서화한다.
- `reviews` 정렬은 상관 서브쿼리 대신 `review_count DESC, distance ASC, rating DESC, idx ASC`로 단순화한다.
- 마이그레이션은 `backend/main/resources/sql/migration/locationservice-add-review-count-column.sql`에서 수행한다.

### 2. 대표 카테고리 컬럼 검토

- `category3 > category2 > category1` 우선순위로 대표 카테고리 컬럼을 도입한다.
- 메인 반경 검색과 메인 location 필터는 대표 카테고리 컬럼만 사용한다.
- 원본 `category1~3` 컬럼은 공공데이터 원형 보존용으로 유지한다.
- 기존 데이터는 SQL/배치로 대표 카테고리를 한 번 백필한다.
- 이후 동기화는 우선 애플리케이션/배치 기준으로 유지하고, DB 트리거 도입은 후순위로 둔다.

### 3. sort별 쿼리 분리 검토

- 현재는 `findByRadius(...)`의 `ORDER BY`에서 `sort` 값에 따라 분기한다.
- 장기적으로는 `distance`, `rating`, `reviews`마다 정렬이 고정된 전용 네이티브 쿼리로 분리하는 편이 실행 계획 예측성과 가독성에 유리하다.
- `review_count` 캐시 컬럼 도입 후에는 정렬별 쿼리 분리가 더 단순해진다.

### 4. 중간안: 집계 후 JOIN

- `review_count` 컬럼 추가가 당장 부담이면, 반경 후보에 대해 `service_idx, count(*)`를 먼저 집계한 뒤 JOIN하는 중간안도 가능하다.
- 이 방식은 현재 상관 서브쿼리보다 대체로 낫지만, 장기적으로는 비정규화 캐시 컬럼 전략이 더 적합하다.

### 5. 문서 참조 기준 정리

- 후속 문서에서는 줄 번호 대신 `SpringDataJpaLocationServiceRepository.findByRadius(...) ORDER BY`처럼 메서드/쿼리 이름 기준으로 참조한다.
- 줄 번호는 리팩토링 중 쉽게 바뀌므로 장기 보존 문서에는 부적합하다.

## 비범위

- 개인화 추천 정렬
- 거리/평점 혼합 점수 알고리즘
- 지역 탐색 모드 전체 재설계

## 검증 기준

- `reviews` 정렬이 `review_count` 캐시 기준으로 동작한다.
- 메인 검색 카테고리 조건이 대표 카테고리 단일 비교로 단순화된다.
- 기존 location 메인 UX의 정렬 옵션과 카테고리 필터 계약은 유지된다.

## 1. N+1 · 불필요 로딩

### 1-1. 목록 API에서 `participants` 접근

**현상:** 여러 목록 흐름이 `MeetupConverter#toDTO` / `#toDTOList`를 쓰고, 그 안에서 항상 `meetup.getParticipants()`를 호출한다.

**파일·코드:**

- `backend/main/java/com/linkup/Petory/domain/meetup/converter/MeetupConverter.java` — `toDTO` 내부 `getParticipants()` → 스트림 변환

**배경:** `Meetup.participants`는 `@OneToMany` LAZY + `@BatchSize(size = 50)` (`Meetup.java`). 목록 n건이면 batch로 추가 쿼리가 나가며, **UI에 participants가 필요 없어도** 로딩된다.

**목록 조회가 이 변환을 쓰는 서비스 예 (같은 컨버터):**

- `getAllMeetups` / `getAllMeetups(Pageable)` → `converter.toDTO` / `toDTOList`
- `getMeetupsByLocation`, `searchMeetupsByKeyword`, 주최자별 목록 등 — `MeetupService`에서 `converter.toDTOList` 호출부 참고

**상태:** **미반영.** 권장 방향은 문서 초안과 동일 — 목록 전용 DTO·`toSummaryDTO`(participants 미포함) 또는 “이미 participants가 초기화된 경우에만 매핑” 분기.

## 3. 동시성

### 4-1. `joinMeetup` (반영 완료)

**파일:** `MeetupService.java` — `#joinMeetup`

1. **`meetupRepository.findByIdWithLock(meetupIdx)`** — `SpringDataJpaMeetupRepository#findByIdWithLock` (`@Lock(PESSIMISTIC_WRITE)`). 동일 모임 행에 대한 동시 참가 요청을 직렬화.
2. **`existsByMeetupIdxAndUserIdx`** — 락 이후 중복 여부 확인.
3. **`incrementParticipantsIfAvailable`** — 정원·상태 조건을 WHERE에 넣은 원자적 UPDATE.
4. **`save(participant)`** — `try/catch DataIntegrityViolationException`: 복합 PK 충돌 시 **`decrementParticipantsIfPositive`** 후 `alreadyJoined` (이중 클릭·극단 동시성에 대한 보험).

초안 문서의 “`findByIdWithOrganizer`만 쓰는 TOCTOU” 시나리오는 **현재 코드와 불일치** — 락 조회로 대체됨.

### 4-2. 정원 초과

`incrementParticipantsIfAvailable` / `decrementParticipantsIfPositive` — 타 사용자 동시 참여·취소에 대한 원자적 카운트 (`SpringDataJpaMeetupRepository`).

## 추가 발견점 (2차 분석 — 2026-05-09)

### [신규 A1-1] `Meetup.organizer` — `fetch = FetchType.LAZY` 미선언

**파일:** `Meetup.java:45-47`

```java
@ManyToOne                              // fetch 미선언 → EAGER 기본값
@JoinColumn(name = "organizer_idx", nullable = false)
private Users organizer;
```

**문제:** EAGER 기본값이므로 organizer를 직접 JOIN FETCH하지 않는 쿼리에서도 Hibernate가 `organizer`를 별도 SELECT로 로드한다. 현재 대부분의 쿼리가 `JOIN FETCH m.organizer`를 명시하고 있어 실질 피해는 제한적이지만, 명시 누락 쿼리(`findByIdWithLock` 등)에서 N+1이 잠재한다.

**권장:**

```java
@ManyToOne(fetch = FetchType.LAZY)
@JoinColumn(name = "organizer_idx", nullable = false)
private Users organizer;
```

---

### [신규 A1-2] `MeetupParticipants.meetup` · `.user` — LAZY 미선언 + N+1

**파일:** `MeetupParticipants.java:23-29`

```java
@Id
@ManyToOne                              // EAGER 기본값
@JoinColumn(name = "meetup_idx", nullable = false)
private Meetup meetup;

@Id
@ManyToOne                              // EAGER 기본값
@JoinColumn(name = "user_idx", nullable = false)
private Users user;
```

**문제:**

`findByMeetupIdxOrderByJoinedAtAsc` 쿼리 (`SpringDataJpaMeetupParticipantsRepository.java:21-22`):

```java
@Query("SELECT mp FROM MeetupParticipants mp JOIN FETCH mp.user WHERE mp.meetup.idx = :meetupIdx ...")
```

- `mp.user`는 JOIN FETCH 있음 → 정상.
- `mp.meetup`은 JOIN FETCH 없음. `meetup`이 EAGER이므로 Hibernate가 참여자 N명에 대해 **동일한 meetup 행을 N번 별도 SELECT**한다.
- 예: 참여자 50명 → meetup SELECT 50회 (모두 동일 row).

**권장:**

```java
@ManyToOne(fetch = FetchType.LAZY)
@JoinColumn(name = "meetup_idx", nullable = false)
private Meetup meetup;

@ManyToOne(fetch = FetchType.LAZY)
@JoinColumn(name = "user_idx", nullable = false)
private Users user;
```

그리고 `findByMeetupIdxOrderByJoinedAtAsc`에 `JOIN FETCH mp.meetup` 추가:

```java
@Query("SELECT mp FROM MeetupParticipants mp JOIN FETCH mp.meetup JOIN FETCH mp.user WHERE mp.meetup.idx = :meetupIdx ...")
```

---

### [신규 A1-3] `findByIdWithLock` — organizer JOIN FETCH 없음

**파일:** `SpringDataJpaMeetupRepository.java:97-99`

```java
@Query("SELECT m FROM Meetup m WHERE m.idx = :idx AND (m.isDeleted = false OR m.isDeleted IS NULL)")
Optional<Meetup> findByIdWithLock(@Param("idx") Long idx);
```

**문제:** `joinMeetup` (MeetupService.java:320)에서 `meetup.getOrganizer().getIdx()`를 접근한다. 현재 `organizer`가 EAGER이므로 Hibernate가 별도 SELECT를 내지만, 위 A1-1 수정(`LAZY` 적용) 후에는 **LazyInitializationException** 또는 추가 쿼리가 발생한다.

**권장:**

```java
@Lock(LockModeType.PESSIMISTIC_WRITE)
@Query("SELECT m FROM Meetup m JOIN FETCH m.organizer WHERE m.idx = :idx AND (m.isDeleted = false OR m.isDeleted IS NULL)")
Optional<Meetup> findByIdWithLock(@Param("idx") Long idx);
```

---

### [신규 A2] `toDTO` — Hibernate lazy 컬렉션 proxy가 null 체크를 통과

**파일:** `MeetupConverter.java:27-31`

```java
List<MeetupParticipants> participants = meetup.getParticipants();
if (participants != null) {             // Hibernate proxy는 null이 아님 → 항상 통과
    participantsDTO = participants.stream()  // 여기서 lazy load 트리거
            .map(...)
            .collect(...);
}
```

**문제:** `findByIdWithOrganizer`로 로드된 meetup(`updateMeetup` 등)은 `participants`를 JOIN FETCH하지 않는다. `meetup.getParticipants()`는 `null`이 아닌 Hibernate 초기화 전 proxy를 반환하므로 `if (participants != null)` 체크를 통과하고, `.stream()` 호출 시 lazy load가 트리거된다.

- `createMeetup`: builder로 생성 → `participants` 필드가 literal `null` → null 체크 통과로 안전.
- `updateMeetup` (L185) · `getMeetupsByLocation` · `searchMeetupsByKeyword` 등 목록 경로: lazy proxy → 추가 쿼리 발생.

**권장:** 우선순위 4번(목록용 `toSummaryDTO` 분리)와 함께 해결. 단기 패치로는:

```java
if (participants != null && Hibernate.isInitialized(participants)) {
```

## 실제 수정 내역 (커밋 `adf3bc4` — 2026-05-09)

### 수정 1. `Meetup.organizer` EAGER → LAZY

**문제:** `Meetup.organizer`에 `fetch` 속성을 명시하지 않아 JPA 기본값인 EAGER로 동작했다. EAGER이면 organizer를 JOIN FETCH하지 않는 쿼리(`findByIdWithLock` 등)에서도 Hibernate가 organizer를 별도 SELECT로 자동 로드한다. 의도하지 않은 추가 쿼리가 발생하고, 나중에 LAZY로 바꿀수록 영향 범위가 커진다.

**수정:** `Meetup.java:45`

```java
// before
@ManyToOne
@JoinColumn(name = "organizer_idx", nullable = false)
private Users organizer;

// after
@ManyToOne(fetch = FetchType.LAZY)
@JoinColumn(name = "organizer_idx", nullable = false)
private Users organizer;
```

---

### 수정 2. `MeetupParticipants.meetup` · `.user` EAGER → LAZY

**문제:** `MeetupParticipants`의 두 `@ManyToOne` 필드(meetup, user)도 fetch 미선언으로 EAGER였다. 특히 `findByMeetupIdxOrderByJoinedAtAsc`는 `JOIN FETCH mp.user`만 있고 `mp.meetup`은 없었는데, meetup이 EAGER이므로 Hibernate가 참여자 N명에 대해 **같은 meetup 행을 N번 중복 SELECT**했다. 예: 참여자 50명 → meetup SELECT 50회.

**수정:** `MeetupParticipants.java:23,28`

```java
// before
@Id
@ManyToOne
@JoinColumn(name = "meetup_idx", nullable = false)
private Meetup meetup;

@Id
@ManyToOne
@JoinColumn(name = "user_idx", nullable = false)
private Users user;

// after
@Id
@ManyToOne(fetch = FetchType.LAZY)
@JoinColumn(name = "meetup_idx", nullable = false)
private Meetup meetup;

@Id
@ManyToOne(fetch = FetchType.LAZY)
@JoinColumn(name = "user_idx", nullable = false)
private Users user;
```

---

### 수정 3. `findByMeetupIdxOrderByJoinedAtAsc` — meetup JOIN FETCH 추가

**문제:** 수정 2에서 meetup을 LAZY로 바꿨으므로, 해당 쿼리에서 meetup을 명시적으로 로드하지 않으면 이후 접근 시 LazyInitializationException이 발생한다. 또한 N번 중복 SELECT를 한 번의 JOIN으로 대체할 수 있다.

**수정:** `SpringDataJpaMeetupParticipantsRepository.java:21`

```java
// before
@Query("SELECT mp FROM MeetupParticipants mp JOIN FETCH mp.user " +
       "WHERE mp.meetup.idx = :meetupIdx ORDER BY mp.joinedAt ASC")

// after
@Query("SELECT mp FROM MeetupParticipants mp JOIN FETCH mp.meetup JOIN FETCH mp.user " +
       "WHERE mp.meetup.idx = :meetupIdx ORDER BY mp.joinedAt ASC")
```

### 수정 4. `findByIdWithLock` — organizer JOIN FETCH 추가

**문제:** `joinMeetup`에서 비관적 락으로 meetup을 조회한 뒤 `meetup.getOrganizer().getIdx()`를 접근한다. 수정 1에서 organizer가 LAZY가 됐으므로, JOIN FETCH 없이 그냥 조회하면 organizer 접근 시 추가 SELECT가 발생한다(또는 트랜잭션 범위 밖이면 LazyInitializationException).

**수정:** `SpringDataJpaMeetupRepository.java:98`

```java
// before
@Query("SELECT m FROM Meetup m " +
       "WHERE m.idx = :idx AND (m.isDeleted = false OR m.isDeleted IS NULL)")

// after
@Query("SELECT m FROM Meetup m JOIN FETCH m.organizer " +
       "WHERE m.idx = :idx AND (m.isDeleted = false OR m.isDeleted IS NULL)")
```

### 수정 5. `getAllMeetups()` no-arg — `@Deprecated` 처리

**문제:** `MeetupController`는 `getAllMeetups(Pageable)`만 호출하고 인자 없는 `getAllMeetups()`는 호출하지 않는다. 페이징 없이 전체를 조회하는 메서드가 외부에서 호출 가능한 상태로 남아 있으면 실수로 쓰일 위험이 있다.

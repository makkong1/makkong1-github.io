## 🎯 핵심 성과 후보

### 1️⃣ N+1 문제 해결 (배치 조회 패턴) ⭐⭐⭐ **최우선 추천**

**발생 도메인:**

- ✅ **Chat 도메인**: 채팅방 목록 조회 (`ConversationService.getMyConversations`; 로그인 직후 프론트가 로드하는 시나리오 — 인증/로그인 코드가 아니라 Chat 조회 쿼리다)
- ✅ **Board 도메인**: 게시글 목록 조회 시 반응 정보

**문제 상황:**

- Chat: 채팅방 N개 조회 시 21개 쿼리 발생 (이론상 31개)
- Board: 게시글 1000개 조회 시 2001개 쿼리 발생

**해결 방법:**

- **배치 조회 패턴**: `IN` 절을 사용한 일괄 조회
- **Fetch Join**: 연관 엔티티 함께 조회
- **Map 기반 매핑**: 메모리에서 빠른 데이터 매핑

**개선 효과:**

- **Chat**: 21개 → 4개 쿼리 (**80.95% 개선**)
  - 실행 시간: 305ms → 55ms (**81.97% 단축**)
  - 메모리: 0.58MB → 0.13MB (**77.24% 감소**)
- **Board**: 301개 → 3개 쿼리 (**99% 개선**, 100개 게시글 실측)
  - 실행 시간: 745ms → 30ms (**24.8배 개선**)
  - 메모리: 22.5MB → 2MB (**91% 감소**)
  - 확장성: 게시글 수 무관, 쿼리 수 3개로 고정 (1000개 기준 이론상 2001개 → 3개)
  - 병렬화: 좋아요/댓글/조회수 3개 카운트 배치를 `CompletableFuture.supplyAsync` + `allOf`로 **병렬 실행** 후 `Map<Long, BoardCounts>`로 통합 (`board/service/BoardPopularityService.fetchBoardCountsInParallel`)

**적용 패턴:**

```java
// 1. ID 목록 추출
List<Long> ids = entities.stream().map(Entity::getId).collect(Collectors.toList());

// 2. 배치 조회 (IN 절)
@Query("SELECT e FROM Entity e WHERE e.id IN :ids")
List<Entity> findByIds(@Param("ids") List<Long> ids);

// 3. Map으로 변환하여 메모리에서 매핑
Map<Long, Entity> entityMap = entities.stream()
    .collect(Collectors.toMap(Entity::getId, e -> e));
```

**이력서/포트폴리오 표현:**

> **N+1 문제를 배치 조회 패턴으로 해결하여 쿼리 수 대폭 감소**
>
> - Board 도메인: 301개 → 3개 쿼리 (99% 개선), 실행 시간 745ms → 30ms (24.8배), 메모리 22.5MB → 2MB (91% 감소)
> - Chat 도메인: 21개 → 4개 쿼리 (80.95% 개선), 실행 시간 81.97% 단축
> - **공통 해결 패턴**: IN 절 배치 조회 + Fetch Join + Map 기반 매핑
> - **확장성**: 게시글 수 무관하게 쿼리 수 3개로 고정 (100개→1000개 게시글에도 동일)

### 2️⃣ 동시성 제어 — 시나리오별 전략 의사결정 ⭐ 대표 심화 사례

> 성격: 성능이 아니라 **데이터 정합성(correctness)**. 동시성으로 인원 초과·Lost Update·중복 생성 같은 **잘못된 결과**를 재현하고 막은 작업.
> 핵심은 "버그 몇 개 고침"이 아니라 **8개 시나리오를 재현하고 시나리오 특성에 따라 4전략을 구분 적용한 의사결정**이다.
> 상세·검증·벤치마크: [`docs/concurrency/concurrency-strategy-master.md`](./concurrency/concurrency-strategy-master.md)

**전략 선택 프레임워크 (제일 중요):**

> **"현재 값을 읽어서 검증/분기해야 하는가?"**
> - 값 읽고 판단해야 함(잔액 부족, 두 주체 상태) → **비관적 락** `FOR UPDATE`
> - 단순 조건부 증가(인원 < 최대, 카운터+1) → **원자적 조건부 UPDATE**
> - 유일성 보증(닉네임, 소셜계정) → **DB Unique 제약**
> - 부분 실패 시 전체 롤백 → **트랜잭션 경계**

**검증된 시나리오** (CountDownLatch 동시성 테스트 통과, 로컬 MySQL 8 · 2026-07-07 실행 24 tests 0 fail):

| 시나리오 | 문제 (정합성 붕괴) | 선택 전략 | 상태 |
|----------|---------------------|-----------|------|
| PetCoin 잔액 | Lost Update (동시 충전 5×10=예상150 → 실제110) | 비관적 락 `findByIdForUpdate` | ✅ |
| Meetup 참가 | 최대 3명인데 동시 요청으로 **4명 참가**(제약 위반) | **원자적 조건부 UPDATE** `WHERE cur<max` | ✅ |
| Care 거래 확정 | 격리수준 탓 로직 skip → OPEN에 멈춤(stuck state) | 비관적 락 (Conversation) | ✅ |
| 경고 횟수 | 동시 부여 시 Lost Update | 원자적 UPDATE `+1` | ✅ |
| 닉네임/가입 중복 | 동시 가입 중복 생성 | DB Unique 제약 | ✅ (예외처리 개선여지) |
| 소셜 로그인 중복 | 동시 최초 로그인 중복 계정 | DB Unique 제약(backstop) | 🟡 무결성 O, 패자 예외 미처리 |

**추가 적용 패턴** (코드엔 반영, 전용 동시성 테스트는 미작성 → 과대주장 안 함):
Chat 읽지않음 수 원자적 UPDATE, Board 반응 DB Unique, Report 중복신고 DB Unique — 위 프레임워크의 동일 원리를 확장 적용.

**정직하게 분리한 미해결** (본문 성과 아님, 면접용 "여기까지 파악" 카드):
Refresh Token 동시 갱신(탐색 테스트, 설계 결정 필요), 게시글 삭제 중 댓글 추가(문제 식별 단계).

> ⚠️ **정정**: Meetup은 Pessimistic Lock이 아니라 **원자적 조건부 UPDATE**로 해결(비관적 락은 §부록 비교 대상). 벤치마크 실측상 저경합에선 비관적 락이 오히려 빨라, 원자적 UPDATE 선택 이유는 속도가 아니라 **락/데드락 관리 부담 제거·코드 단순성**임을 데이터로 확인.

**적용 패턴:**

```java
// 1. 원자적 조건부 UPDATE (Meetup) — 체크+증가를 DB 한 문장으로 원자화
@Modifying
@Query("UPDATE Meetup m SET m.currentParticipants = m.currentParticipants + 1 " +
       "WHERE m.idx = :idx AND m.currentParticipants < m.maxParticipants")
int incrementParticipantsIfAvailable(@Param("idx") Long idx); // updated==0 → 만석 예외

// 2. 비관적 락 (PetCoin 잔액) — 부족 검증이 필요해 현재값을 읽고 직렬화
@Lock(LockModeType.PESSIMISTIC_WRITE)
@Query("SELECT u FROM Users u WHERE u.idx = :idx")
Optional<Users> findByIdForUpdate(@Param("idx") Long idx);

// 3. 원자적 증가 (경고 횟수) — 증가만 하는 카운터
@Modifying
@Query("UPDATE Users u SET u.warningCount = u.warningCount + 1 WHERE u.idx = :userId")
int incrementWarningCount(@Param("userId") Long userId);
```

**이력서/포트폴리오 표현:**

> **동시성으로 인한 데이터 정합성 붕괴를 재현하고, 시나리오 특성에 따라 전략을 구분 적용**
>
> - 판단 기준: "현재 값 검증이 필요한가" → 필요하면 비관적 락, 단순 조건부 증가면 원자적 UPDATE, 유일성은 DB Unique
> - Meetup: 인원 초과(3명 제한에 4명)를 **원자적 조건부 UPDATE**로 구조적으로 차단
> - PetCoin: 잔액 부족 검증이 필요해 **비관적 락**으로 Lost Update 방지
> - Care: 격리수준 탓 로직이 skip되는 **stuck state**를 상위 엔티티 락으로 해결
> - 두 전략을 실측 비교해 "선택 이유가 속도가 아니라 운영 단순성"임을 데이터로 확인 (측정으로 통념 반증)

### 3️⃣ 성능 최적화 (인덱스 전략) ⭐

**발생 도메인:**

- ✅ **전체 도메인**: 조회 성능 개선

**문제 상황:**

- 복합 조건 조회 시 테이블 스캔 발생
- 정렬 시 FileSort 발생
- 페이징 시 성능 저하

**해결 방법:**

- **복합 인덱스**: 카테고리 + 삭제 여부 + 생성일 정렬
- **커버링 인덱스**: 필요한 컬럼만 인덱스에 포함
- **Full-Text 인덱스**: 검색 성능 향상

**적용 예시:**

```sql
-- 복합 인덱스
CREATE INDEX idx_board_category_deleted_created
ON board(category, is_deleted, created_at DESC);

-- 커버링 인덱스
CREATE INDEX idx_board_list_covering
ON board(category, is_deleted, created_at DESC, idx, title, user_idx);

-- Full-Text 인덱스
CREATE FULLTEXT INDEX idx_board_fulltext
ON board(title, content);
```

**이력서/포트폴리오 표현:**

> **복합 인덱스 전략으로 조회 성능 개선**
>
> - 카테고리별 조회: 테이블 스캔 → 인덱스 스캔 (FileSort 제거)
> - 커버링 인덱스: 테이블 접근 없이 인덱스만으로 쿼리 완성
> - Full-Text 인덱스: 검색 성능 향상

### 4️⃣ 인메모리 필터링 → DB 쿼리 전환 (Meetup 근처 모임 조회) ⭐

> ⚠️ **세대 주의 (2026-07 갱신)**: 아래 Before/After 수치(486→273ms, 스캔 2,958→117행, 96%)는 **Gen 1 (B-tree BETWEEN bounding box)** 측정치다. **현재 코드는 한 세대 더 나아가** `findNearbyMeetupIds`가 **공간 인덱스** `ST_Within(POLYGON, SRID 4326)` + `ST_Distance_Sphere` 정밀 반경 필터 + 거리순 `LIMIT`로 재구현돼 있다(`meetup/repository/SpringDataJpaMeetupRepository.java`, 마이그레이션 `meetup-spatial-location.sql`). 면접에선 "초기엔 bounding box로 스캔을 96% 줄였고, 이후 공간 인덱스로 재구현했습니다"처럼 **세대를 명시**할 것. (세대 지도: `docs/analysis/리팩토링-순서-2026-07.md` §3)

**발생 도메인:**

- ✅ **Meetup 도메인**: 근처 모임 조회 (`getNearbyMeetups`)

**문제 상황:**

- `findAllNotDeleted()`로 전체 모임을 메모리에 로드한 뒤 Java에서 Haversine 거리 계산 및 필터링
- 데이터 증가 시 O(n) 메모리·시간 비례 증가

**해결 방법:**

- DB 쿼리 레벨에서 Bounding Box(위도·경도 범위) 필터링 적용
- 기존 위치 인덱스(`idx_meetup_location`) 활용하여 불필요한 행 스캔 제거

**적용 패턴:**

```java
// 수정 전: 전체 로드 후 메모리 필터링
List<Meetup> all = meetupRepository.findAllNotDeleted();
all.stream().filter(m -> calculateDistance(...) <= radius)...

// 수정 후: DB에서 범위 필터링
List<Meetup> nearby = meetupRepository.findNearbyMeetups(lat, lng, radius, currentDate);
```

**개선 효과 (실측, 1,000개 데이터):**
| 지표 | Before | After | 개선율 |
|------|--------|-------|--------|
| 전체 실행 시간 | 486ms | 273ms | **43.8% 감소** |
| DB 쿼리 시간 | 241ms | 143ms | **40.7% 감소** |
| 메모리 사용량 | 1.48MB | 0.21MB | **85.8% 감소** |
| 스캔 행 수 | 2,958개 | 117개 | **96.0% 감소** |

**이력서/포트폴리오 표현:**

> **인메모리 필터링을 DB 쿼리로 전환하여 근처 모임 조회 성능 개선**
>
> - 전체 데이터 메모리 로드 → Bounding Box 기반 DB 필터링으로 스캔 행 2,958개 → 117개 (96% 감소)
> - 실행 시간 486ms → 273ms (43.8% 단축), 메모리 1.48MB → 0.21MB (85.8% 감소)
> - 기존 위치 인덱스를 IS NOT NULL 조건 제거로 활용 가능하게 쿼리 구조 개선

## 📈 최종 추천 순위

### 🥇 1순위: N+1 문제 해결 (배치 조회 패턴)

**이유:**

- ✅ **2개 이상 도메인에서 해결** (Chat, Board)
- ✅ **정량적 개선 효과 명확** (80% 이상 개선)
- ✅ **확장 가능한 패턴** (다른 도메인에도 적용 가능)
- ✅ **프로젝트 전반을 대표** (JPA 사용 시 흔한 문제)

**이력서 한 줄 요약:**

> N+1 문제를 배치 조회 패턴으로 해결하여 쿼리 수 301개 → 3개 (99% 감소), 실행 시간 745ms → 30ms (24.8배 개선)

**포트폴리오 상세:**

> **문제**: 게시글 목록 조회 시 작성자·반응·첨부파일을 게시글마다 개별 조회하여 301개 쿼리 발생 (100개 게시글 기준)
> **해결**: IN 절 배치 조회 + Fetch Join으로 3개 쿼리로 감소 / 게시글 수 증가와 무관하게 3개 유지
> **효과**: 745ms → 30ms (24.8배), 메모리 22.5MB → 2MB (91% 감소), 동일 패턴을 Chat에도 적용 (21개 → 4개)

---

### 🥈 2순위: 동시성 제어 — 시나리오별 전략 의사결정

**이유:**

- ✅ **8개 시나리오 재현 + 4전략 구분 적용** (비관적 락 / 원자적 UPDATE / DB Unique / 트랜잭션 경계)
- ✅ **데이터 정합성(correctness) 보장** (비즈니스 로직 핵심 — 잘못된 결과 제거)
- ✅ **선택 근거가 프레임워크로 정리됨** ("현재 값 검증이 필요한가")
- ✅ **동시성 테스트로 검증** (24 tests 0 fail) + 전략 벤치마크로 통념 반증
- 💡 정량 수치형 성과(N+1)와 달리 **"판단력·깊이" 어필형** — 백엔드 면접 최다 질문 주제

**이력서 한 줄 요약:**

> 동시성 정합성 문제를 8개 시나리오에서 재현하고, "현재 값 검증 필요 여부" 기준으로 비관적 락·원자적 UPDATE·DB Unique를 구분 적용해 인원 초과·Lost Update·중복 생성을 제거

**포트폴리오 상세:**

> **문제**: 동시 요청으로 인원 초과(3명 제한에 4명), Lost Update, 중복 계정/신고 등 **데이터 정합성 붕괴**
> **해결** (시나리오 특성별 전략 선택):
>
> - 원자적 조건부 UPDATE (Meetup): 체크+증가를 DB 한 문장으로 원자화해 인원 초과 구조적 차단
> - 비관적 락 (PetCoin/Care): 잔액 부족 검증·두 주체 상태 판단처럼 현재값을 읽어야 하는 경우 직렬화
> - DB Unique 제약 (닉네임/소셜): 앱 레벨 사전체크로 못 막는 TOCTOU를 DB가 최종 보증
>   **효과**: 인원 초과·Lost Update·중복 생성 제거. 두 전략 실측 비교로 "선택 이유는 속도가 아니라 운영 단순성"임을 데이터로 확인

### 🥉 3순위: 성능 최적화 (인덱스 전략)

**이유:**

- ✅ **전체 도메인에 적용**
- ⚠️ **단독 성과로는 부족** (N+1, Race Condition과 함께 언급 가능)

**이력서 한 줄 요약:**

### 조합 표현 (더 강력한 성과)

**이력서 표현:**

> **JPA 성능 최적화 및 동시성 제어**
>

**포트폴리오 표현:**

> **프로젝트 전반 성능 및 안정성 개선**
>
> **1. N+1 문제 해결 (배치 조회 패턴)**
>
> - 문제: 게시글 목록 조회 시 작성자·반응·첨부파일을 개별 조회하여 301개 쿼리 발생 (100개 게시글)
> - 해결: IN 절 배치 조회 + Fetch Join + Map 기반 매핑
> - 효과: 301개 → 3개 쿼리 (99% 감소), 745ms → 30ms, 22.5MB → 2MB / 게시글 수 무관 3개 유지
>
> **2. 동시성 제어 (시나리오별 전략 의사결정)**
>
> - 문제: 동시 요청으로 인원 초과(3명 제한에 4명), Lost Update, 중복 계정/신고 등 데이터 정합성 붕괴
> - 해결 (시나리오 특성별 전략):
>   - 원자적 조건부 UPDATE (Meetup): 체크+증가 원자화로 인원 초과 구조적 차단
>   - 비관적 락 (PetCoin/Care): 잔액 부족 검증·두 주체 상태 판단 등 현재값 검증이 필요한 경우
>   - DB Unique 제약 (닉네임/소셜): TOCTOU를 DB가 최종 보증
> - 효과: 잘못된 결과(초과·Lost Update·중복) 제거. 8시나리오 재현 + 동시성 테스트 검증(24 tests 0 fail)

> **3. 인덱스 최적화**
>
> - 복합 인덱스로 FileSort 제거, 커버링 인덱스로 테이블 접근 최소화
>
> **4. 인메모리 필터링 → DB 쿼리 전환 (Meetup)**
>
> - 문제: 전체 모임 메모리 로드 후 Java에서 거리 계산 및 필터링 (O(n))

### 5️⃣ NLP 기반 반려생활 의도 분석 추천 (Spring ↔ petory-nlp-server 연동) ⭐⭐

**발생 도메인:**

- ✅ **petRecommendation 도메인**: 사용자 활동 → 의도 signal → 주변 시설 추천 오케스트레이션
- ✅ **petory-nlp-server (FastAPI, 별도 서비스)**: 한국어 케어 의도 분석 API (`POST /api/pet-intent/analyze`)

**배경 (설계 의도):**

- 위치검색·커뮤니티·케어 활동에서 사용자 "의도"를 추론해 주변 시설을 추천
- 임베딩·형태소 같은 무거운 NLP를 요청 경로에 동기로 넣으면 응답 지연 + 외부 NLP 서버 장애가 본 서비스로 전파될 위험 → 이를 피하는 구조가 핵심

**해결 방법:**

1. **NLP 분리 서버 (FastAPI)**: 임베딩 모델 + intent classifier(rule substring/형태소 exact match → miss 시 centroid dot-product, `confidence_threshold=0.45`) + Kiwi 형태소 태그 추출. `sentence-transformers` 미설치 시 문자 n-gram 해시 fallback으로 기동 보장, lifespan에서 모델·centroid 워밍업(첫 요청 timeout 방지).
2. **비동기 이벤트 연동**: 활동 이벤트를 `@TransactionalEventListener(AFTER_COMMIT)`(커뮤니티/케어)·`@EventListener`(위치검색)로 받아 **전용 스레드풀**(core 2 / max 6 / queue 500, `DiscardWithWarnPolicy`)에서 `PetIntentClient`가 nlp-server 호출 → `UserPetIntentSignal` 저장.
3. **회복탄력성**: NLP 전용 `RestTemplate` **timeout 3s 격리**, 실패 시 `Optional.empty()` **fail-closed**, 위치검색은 자연어 휴리스틱(keyword 길이 ≥ 7 + 공백 포함) + **Redis dedup(`nlp:loc-dedup:*`, TTL 10분)**.
4. **추천 스코어링** (`PetRecommendScoreCalculator`): 거리 0.35 / 평점 0.30 / 리뷰수 0.20 / 30일 로그스케일 인기도 0.10 / 태그매칭 0.05 가중합 → `matchReasons`(nearby/high_rating/many_reviews/popular/tag_match) 산출.
5. **signal 수명 설계** (`UserPetIntentSignalService`): domain × urgency threshold + TTL 차등(MEDICAL+HIGH 1일 … LODGING_TRAVEL 14일), MEDICAL+HIGH는 헬스 알림(`PET_HEALTH_ALERT`) 발행.

**개선 효과:**

- 무거운 NLP를 요청 경로 밖(AFTER_COMMIT + 전용 스레드풀)으로 분리 → 사용자 응답 지연 없음
- NLP 서버 장애가 본 서비스로 전파되지 않음(timeout 격리 + fail-closed)
- Spring ↔ FastAPI 폴리글랏을 이벤트·HTTP 클라이언트 경계로 느슨하게 결합

> **참고**: 옛 `pet-data-api` 폴링(`FacilitySyncService` `GET /facilities`) 기반 시설 인게스천은 제거되었고, 시설 데이터는 현재 공공데이터 **파일 임포트**(`PublicDataLocationService` → `LocationServiceBatchWriter` batch upsert)로 적재된다.

**이력서/포트폴리오 표현:**

> **Spring ↔ FastAPI(NLP) 폴리글랏 추천 파이프라인을 이벤트 기반으로 연동**
>
> - 사용자 활동(위치검색/커뮤니티/케어)을 AFTER_COMMIT 이벤트로 받아 전용 스레드풀에서 NLP 의도 분석 호출 → 요청 경로와 분리
> - NLP 호출 timeout 격리 + 실패 시 fail-closed로 외부 서버 장애 격리, Redis dedup으로 중복 분석 방지
> - 거리·평점·리뷰·인기도·태그 가중합 스코어링 + 의도 urgency별 signal TTL 차등 설계

### 6️⃣ 도메인 순환 의존 제거 (아키텍처 리팩토링) ⭐⭐

**발생 범위:**

- ✅ 15개 도메인 전반의 import 의존성 구조 (user·report·location·payment·statistics·petRecommendation)

**문제 상황:**

- import 의존성 그래프를 추출해 보니 `user`가 leaf가 아니라 **허브**였다 — care/location/meetup/report를 역방향 의존
- 그래프만으로는 안 보이던 `statistics → care → payment → statistics` **3노드 순환**이 호출부에 숨어 있었다
- 특히 `recordPayment`(통계 누적)가 `releaseToProvider`의 **코인 지급과 같은 트랜잭션 안**에 있어, 통계 기록이 실패하면 **실제 코인 지급까지 롤백**되는 잠재 버그 동반 (non-critical 통계가 money 흐름을 인질로 잡음)

**해결 방법 (하나의 처방이 아니라 성격별로 분류):**
| 역방향 의존 성격 | 처방 | 제거된 엣지 |
|------------------|------|-------------|
| 읽기 집계 | `profile` 집계 계층 신설, `UserProfileController`·DTO 이동 | `user→care/location/meetup` |
| 공유 타입(enum) | `ReportActionType`를 `common`으로 추출 | `user→report` (user 완전 leaf화) |
| 이벤트 클래스 소유권 | `LocationSearchPerformedEvent`를 발행자(location) 소유로 이동 | `location→petRecommendation` |
| non-critical 쓰기 | `recordPayment`를 `PaymentRecordedEvent` + `@TransactionalEventListener(AFTER_COMMIT, REQUIRES_NEW)`로 이벤트화 | `payment→statistics` (+ 잠재버그 수정) |
| 데이터 모델 FK + money 흐름 | `care↔payment`는 결제-거래 애그리거트로 수용 | (의도 유지) |

**개선 효과 (import 그래프 재추출로 정량 검증):**

- 역방향·순환 엣지 **6개 제거**, 새로 생긴 엣지는 전부 정방향(A→B)
- `user` = **완전한 leaf**(common·file만 의존), `statistics` = **유입 엣지 0의 순수 집계 도메인**
- `statistics→care→payment→statistics` **3노드 순환 해소**
- **money 트랜잭션 잠재버그 제거**: 통계 집계 실패가 코인 지급을 롤백하던 결합을 AFTER_COMMIT 이벤트로 분리
- `./gradlew compileJava` + 전체 테스트 통과 (구조 이동 회귀 0)
- 근거: `docs/analysis/dependency-graph/{before,after-step1-4}-2026-07-06.txt`

**이력서/포트폴리오 표현:**

> **도메인 import 의존성 그래프를 분석해 순환 의존을 성격별로 분류·해소**
>
> - 집계 계층 분리(profile), 공용 타입 추출(common), 이벤트 역전, 애그리거트 수용을 상황별로 적용
> - "전부 이벤트로"가 아니라 **트랜잭션 필수성**을 기준으로 분리 — non-critical 통계만 이벤트화하며 money 트랜잭션 잠재버그(통계 실패 시 코인 지급 롤백)를 함께 수정
> - `user`를 허브에서 leaf로, `statistics`를 순수 집계 도메인으로 정리 (역방향·순환 엣지 6개 제거, 전체 테스트 회귀 0)

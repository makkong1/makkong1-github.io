# 관리자 대시보드 & 통계 시스템 아키텍처

> 기준 코드: `domain/statistics`, `AdminStatisticsController`, `SystemDashboardSection`, `adminApi`, `PetCoinEscrowService`, `RedisConfig`.

이 문서는 Admin 전체 운영 구조가 아니라 **통계 집계 시스템과 관리자 대시보드 연결**을 설명한다. Admin facade, 감사 로그, 사용자/신고/파일 관리 구조는 `docs/architecture/admin/관리자 운영 아키텍처.md`에서 다룬다.

---

## 1. 전체 구조

```mermaid
flowchart LR
  UI["SystemDashboardSection"]
  APIJS["adminApi"]
  CTRL["AdminStatisticsController<br/>/api/admin/statistics<br/>MASTER only"]
  SVC["StatisticsService"]
  SCH["StatisticsScheduler<br/>00:05 daily batch"]
  AGG["StatisticsAggregator<br/>REQUIRES_NEW daily merge"]
  REDIS["Redis<br/>todayStats::today TTL 1m"]
  DAILY[("dailystatistics")]
  WEEKLY[("weekly_statistics")]
  MONTHLY[("monthly_statistics")]
  SRC["Source domain tables<br/>users / login_events / care_request / board / meetup / report"]
  PAY["PetCoinEscrowService<br/>releaseToProvider"]

  UI --> APIJS
  APIJS --> CTRL
  CTRL --> SVC
  SVC --> DAILY
  SVC --> WEEKLY
  SVC --> MONTHLY
  SVC --> REDIS
  SCH --> AGG
  AGG --> SRC
  AGG --> DAILY
  SCH --> WEEKLY
  SCH --> MONTHLY
  PAY --> SVC
```

핵심 구조:

- 조회 API는 저장된 statistics 테이블을 읽는다.
- 배치 집계는 `StatisticsAggregator`가 source 도메인 repository의 count 쿼리로 daily row를 merge한다.
- weekly/monthly는 대부분 daily row를 합산하지만, `activeUsers`는 `login_events` 기간 내 DISTINCT 사용자 수를 직접 집계한다.
- 에스크로 지급 완료는 배치를 기다리지 않고 `recordPayment()`의 native upsert로 당일 revenue를 즉시 갱신한다.
- 오늘 스냅샷 API는 Redis 1분 캐시를 쓰며, 결제 반영 시 `todayStats::today`를 evict한다.

## 2. API와 권한

```mermaid
sequenceDiagram
  participant UI as Admin Dashboard
  participant API as adminApi
  participant C as AdminStatisticsController
  participant S as StatisticsService
  participant DB as Statistics Tables

  UI->>API: fetchDailyStatistics()
  API->>C: GET /api/admin/statistics/daily
  C->>C: @PreAuthorize hasRole('MASTER')
  C->>S: getDailyStatistics(startDate, endDate)
  S->>DB: findByStatDateBetweenOrderByStatDateAsc
  DB-->>S: DailyStatistics rows
  S-->>C: DailyStatisticsResponse list
  C-->>UI: grouped DTO list
```

`AdminStatisticsController`는 클래스 전체가 `@PreAuthorize("hasRole('MASTER')")`이다. `/api/admin/**` 보안 경로는 ADMIN도 진입 가능한 구조지만, 통계 컨트롤러 메서드 권한은 MASTER로 다시 좁혀진다.

| Method | Path | Service | 현재 프론트 사용 |
| --- | --- | --- | --- |
| GET | `/daily` | `getDailyStatistics` | 사용 |
| GET | `/weekly` | `getWeeklyStatistics` | 미사용 |
| GET | `/monthly` | `getMonthlyStatistics` | 미사용 |
| GET | `/summary` | `getTodaySnapshot` | 미사용 |
| POST | `/backfill` | `backfill` | 사용 |

## 3. 자동 집계 플로우

```mermaid
sequenceDiagram
  participant SCH as StatisticsScheduler
  participant AGG as StatisticsAggregator
  participant Daily as DailyStatisticsRepository
  participant Sources as Source Repositories
  participant Login as LoginEventRepository
  participant Weekly as WeeklyStatisticsRepository
  participant Monthly as MonthlyStatisticsRepository

  Note over SCH: 매일 00:05
  SCH->>Daily: 최근 7일 existing statDate 조회
  loop missing date
    SCH->>AGG: aggregateForDate(date)
    AGG->>Daily: findByStatDate (기존 row 확인)
    AGG->>Sources: 날짜별 count 쿼리
    AGG->>Login: countDistinctUsersBetween (DAU)
    AGG->>Daily: merge 저장 (활동 지표 덮어쓰기, 결제 유지)
  end
  SCH->>AGG: aggregateForDate(yesterday)
  AGG->>Daily: findByStatDate (기존 row 확인)
  AGG->>Sources: yesterday count 쿼리
  AGG->>Login: countDistinctUsersBetween (DAU)
  AGG->>Daily: merge 저장
  alt yesterday is Sunday
    SCH->>Daily: week range daily rows 조회
    SCH->>Weekly: WeeklyStatistics 저장
  end
  alt yesterday is last day of month
    SCH->>Daily: month range daily rows 조회
    SCH->>Monthly: MonthlyStatistics 저장
  end
  SCH->>Daily: now - 1년 이전 daily 삭제
```

집계 시각:

- Cron: `0 5 0 * * ?` (매일 00:05, 구: `0 0 18 * * ?`)
- 기준 날짜: `LocalDate.now().minusDays(1)`
- 일별 보관: 1년 초과 row 삭제

집계 정책:

- daily row가 이미 있으면 **활동 지표를 merge/update한다** (구: skip). 결제 데이터가 있으면 매출 필드는 유지.
- weekly/monthly row가 이미 있으면 다시 만들지 않는다.
- backfill은 날짜별 예외를 잡아서 한 날짜 실패가 전체 범위를 중단하지 않게 한다.
- `StatisticsAggregator` 별도 빈: self-invocation 없이 `@Transactional(REQUIRES_NEW)`가 정상 적용된다.

## 4. 지표 소스

| 그룹 | 지표 | 원천 |
| --- | --- | --- |
| User | 신규 가입 | `Users.createdAt` |
| User | 활성 사용자 | `login_events.login_at` DISTINCT user_id (구: `Users.lastLoginAt`) |
| User | 신규 제공자 | `Users.role = SERVICE_PROVIDER`, `createdAt` |
| Care | 신규 케어 요청 | `CareRequest.createdAt` |
| Care | 완료 케어 | `CareRequest.completedAt` |
| Care | 취소 케어 | `CareRequest.status = CANCELLED`, `updatedAt` |
| Care | 완료율 | `completed / (completed + cancelled)` |
| Payment | 매출 | `PetCoinEscrowService.releaseToProvider()`에서 즉시 반영 |
| Payment | 거래 수/평균 거래액 | `recordPayment()`에서 갱신 |
| Board | 신규 게시글 | `Board.createdAt` |
| Meetup | 신규 모임 | `Meetup.createdAt` |
| Meetup | 모임 참여 | `MeetupParticipants.joinedAt` |
| Report | 신규 신고 | `Report.createdAt` |
| Report | 처리 신고 | `Report.status = RESOLVED`, `updatedAt` |

현재 통계에 없는 지표:

- MissingPet 제보/해결 수
- 댓글/반응 수
- 위치 리뷰/검색 수
- 파일 업로드 수
- 케어 지원 수

## 5. 매출 즉시 반영 플로우

```mermaid
sequenceDiagram
  participant Care as Care completion flow
  participant Escrow as PetCoinEscrowService
  participant Coin as PetCoinService
  participant Stats as StatisticsService
  participant Daily as dailystatistics
  participant Redis as Redis todayStats

  Care->>Escrow: releaseToProvider(escrow)
  Escrow->>Escrow: findByIdForUpdate
  Escrow->>Escrow: escrow.release()
  Escrow->>Coin: payoutCoins(provider, amount)
  Escrow->>Stats: recordPayment(amount)
  Stats->>Daily: INSERT ... ON DUPLICATE KEY UPDATE
  Stats->>Daily: totalRevenue, transactionCount, avgTransaction 원자 갱신
  Stats->>Redis: @CacheEvict(todayStats::today)
  Escrow->>Escrow: escrow 저장
```

이 흐름은 매출을 빠르게 보여주는 장점이 있다.

**동시성 보호 (2026-06-28 수정)**:
- `recordPayment()`는 read-modify-write를 하지 않고 `upsertPayment()` native SQL을 호출한다.
- `INSERT ... ON DUPLICATE KEY UPDATE` 한 문장에서 매출 합계, 거래 수, 평균 거래액을 갱신한다.
- `total_revenue`가 과거 데이터에서 null이어도 `COALESCE(total_revenue, 0)`로 계산한다.
- 결제 반영 후 `todayStats::today` 캐시를 evict한다.

**배치 충돌 해결 (2026-06-28 수정)**:
- 배치는 기존 row가 있어도 skip하지 않고 활동 지표를 항상 merge/update한다.
- 결제 데이터(`transactionCount > 0`)가 있으면 매출 필드는 그대로 두고 활동 지표만 덮어쓴다.

## 6. Redis 캐시

```mermaid
flowchart TD
  C["GET /summary"]
  CACHE{"Redis todayStats::today"}
  SVC["StatisticsService.getTodaySnapshot"]
  DB[("dailystatistics")]

  C --> CACHE
  CACHE -->|hit| C
  CACHE -->|miss| SVC
  SVC --> DB
  SVC --> CACHE
```

| 항목 | 내용 |
| --- | --- |
| 캐시명 | `todayStats` |
| 키 | `'today'` |
| 실제 Redis key 형태 | `todayStats::today` |
| TTL | 1분 |
| 설정 위치 | `RedisConfig.cacheManager()` |
| 사용 API | `GET /api/admin/statistics/summary` |

현재 프론트 대시보드는 `/summary`를 호출하지 않고 `/daily` 응답의 마지막 row를 요약 카드에 사용한다. 따라서 Redis 캐시는 백엔드 API로는 준비되어 있지만, 현 UI의 기본 경로에는 직접 쓰이지 않는다.

## 7. 프론트 대시보드 연결

```mermaid
flowchart LR
  Panel["AdminPanel"]
  Section["SystemDashboardSection"]
  Api["adminApi"]
  Daily["GET /statistics/daily"]
  Backfill["POST /statistics/backfill"]

  Panel --> Section
  Section --> Api
  Api --> Daily
  Api --> Backfill
```

현재 화면 기능:

- 최근 30일 daily 통계 조회
- 요약 카드 표시
- 성장 추이 LineChart
- 서비스 활성화 BarChart
- MASTER 전용 수동 집계 버튼

현재 연동 갭:

| 위치 | 현재 코드 | 백엔드 응답 |
| --- | --- | --- |
| 요약 카드 | `latest.newUsers` | `latest.users.newUsers` |
| 매출 카드 | `latest.totalRevenue` | `latest.revenue.totalRevenue` |
| 라인 차트 | `dataKey="newUsers"` | `users.newUsers` 또는 데이터 평탄화 필요 |
| 바 차트 | `dataKey="newPosts"` | `community.newPosts` |

즉, 현재 구조에서는 프론트에서 중첩 DTO를 평탄화하거나, Recharts `dataKey`를 중첩 경로에 맞춰 바꿔야 한다.

## 8. 저장 모델

```mermaid
erDiagram
  DAILYSTATISTICS {
    bigint id PK
    date stat_date UK
    bigint new_users
    bigint active_users
    bigint new_providers
    bigint new_care_requests
    bigint completed_cares
    bigint cancelled_cares
    decimal care_completion_rate
    decimal total_revenue
    bigint transaction_count
    decimal avg_transaction
    bigint new_posts
    bigint new_meetups
    bigint meetup_participants
    bigint new_reports
    bigint resolved_reports
  }

  WEEKLY_STATISTICS {
    bigint id PK
    int year
    int week_number
    date start_date
    date end_date
    decimal weekly_retention_rate
  }

  MONTHLY_STATISTICS {
    bigint id PK
    int year
    int month
    decimal monthly_retention_rate
    decimal churn_rate
  }
```

주/월 테이블은 daily와 같은 집계 컬럼을 대부분 반복해서 가진다. 주/월 조회 성능은 좋지만, 컬럼 추가 시 세 엔티티와 세 DTO를 함께 수정해야 한다.

## 9. 운영 리스크와 개선 포인트

| 항목 | 현재 상태 | 개선 방향 |
| --- | --- | --- |
| 매출 즉시 반영과 daily skip 충돌 | ✅ 수정됨 (2026-06-28) merge 방식으로 전환 | — |
| Self-invocation @Transactional | ✅ 수정됨 (2026-06-28) StatisticsAggregator 빈 분리 | — |
| recordPayment 레이스 컨디션 | ✅ 수정됨 (2026-06-28) native upsert | — |
| DAU 원천 오류 (lastLoginAt) | ✅ 수정됨 (2026-06-28) login_events DISTINCT | — |
| ISO 53주차 하드코딩 | ✅ 수정됨 (2026-06-28) 동적 계산 | — |
| WAU/MAU = DAU 합산 | ✅ 수정됨 (2026-06-28) 주/월 기간 DISTINCT 집계 | — |
| 프론트 DTO 불일치 | 화면은 평면 필드, API는 중첩 DTO | `adminApi` 변환 계층 또는 화면 dataKey 수정 |
| `/summary` 미사용 | Redis 캐시 API는 있지만 기본 화면 경로는 `/daily` | 요약 카드는 `/summary` 사용 검토 |
| 캐시 무효화 | ✅ 수정됨 (2026-06-28) `recordPayment()` 후 `todayStats` evict | — |
| 통계 범위 | MissingPet/Comment/Location/File 미포함 | 필요한 운영 지표만 선별 추가 |
| 감사 로그 | 통계 backfill 요청 감사 로그 없음 | Admin facade/audit 정책 편입 검토 |

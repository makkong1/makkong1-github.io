# 위치 기반 서비스 아키텍처 (프론트·백엔드 대조)

이 문서는 **Location 도메인(시설 검색·지오코딩·리뷰)** 과, 같은 “위치” 맥락에서 쓰이는 **통합 탐색 지도(UnifiedPetMap)** 를 **프론트엔드 모듈 ↔ 백엔드 API** 기준으로 정리한 것이다.  
백엔드 구현 기준 경로는 `Petory/backend/main/java/com/linkup/Petory/domain/location/`, 프론트는 `Petory/frontend/src/` 기준이다.

---

## 1. 범위 정리

| 구분 | 설명 |
|------|------|
| **Location (시설)** | DB에 저장된 장소 POI 검색, 카테고리·키워드·반경·지역 계층, AI 추천(`GET /api/location-services/recommend`; **향후 폐기·Pet Data API 단일화 예정** — [Recommendation §1.4](../domains/recommendation.md)) |
| **Geocoding** | 주소↔좌표, 길찾기 (NCP Maps API를 서버 `NaverMapService`에서 호출) |
| **Location Review** | 시설 리뷰 CRUD, 시설 `rating` 집계 갱신 |
| **통합 탐색 (프론트)** | 한 지도에서 **주변서비스 / 모임 / 펫케어** 탭별로 서로 다른 API만 호출 (`UnifiedPetMapPage`, `unifiedMapApi.js`) |

> 통합 탐색은 **백엔드에 단일 “통합 지도” BFF가 있는 구조가 아니라**, 프론트가 도메인별 REST를 조합하는 구조다.

## 4. API 대조표 (프론트 → 백엔드)

### 4.1 시설 (Location)

| 프론트 | HTTP | 백엔드 엔드포인트 | 비고 |
|--------|------|-------------------|------|
| `locationServiceApi.searchPlaces` | GET | `/api/location-services/search` | 쿼리: `latitude`, `longitude`, `radius`(m), `keyword`, 지역 필드, `category`, `size`. **JWT 등 인증 필요**(`SecurityConfig` `/api/**`) |
| `locationServiceApi.recommendPlaces` | GET | `/api/location-services/recommend` | **인증 필요**(`SecurityConfig`의 `/api/**` — 컨트롤러에 메서드별 `@PreAuthorize`는 없음). Spring AI `ChatModel`(Ollama 등)으로 재순위·추천 이유 |
| (관리자 UI) `listLocationServices`, `importPublicData` 등 | GET/POST | `/api/admin/location-services` | `AdminLocationController` — 역할 MASTER/ADMIN 등 컨트롤러 주석·시큐리티와 일치 |

**반경 단위 주의 (실제 버그 방지 포인트)**  
- 백엔드 `LocationServiceController` / `LocationServiceService`는 **`radius`를 미터(m)** 로 받는다.  
- 프론트 `unifiedMapApi.fetchActiveMapItems`는 UI에서 km로 받아 **`radius * 1000`으로 m 변환** 후 `searchPlaces`에 넘긴다.

**검색 분기 (서비스 단일 진입점) — B 방향(위치 우선)**  
`LocationServiceService.searchLocationServices()` 내부: **위치(lat·lng) → 지역 계층(sido/sigungu/eupmyeondong/roadName) → 키워드 단독(fallback) → 전체 평점순** 순.  
`keyword`는 위치·지역 경로에서 SQL `name LIKE '%keyword%'` 필터로 함께 적용된다 (위치가 없을 때만 FULLTEXT fallback 경로 진입).  
`category`·`keyword` 파라미터는 서비스 진입 시 `normalize()`로 빈 문자열 `""` → `null` 정규화 처리된다 (SQL `IS NULL` 조건 정상 작동 보장).  
컨트롤러는 `size`를 정규화한다: 미지정 → 기본 100, `<= 0` → 상한 없음(`null`).

### 4.4 통합 탐색 지도 (프론트 조합)

| 탭 (프론트 라벨) | `unifiedMapApi` 분기 | 호출 API |
|------------------|---------------------|----------|
| 주변서비스 | `type === 'location'` | `locationServiceApi.searchPlaces` |
| 모임 | `type === 'meetup'` | `meetupApi.getNearbyMeetups` → `GET /api/meetups/nearby` |
| 펫케어 | `type === 'care'` | `careRequestApi.getNearby` → `GET /api/care-requests/nearby` |

백엔드 Meetup/Care는 **Location 패키지가 아닌** 각각 `meetup`, `care` 도메인에 있다. 지도 UX만 공유한다.

## 5. 통합 탐색 UI 동작 (프론트)

- **상단 세그먼트**: 주변서비스 / 모임 / 펫케어 중 하나만 활성.
- **옵션 접기**: 반경·내 위치·탭별 컨트롤(검색/생성 등)을 접어 지도 영역 확보 (`UnifiedPetMapPage` + `DomainTabHeader`).
- **데이터**: 활성 탭에 대해서만 `fetchActiveMapItems` 호출 (불필요한 API 동시 호출 방지).
- **정규화**: `toMapItem()`이 `MapContainer`가 기대하는 필드(`idx`, `name`, `latitude`, `longitude`, `markerColor` 등)와 통합 메타(`type`, `title`, `subtitle`)를 함께 둔다.

## 6. Location 검색·저장 (백엔드 핵심)

### 6.1 반경 검색

- 저장소: `SpringDataJpaLocationServiceRepository.findByRadius` — `ST_Within(POLYGON)` + `ST_Distance_Sphere` 이중 조건 (공간 인덱스 활용 + 원형 정밀 필터).
- `keyword`·`category` 필터가 동일 쿼리 WHERE에 포함된다. Java 후처리 없음.
- 애플리케이션: `LocationServiceService`에서 후보에 대해 Haversine으로 **표시용 거리(m)** 를 붙인다. 반경 필터 자체는 DB 쿼리가 담당.

### 6.2 지역 계층 검색

- `findByRoadName` / `findByEupmyeondong` / `findBySigungu` / `findBySido` / `findByOrderByRatingDesc` 5개 쿼리 모두 `keyword`·`category` 필터를 SQL WHERE에 포함한다.
- `findByEupmyeondong`, `findBySigungu`는 복합 인덱스 힌트 `USE INDEX` 적용.

### 6.3 키워드 전용 검색 (fallback)

- 위치·지역 파라미터가 모두 없을 때만 진입하는 FULLTEXT 경로.
- `MATCH(name, description, category1, category2, category3) AGAINST(... IN BOOLEAN MODE)` (Native 쿼리).
- 반경·지역 경로의 `name LIKE '%keyword%'`(이름만)와 달리 설명·카테고리 필드까지 검색 — 동일 키워드라도 경로에 따라 결과가 다를 수 있다.

### 6.4 AI 추천

- `/recommend`: `searchLocationServices`로 후보(컨트롤러에서 `effectiveSize = 30`)를 만든 뒤 `LocationRecommendAgentService`가 Spring AI `ChatModel` 호출로 상위 10개 재정렬·한 줄 추천 이유를 붙인다(구현: `LocationRecommendAgentService`, Ollama 옵션·타임아웃은 `app.ai.recommend.timeout-seconds` 등).
- LLM 타임아웃·파싱 실패 시 원본 상위 10개를 그대로 반환(fallback).

## 7. 리뷰 및 평점 (현행 구현)

리뷰 생성·수정·삭제 후 `LocationServiceReviewService.updateServiceRating(serviceIdx)` 호출.

**현행**: `LocationServiceRepository.updateRatingByAvg(serviceIdx)` — Native `@Modifying` UPDATE 단일 쿼리로 `locationservice.rating`을 리뷰 평균과 동기화.

```sql
UPDATE locationservice SET rating = (
  SELECT avg_rating FROM (
    SELECT COALESCE(AVG(r.rating), 0.0) AS avg_rating
    FROM locationservicereview r
    WHERE r.service_idx = :serviceIdx AND (r.is_deleted IS NULL OR r.is_deleted = 0)
  ) t
) WHERE idx = :serviceIdx
```

인라인 뷰(`t`)로 감싸 MySQL의 “동일 테이블 서브쿼리 제한” 회피. read-modify-write 대신 DB 단일 UPDATE로 **Lost Update 방지**.

## 8. 외부 연동 (Naver Cloud Platform Maps)

- `NaverMapService`가 Geocoding / Reverse Geocoding / Directions URL을 호출한다.
- 클라이언트 ID/시크릿은 **서버 설정**에 두는 것이 정상이며, 프론트에 키를 심지 않는다.

---

## 9. 관리자·배치 (정정)

이전 문서에 “관리자 Location 컨트롤러 없음”으로 적힌 부분은 **현재 코드와 불일치**한다.

- **`AdminLocationController`**: `@RequestMapping("/api/admin/location-services")` 로 존재한다. 공공데이터 임포트, 목록 조회(`q` 키워드 필터 포함)를 노출한다.
- **`PublicDataLocationService`**: CSV 배치 임포트. 배치 저장은 **`LocationServiceBatchWriter`** 별도 빈에 위임한다 (`@Transactional(REQUIRES_NEW)` AOP 정상 적용 보장).
- 배치 사이즈: `app.location.import.batch-size` 설정값 (기본 1000).

---

## 10. 도메인 간 관계 (Meetup / Care)

- **Meetup**: 위치·반경 API는 Meetup 도메인의 **`findNearbyMeetupIds` + `findByIdxInWithOrganizer`** 흐름을 사용한다. Location 서비스와 직접 엔티티 FK로 묶여 있지 않다.
- **Care Request**: 지도 nearby는 `CareRequest`에 **위도/경도/주소** 필드가 있어야 하며, 마이그레이션 스크립트는 `backend/main/resources/sql/migration/carerequest-add-location-columns.sql` 등을 참고한다.

---

## 11. 운영·보안 체크리스트

- `GET /search`의 `size`: 기본 100, 0 이하 무제한 — **대량 응답·OOM** 가능성, 프론트·모바일에서 상한 정책 검토.
- Geocoding/외부 API: 실패 시 사용자 메시지·HTTP 상태 일관성 (`GeocodingController` 동작 기준).
- 리뷰: 이메일 인증, 중복은 애플리케이션 로직 — DB unique는 별도 검토 여지.

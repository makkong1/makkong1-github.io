# 제재 상태 도메인 영향 작업 목록 (2026-06-28)

## 1. 목적

이 문서는 코드 수정 전에, 기존 도메인 문서만 기준으로 `SUSPENDED` / `BANNED` 사용자가 각 도메인에 미치는 영향을 정리한다.

현재 1차 작업 대상은 다음 두 축이다.

1. 문제상황 문서화: [제재 상태 인증 우회 문제 정리](../../troubleshooting/users/sanction-auth-gap-2026-06-28.md)
2. 도메인별 추가 로직 설계: 이 문서

## 2. 판단 기준

제재 상태는 로그인 차단만으로 끝나지 않는다. 사용자가 이미 도메인 객체의 작성자, 참가자, 제공자, 채팅 참여자인 경우가 있기 때문이다.

기본 판단 기준:

| 상태        | 기본 정책                                                        |
| ----------- | ---------------------------------------------------------------- |
| `ACTIVE`    | 기존 기능 사용 가능                                              |
| `SUSPENDED` | 기간제 접근 차단. 만료 전에는 보호 API/WebSocket 접근 차단       |
| `BANNED`    | 영구 접근 차단. 보호 API/WebSocket 접근 및 일반 신고 생성도 차단 |

세부 판단 기준:

- `BANNED`는 항상 제재 상태로 본다.
- `SUSPENDED`는 `suspendedUntil`이 현재 시각보다 미래일 때만 제재 상태로 본다.
- 만료된 `SUSPENDED`는 `ACTIVE` 복구 대상이며, 도메인 차단 대상에서 제외한다.
- 도메인 로직은 상태 문자열 직접 비교보다 `Users.isSanctioned()` 기준으로 판단한다.
- 1차 구현에서는 `SUSPENDED` 사용자도 일반 보호 API와 WebSocket 접근을 차단한다. `SUSPENDED` 전용 조회 허용은 하지 않는다.
- 단, `POST /api/reports` 신고 생성은 `SUSPENDED` 사용자에게 예외적으로 허용한다. 이 예외는 유효한 access token이 남아 있는 경우에 한하며, `BANNED` 사용자는 신고 생성도 차단한다.
- access token 만료 후의 제재 이의제기/신고 경로는 별도 최소 엔드포인트로 후속 설계한다.

| -------- | ------------------------------- | --------------------------------------------------------------------------------- |
| 0        | User/Auth/Report                | 제재 상태가 인증과 신고 처리에서 실제로 적용되어야 다른 도메인 정책이 의미 있음   |
| 1        | Care                            | 요청자, 제공자, 채팅, 결제/에스크로가 얽혀 있어 제재 영향이 가장 큼               |
| 2        | Meetup                          | 모임 생성자/참가자/그룹채팅이 있고, 차단 시 모임 취소 정책이 필요                 |
| 3        | Chat                            | Care와 Meetup의 사용자 간 접점. 제재 사용자 안내/전송 차단/방 상태 처리가 필요    |
| 4        | Payment                         | Care 진행 중 제재 시 지급/환불/보류 정책이 필요. 자동 처리보다 관리자 검토가 안전 |
| 5        | Board/MissingPet/Comment/Review | 콘텐츠 노출과 작성자 비활성화 정책이 필요하지만, 1차 범위는 Care/Meetup 이후      |
| 6        | Notification/Statistics         | 제재 후 알림 발송 대상과 통계 집계 기준 조정 여부 확인                            |

## 4. 1차 필수 전제 작업

Care/Meetup 도메인 로직을 넣기 전에 아래가 먼저 정리되어야 한다.

| 작업                                        | 현재 판단                                                             |
| ------------------------------------------- | --------------------------------------------------------------------- |
| refresh token 제재 검사                     | 1차 인증 보정에서 처리됨. 도메인 작업 전 재회귀 테스트 필요           |
| JWT 인증 제재 검사                          | 1차 인증 보정에서 처리됨. `SUSPENDED` 신고 생성 예외는 별도 구현 필요 |
| WebSocket 기존 세션 제재 반영               | 남은 작업. 메시지 전송 경로에서 최신 상태 확인 필요                   |
| 정지/차단 시 refresh token 제거             | 1차 인증 보정에서 처리됨. 관리자 상태 변경 경로는 이력 통일 작업 남음 |
| 신고 target별 제재 대상 user resolve        | 1차 신고 보정에서 처리됨. 재회귀 테스트 필요                          |
| 제재 도메인 이벤트 또는 공통 처리 지점 정의 | 남은 작업. Care/Meetup/Chat 후속 처리를 이벤트 기반으로 호출          |

후속 처리 방식:

- 사용자가 실제 제재 상태(`BANNED` 또는 만료 전 `SUSPENDED`)가 되면 User 도메인은 제재 적용 이벤트를 발행한다.
- Care/Meetup/Chat은 이 이벤트를 구독해 취소, 참가 취소, 채팅 참여 비활성화, 자동 처리 보류 같은 후속 처리를 수행한다.
- 후속 처리는 제재 상태 저장 커밋 이후 별도 트랜잭션에서 수행하며, 실패해도 제재 자체는 롤백하지 않는다.
- 1차 구현에서 이벤트는 Spring in-memory 이벤트 기반이다. 이벤트 유실이나 처리 실패 가능성은 허용하고, 로그 기록과 관리자 수동 재처리로 대응한다.
- 일반 사용자 노출 차단은 이벤트 처리에만 의존하지 않고 조회 시점에도 사용자 제재 상태를 확인한다.
- `SUSPENDED` 해제 이벤트는 만들지 않는다. 복구 가능한 화면 상태는 조회 시점의 사용자 상태를 기준으로 처리한다.

### 5.1 요청자가 제재된 경우

사용자 말 기준 정책: “사용자는 글 자체가 안 보이게”

작업:

| 상황                                    | 필요한 동작                                                                  |
| --------------------------------------- | ---------------------------------------------------------------------------- |
| 요청자가 `SUSPENDED`                    | 신규 케어 요청 생성 차단                                                     |
| 요청자가 `BANNED`                       | 신규 케어 요청 생성 차단                                                     |
| `SUSPENDED` 요청자의 기존 `OPEN` 요청   | 상태 변경 없이 목록/검색/nearby/상세에서 비노출. 해제 후 재노출 가능         |
| `BANNED` 요청자의 기존 `OPEN` 요청      | 제재 이벤트 후 `CANCELLED` 전환. DB row와 이력은 유지하고 일반 사용자 비노출 |
| 요청자가 제재된 기존 `IN_PROGRESS` 요청 | 자동 완료/지급/환불 금지. 관리자 검토 대상으로 보류                          |
| 요청자가 제재된 기존 채팅방             | 상대 제공자에게 “제재된 사용자” 안내, 신규 거래 확정 차단                    |

확인할 API/흐름:

- 케어 목록, 검색, nearby, 상세
- 케어 요청 생성/수정/삭제
- 케어 상태 변경
- 케어 자동 완료 스케줄러
- 채팅 거래 확정
- 에스크로 생성/환불/지급

주의:

- `SUSPENDED` 요청자의 `OPEN` 케어는 공개 탐색 화면에서만 숨긴다. 제공자의 “내 지원/내 대화” 같은 당사자 이력은 DB 이력을 유지하고, 필요하면 “제재된 사용자” 안내 플래그로 표시한다.
- `BANNED` 요청자의 `OPEN` 케어만 `CANCELLED` 전환 대상으로 제한한다. `IN_PROGRESS` 케어는 자동 환불/지급하지 않는다.

### 5.2 제공자가 제재된 경우

사용자 말 기준 정책: “제공자는 채팅에서 ‘신고당한 유저입니다’처럼 안내하고 안 보이게”

작업:

| 상황                                     | 필요한 동작                                         |
| ---------------------------------------- | --------------------------------------------------- |
| 제공자가 `SUSPENDED`                     | 신규 케어 댓글/지원/거래 확정/채팅 전송 차단        |
| 제공자가 `BANNED`                        | 신규 케어 댓글/지원/거래 확정/채팅 전송 차단        |
| 제공자가 제재된 기존 댓글/지원자 목록   | 요청자 화면에 “제재된 사용자” 안내 표시             |
| 제공자가 제재된 채팅방                   | 요청자 화면에 “제재된 사용자” 안내 표시             |
| 제공자가 제재된 기존 `OPEN` 매칭 전 대화 | 거래 확정 차단. 채팅방은 닫지 않고 기존 이력 유지   |
| 제공자가 제재된 기존 `IN_PROGRESS` 케어  | 자동 완료/지급/환불 금지. 관리자 검토 대상으로 보류 |

주의:

- 현재 Care 매칭은 Chat 도메인의 `confirmCareDeal()`과 강하게 연결되어 있다.
- 따라서 Care만 막아서는 부족하고 Chat 메시지 전송, 거래 확정, 참가자 노출 정책을 같이 수정해야 한다.
- 1차 구현에서는 `SUSPENDED`와 `BANNED`를 구분하지 않고 모두 “제재된 사용자”로 안내한다. 메시지/댓글 DB 이력은 삭제하지 않는다.
- 케어 요청 상세의 기존 댓글/지원자 목록에서도 채팅방과 같은 안내 정책을 적용한다. 1차에서는 댓글/지원 row를 숨기거나 삭제하지 않고 표시명/상태 플래그로 “제재된 사용자”임을 알린다.

### 5.3 Care 테스트 후보

| 테스트                                                                     | 기대                              |
| -------------------------------------------------------------------------- | --------------------------------- |
| `SUSPENDED` 요청자의 `OPEN` 케어 요청이 목록/검색/nearby/상세에서 제외된다 | 상태 유지 + 비노출                |
| `BANNED` 요청자의 `OPEN` 케어 요청이 제재 이벤트 후 `CANCELLED`가 된다     | 이력 유지 + 일반 사용자 비노출    |
| 제재 요청자의 케어 상세 접근 정책이 적용된다                               | 일반 사용자 비노출 또는 안내 응답 |
| 제재 요청자는 케어 요청을 새로 만들 수 없다                                | 403                               |
| 제재 제공자는 케어 댓글/지원/거래 확정을 할 수 없다                        | 403                               |
| 제재 제공자가 포함된 채팅방은 상대방에게 안내 상태를 내려준다              | 안내 표시 가능 DTO                |
| 진행 중 케어의 요청자/제공자가 제재되면 자동 완료/지급이 진행되지 않는다   | 관리자 검토/보류                  |

## 6. Meetup 도메인 작업 목록

문서 근거:

- `docs/domains/meetup.md`
- `docs/domains/chat.md`
- `docs/architecture/meetup/산책 & 오프라인 모임 아키텍처.md`

사용자 말 기준 정책: “모임은 당연히 취소”

Care와 Meetup의 `SUSPENDED` 정책은 의도적으로 비대칭이다. Care `OPEN` 요청은 게시글 성격이 강해 정지 해제 후 재노출이 가능하지만, Meetup은 날짜·장소·참가자가 있는 오프라인 약속이므로 주최자가 정지되어도 기존 모집 중 모임은 취소한다.

### 6.1 주최자가 제재된 경우

작업:

| 상황                               | 필요한 동작                                                       |
| ---------------------------------- | ----------------------------------------------------------------- |
| 주최자가 `SUSPENDED`               | 신규 모임 생성 차단                                               |
| 주최자가 `BANNED`                  | 신규 모임 생성 차단                                               |
| 주최자가 제재된 기존 모집 중 모임  | `SUSPENDED`/`BANNED` 모두 취소 처리. 목록/검색/nearby/home 비노출 |
| 주최자가 제재된 기존 `CLOSED` 모임 | 변경 없음. 종료 이력은 유지                                       |
| 주최자가 제재된 모임 채팅방        | 신규 메시지 차단. 기존 채팅 이력은 유지                           |

정지 해제 후 취소된 모임은 자동 복구하지 않는다. 필요하면 주최자가 새 모임을 생성해야 한다.

### 6.2 참가자가 제재된 경우

작업:

| 상황                         | 필요한 동작                             |
| ---------------------------- | --------------------------------------- |
| 제재 사용자의 모임 참가      | 차단                                    |
| 제재 사용자의 기존 참가 row  | 참가 취소 처리. 참가자 목록에서 제외    |
| 제재 사용자의 모임 채팅 참여 | 참여 상태를 `LEFT` 또는 비활성으로 전환 |
| 제재 사용자의 히스토리       | 기존 모임 히스토리는 DB에서 유지        |

`SUSPENDED` 해제 후 기존 참가를 자동 복구하지 않는다. 필요하면 사용자가 다시 참가해야 한다.

### 6.3 Meetup 테스트 후보

| 테스트                                                      | 기대                           |
| ----------------------------------------------------------- | ------------------------------ |
| 제재 사용자는 모임을 생성할 수 없다                         | 403                            |
| 제재 사용자는 모임에 참가할 수 없다                         | 403                            |
| 주최자가 제재되면 모집 중 모임이 취소/비노출된다            | 목록/검색/nearby/home 제외     |
| 주최자가 제재되면 모임 채팅방 신규 메시지가 차단된다        | 메시지 전송 차단               |
| 참가자가 제재되면 참가 취소와 채팅 참여 비활성화가 적용된다 | 참가자 목록 제외 + 채팅 비활성 |

## 7. Chat 도메인 공통 작업

Care와 Meetup 모두 Chat을 거치므로 공통 정책이 필요하다.

작업:

| 상황                                  | 필요한 동작                                              |
| ------------------------------------- | -------------------------------------------------------- |
| 제재 사용자의 REST 메시지 전송        | 차단                                                     |
| 제재 사용자의 STOMP 메시지 전송       | 차단. 1차 구현은 전송 경로에서 최신 상태 확인            |
| 제재 사용자가 포함된 채팅방 목록/상세 | 상대방에게 “제재된 사용자” 안내 플래그를 내려줄 DTO 필요 |
| 진행 중 케어 채팅 상대가 제재됨       | 거래 확정 버튼 비활성화 및 안내                          |
| 모임 채팅 참여자가 제재됨             | 메시지 전송 차단, 참가자 표시 정책 적용                  |

주의:

- 제재 전에 열린 WebSocket 세션도 메시지 전송 시점에 차단되어야 한다. 1차 구현은 `SEND` 처리 경로, 즉 채널 인터셉터 또는 `ChatMessageService.sendMessage()`에서 최신 사용자 상태를 확인하는 방식으로 처리하고, Redis/TTL 캐시 전략은 2차에서 검토한다.
- “제재된 사용자” 안내는 실시간 push가 아니라 채팅방 목록/상세 조회 응답에 안내 플래그를 포함하는 방식으로 처리한다.
- 기존 채팅 메시지는 DB에서 삭제하지 않고, 1차 구현에서는 메시지 내용도 마스킹하지 않는다.
- 채팅방은 1차에서 닫지 않고, 비제재 상대방은 기존 이력을 읽을 수 있게 둔다.
- 거래 확정, 모임 참가/재참여 같은 도메인 상태 변경 액션은 차단한다.

### 8.1 구현 재점검 후 추가 보정 (2026-06-30)

1차 구현 후 코드 기준으로 재점검한 결과, 단순 차단은 들어갔지만 일부 도메인 후속 상태가 정책과 어긋날 수 있어 아래 항목을 추가 보정했다.

| 영역 | 발견된 문제 | 보정 내용 |
|------|-------------|-----------|
| Auth | 제재된 access token 요청이 인증 실패처럼 401로 보일 수 있음 | `JwtAuthenticationFilter`에서 `POST /api/reports` 예외 외 제재 계정 요청은 즉시 403 JSON 응답 |
| Auth | 만료된 `SUSPENDED`가 상태 문자열 때문에 계속 인증 차단될 수 있음 | `suspendedUntil`이 지난 `SUSPENDED`는 조회 시점 기준으로 인증 허용 |
| User/Admin | 관리자 상태 변경 경로가 `UserSanctionAppliedEvent`를 발행하지 않음 | `UsersService.updateUserStatus()`에서 `SUSPENDED`/`BANNED` 적용 시 이벤트 발행 |
| Care 상세 | 목록 필터와 달리 단건 상세에서 제재 요청자의 사전 매칭 요청이 직접 노출될 수 있음 | 일반 사용자 단건 조회에서 제재 요청자의 `OPEN`/`CANCELLED` 요청 비노출. 관리자는 확인 가능 |
| Care 거래 확정 | 채팅 거래 확정 시 현재 클릭한 사용자만 제재 확인함 | 채팅방 활성 참여자 전체와 실제 requester/provider를 재확인한 뒤 `IN_PROGRESS` 전환/에스크로 생성 |
| Care 자동 완료 | 스케줄러가 요청자 제재만 확인하고 제공자 제재는 놓침. `SUSPENDED OPEN` 요청도 자동 완료될 수 있음 | 자동 완료 대상에서 요청자 또는 ACCEPTED 제공자 중 제재자가 있는 케어는 제외 |
| Care 정산/취소 | 수동 상태 변경으로 제재 당사자가 포함된 케어의 지급/환불이 진행될 수 있음 | 비관리자 상태 변경에서 `COMPLETED`/`CANCELLED` 전환 전 당사자 제재 여부 확인 |
| Meetup 노출 | `CANCELLED` 모임이 일반 목록/검색/nearby/상세에 계속 노출될 수 있음 | 일반 사용자 조회 쿼리에서 `CANCELLED` 제외. 관리자 조회는 유지 |
| Meetup 배치 | 날짜가 지나면 `CANCELLED` 모임이 `COMPLETED`로 덮일 수 있음 | `completePastMeetups()`에서 `CANCELLED` 제외 |
| Meetup 참가 취소 | 제재 참가자 row 삭제 시 `currentParticipants`가 감소하지 않음 | 진행 예정 모임의 일반 참가자 row 삭제 시 인원 수 원자적 감소 |
| Meetup 이력 | 제재 참가자의 과거 참가 이력까지 삭제될 수 있음 | 제재 후속 처리는 취소되지 않은 진행 예정 모임의 일반 참가 row로 제한. 과거 히스토리 유지 |
| Meetup 채팅 | 채팅방 퇴장 실패가 참가 취소까지 막을 수 있음 | 참가 row 삭제와 인원 감소를 먼저 확정하고, 채팅방 퇴장 실패는 로그만 남김 |

후속 및 2차 과제 점검 중 1차 범위 안에서 바로 보정해야 할 보안/정책 불일치를 확인해 아래 항목을 반영했다.

## 9. 1차 확정 정책 요약

| 정책                                  | 확정 내용                                                                    |
| ------------------------------------- | ---------------------------------------------------------------------------- |
| 제재 판정 기준                        | `BANNED` 항상 제재, `SUSPENDED`는 `suspendedUntil`이 미래일 때만 제재        |
| 보호 API/WebSocket 접근               | `SUSPENDED`/`BANNED` 모두 차단. 단 `POST /api/reports`는 `SUSPENDED` 예외    |

| `BANNED` 요청자의 기존 `OPEN` 케어    | 제재 이벤트 후 `CANCELLED` 전환. DB row와 이력 유지                          |
| 진행 중 케어의 요청자/제공자 제재     | 자동 완료/지급/환불 금지. 관리자 검토 대상으로 보류                          |
| 제재 제공자 표시                      | 1차에서는 `SUSPENDED`/`BANNED` 모두 “제재된 사용자”로 통일                   |
| 제재 사용자의 과거 채팅 메시지        | DB 유지, 1차에서는 메시지 내용 마스킹 없음                                   |
| 모임 주최자 제재                      | `SUSPENDED`/`BANNED` 모두 기존 모집 중 모임 취소. 자동 복구 없음             |
| 모임 참가자 제재                      | 참가 취소, 참가자 목록 제외, 채팅 참여 비활성화. 자동 재참가 없음            |
| 후속 처리 이벤트                      | Spring in-memory 이벤트. 유실/실패 가능성은 로그 + 관리자 수동 재처리로 대응 |

# 그루밍 추천 MVP — 설계·리스크·Petory 연동 (**최종본**)

**상태:** 계약·플래그·운영 파라미터까지 반영한 **구현·Petory DTO SSOT**. 크롤링 없이 **공공 `pet_facilities` + 네이버 블로그(언급 신호) + Kakao Local(장소 보강)** 조합을 목표로 할 때, **현실적으로 까다로운 지점**과 **Petory 쪽에 필요한 변경**을 한곳에 정리한다.

관련 흐름: [`DATA-AND-API-FLOW.md`](DATA-AND-API-FLOW.md), [`INGESTION-VS-SERVING.md`](INGESTION-VS-SERVING.md).

## 0. 경로 정합성 (이 레포 기준)

과거 설계 문서·플랜에는 `app/api`, `app/collector`, `app/recommender`가 남아 있을 수 있다. **이 레포의 실제 코드 기준 경로**는 아래이며, 구현·리뷰 시 이 표를 우선한다.

| 역할                               | 실제 경로 (canonical)      | 구(旧) 경로 (문서에서만 보일 수 있음) |
| ---------------------------------- | -------------------------- | ------------------------------------- |
| HTTP 라우터                        | `app/serving/api/*.py`     | `app/api/*.py`                        |
| 추천·반경 시설                     | `app/serving/recommender/` | `app/recommender/`                    |
| 공공 수집·네이버·지오코더          | `app/ingestion/`           | `app/collector/`                      |
| 공용 코어·모델·스키마·Redis·스케줄 | `app/platform/`            | `app/core`, `app/models` 등 분산      |

**§6 관련 코드**의 링크는 위 canonical 경로만 사용한다.

## 1. MVP 범위 (고정)

| 항목      | MVP                                                                                                       |
| --------- | --------------------------------------------------------------------------------------------------------- |
| 카테고리  | **`grooming`(미용) 만** (다른 카테고리는 미용이 완료되고 테스트 통과후 잘 작동하면 그후에 업데이트)       |
| 1차 후보  | 반경 내 공공 DB `pet_facilities`, 타입 `BUSINESS` (현행 매핑)                                             |
| 블로그    | 네이버 블로그 검색 API — **언급·후보명 추출용** (키워드 빈도만 쓰는 현행 파이프와 **별 트랙**으로 확장)   |
| 장소 보강 | Kakao Local API — **키워드 검색으로 상호 후보 → 주소·좌표** (기존 지오코딩 API와 엔드포인트 다를 수 있음) |
| LLM       | **후보·점수는 규칙 기반**; LLM은 선택적으로 **상위 N개 “추천 이유” 요약**만                               |

**비목표 (MVP):** `hospital` / `supplies`에 **지금 이 MR/스프린트에서** 동일 로직 일괄 적용, **외부 API(블로그·Kakao) 원문·전체 응답을 영구 DB에 적재**하는 것.

### 1.1 Feature flag (신규 파이프 on/off)

| 변수                   | 기본    | 동작                                                                                                                                                                             |
| ---------------------- | ------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `GROOMING_MVP_ENABLED` | `false` | `true`일 때만 `context=grooming`에 §2 이후 **블로그·Kakao·랭킹 확장 파이프** 적용. `false`이면 그루밍도 **현행 `/recommend`와 동일**(§2.6 “비그루밍 유지”와 대칭으로 회귀 방지). |

구현은 `app.platform.core.config.Settings`(또는 동등)에 올리고 `.env.example`에 한 줄 주석으로 명시한다.

### 1.2 이후 확장 (의도)

- **그루밍만 먼저**는 검증·테스트 겸 **첫 카테고리**일 뿐이다.
- 안정화·테스트 통과 후에는 **같은 패턴**(블로그 멘션 → Kakao 보강 → 공공와 랭킹)을 **카테고리별 쿼리·룰·플래그**로 나눠 `hospital` 등에 순차 적용하면 된다.
- 코드상으로는 현재 `GROOMING_MVP_ENABLED` + `context=grooming`만 연결돼 있고, 타 카테고리는 **아직 레거시**인 것이 정상이다.

## 2. 구현 시 까다로운 부분 (미리 알아두기)

### 2.1 가게명(상호) 후보 추출 — 블로그 제목·스니펫

| 리스크   | 설명                                                                  |
| -------- | --------------------------------------------------------------------- |
| **오탐** | 형태소만으로는 일반명사(미용, 애견)가 잡히고 실제 상호는 빠지기 쉬움. |
| **누락** | 브랜드·고유명사가 분석기 사전에 약하면 후보에 안 들어옴.              |

**완화 (권장):**

- **룰 + 사전**: “OO 미용실”, “애견 OO” 패턴, 최소 길이, 불특정 단어 블록리스트.
- **후보 → Kakao 검색으로만 점수화**: Kakao에 안 잡히는 이름은 **멘션 가중 제외** 또는 낮은 가중.
- MVP는 **정확도보다 “거짓 업장 최소화”** 우선: 애매하면 버림.

### 2.2 Kakao 호출량 폭증 (쿼터·비용)

블로그에서 상호 후보를 많이 뽑고 **후보마다 개별 Kakao 검색**을 하면 호출이 선형으로 늘어 **일일 쿼터·레이트리밋**에 걸리기 쉽다.

**필수 통제 (MVP):**

| 조치            | 권장 값                                                                                                     |
| --------------- | ----------------------------------------------------------------------------------------------------------- |
| **후보 상한**   | 블로그 기반 **고유 상호 후보 최대 N** (예: **20**). 초과 시 멘션·후보 품질 점수 상위 N만 Kakao 단계로 진입. |
| **TTL 캐시**    | **§2.2.1 스키마·TTL 고정**. 캐시 값은 **좌표·주소 등 최소 필드만** (원문 JSON 전체 저장 금지).              |
| **동시성 제한** | Kakao HTTP 클라이언트에 **세마포어/동시 요청 상한** (예: 3~5)으로 스파이크 방지.                            |

위 3가지는 설계가 아니라 **구현 체크리스트**로 취급한다.

### 2.2.1 캐시 키 스키마 (Kakao 장소 검색)

Redis(또는 동일 네이밍 규칙의 인메모리)에 **문자열 키**로 통일:

| 항목                        | 값                                                                                                     |
| --------------------------- | ------------------------------------------------------------------------------------------------------ |
| **키**                      | `kakao:place:{normalized_name}:{lat_grid}:{lng_grid}`                                                  |
| **`normalized_name`**       | 공백·특수문자 제거 등 **단일 규칙**으로 정규화한 상호 (구현 SSOT).                                     |
| **`lat_grid` / `lng_grid`** | 사용자 기준 좌표를 **고정 스텝**(예: `round(lat * 100) / 100`)으로 양자화해 캐시 적중률과 정밀도 균형. |
| **TTL**                     | **600초 (10분)**                                                                                       |

`lng_grid`가 음수/소수 표현이 환경별로 달라지지 않도록 **항상 동일 포맷**(예: 소수 넷째 자리 고정 문자열)으로 키에 넣을 것.

### 2.3 가게명 ↔ Kakao POI 매칭 + 공공 DB dedupe (숫자로 고정)

문장형 규칙만 두면 구현자마다 해석이 달라져 **중복 노출·오탐**이 생긴다. MVP에서는 아래 **순서와 기준을 고정**한다.

**입력:** 사용자 `(lat_u, lng_u)`, 반경 `R_m`, Kakao 키워드 검색 결과 POI 후보 목록, 공공 `pet_facilities` 반경 목록, 블로그 멘션으로 얻은 상호 문자열.

1. **반경 필터**

   - POI·공공 시설 모두 사용자 기준 **Haversine 거리 ≤ `R_m`** 인 것만 랭킹 후보에 포함.
   - 반경 밖 POI는 **즉시 제외** (검색 API가 “가까운 순”을 주더라도 거리로 재검증).

2. **공공 DB와의 동일 업장 병합**

   - 후보 POI/멘션 이름과 `pet_facilities.name`에 대해 **정규화 후 유사도**를 계산한다 (예: 공백·특수문자 제거, 소문자/한글만, 자모 단위는 MVP에서 선택).
   - **유사도 임계값** (예: RapidFuzz `ratio ≥ 85` 또는 한글 전용 자카드 ≥ 0.8 — 구현 시 상수로 코드에 박고 주석에 문서화)을 넘기면 **동일 시설로 간주**.
   - 병합 시 **표시 좌표·주소는 공공 `pet_facilities` 우선**(신뢰 소스 단일화). 카카오는 “멘션·거리 보조 신호”로만 쓴다.

3. **동명/체인 (여러 후보가 남을 때)**

   - 1순위: **거리 오름차순** (`distance_m` 작을수록 위).
   - 2순위: **공공 매칭 여부** (공공과 매칭된 항목이 동일 거리대이면 위).
   - 3순위: **`mention_count` 내림차순** (동률이면 멘션 많은 쪽).
   - 4순위: **안정적 tie-break** — `source_id` 또는 `name` 문자열 사전순 (재현 가능한 정렬).

4. **중복 출력 방지**
   - 동일 **공공 `source_id`** 또는 **병합 규칙으로 동일 시설로 판정된 항목**은 **`facilities[]`에 1행만** 남긴다.

임계값·거리 단위는 env/설정으로 빼되, **기본값과 이 문서의 표를 SSOT**로 맞출 것.

### 2.4 `mention_count` — **채택 (닫힘)**

| 결정         | 내용                                                                                                                                        |
| ------------ | ------------------------------------------------------------------------------------------------------------------------------------------- |
| **정의**     | **A안: 블로그 글 1건당 해당 상호 1회** (글 단위 중복 언급은 1표).                                                                           |
| **정규화**   | 카테고리 내 네이버 검색 **쿼리 수로 나눈 값**을 `mention_score`로 병행 (§3 범위 **0.0~1.0**).                                               |
| **API 계약** | 그루밍 MVP 응답의 각 시설에 **`mention_count` 필드 생략 금지**. 네이버 실패·빈 결과·멘션 없음이면 **`0` 고정** (클라이언트 null 분기 제거). |

### 2.5 `freshness_weight` — **채택 (닫힘)**

| 결정     | 내용                                                                                                                           |
| -------- | ------------------------------------------------------------------------------------------------------------------------------ |
| **규칙** | 네이버 응답에 **게시일(`postdate` 등)이 있으면** freshness 항 계산에 사용. **없으면 `freshness_weight = 0`** (항 생략과 동치). |

### 2.6 비그루밍 `context` — **채택 (닫힘)**

| 결정         | 내용                                                                                                              |
| ------------ | ----------------------------------------------------------------------------------------------------------------- |
| **MVP 기간** | `context !== grooming` 은 **현행 `/recommend` 동작 유지** (확장 필드만 optional로 내려가도 됨). 회귀 최소화 우선. |

### 2.7 정책·캐시·로그 (리스크 관리 문구 강화)

| 원칙                | 구현 방향                                                                                                                                                                                             |
| ------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 원본 장기 저장 금지 | 블로그·카카오 **전문/원본 JSON**은 DB에 장기 적재하지 않음.                                                                                                                                           |
| **영구 DB 비적재**  | **외부 API 결과를 그대로 영구 DB 테이블에 적재하지 않음** (공공 `pet_facilities` 등 **자체 소스**만 영구). Kakao/블로그에서 온 문자열·좌표는 **요청·배치 스코프 메모리 또는 짧은 TTL 캐시**로만 처리. |
| 짧은 캐시           | §2.2.1 TTL·키 스키마. 값 필드 최소화.                                                                                                                                                                 |
| 로그                | **구조화 로그 필드 최소화** (PII·원문 스니펫 장문 금지). 관측용 카운터·ms 위주 (§4).                                                                                                                  |
| 영구 집계           | 내부 **집계 점수·클릭 로그**는 별도 설계(MVP 밖 가능).                                                                                                                                                |

각 서비스 **이용약관**에 맞춰 캐시 TTL·보관 필드 조정.

### 2.8 점수식 (`mention_weight + distance_weight + freshness_weight`)

- **가중치 상수**는 설정 파일 또는 env로 빼서 **튜닝 없이 재배포** 가능하게.
- `pet_facilities`만 있는 후보와 “블로그+Kakao로만 나온 후보”의 **source** 다르게 두어 Petory UI에서 구분 가능하게.
- **`mention_score`·`score` (응답에 포함 시)** 모두 **부동소수 `[0.0, 1.0]`** 로 정규화해 노출한다. 내부 랭킹은 스케일이 달라도 최종 API 값만 이 구간으로 clamp/재스케일.

### 2.9 HTTP 타임아웃 기본값 (ms, 구현 SSOT)

공통 클라이언트(`httpx` 등)에 반영하되, **아래를 문서·코드 기본값으로 맞춘다** (운영은 env로 덮어쓰기 권장).

| 대상                          | 기본 타임아웃 (ms) | 비고                                                                                    |
| ----------------------------- | ------------------ | --------------------------------------------------------------------------------------- |
| **Naver** (블로그 검색 등)    | **10_000**         | 재시도 정책이 있으면 **요청당** 상한을 넘기지 않도록 합산 설계.                         |
| **Kakao** (Local·지오코딩 등) | **8_000**          | §2.2 동시성 제한과 함께 tail latency 완화.                                              |
| **LLM** (Ollama 등)           | **30_000**         | 현행 [`llm.py`](../app/serving/recommender/llm.py) `timeout=30` 초와 동일 **30_000ms**. |

## 3. API 계약 — Petory 변경 포인트 (예상)

`POST /recommend` **요청**은 현행 유지 가능: `lat`, `lng`, `context=grooming`, `radius_km`, `top_n`, `pet`.

**응답 루트 (권장):**

| 필드                | 타입   | 설명                                                                                                                                                                                                                                                                                             |
| ------------------- | ------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| `recommend_version` | string | Petory·로그 **호환 추적**. **`GROOMING_MVP_ENABLED=true`이고 `context=grooming`으로 신규 파이프가 응답을 만들 때** 값 **`grooming-mvp-v1` 고정**. 그 외(플래그 off 또는 현행 파이프만)는 **`legacy` 또는 기존 클라이언트가 쓰는 상수**로 통일 — **항상 문자열 필드로 내려 null 생략 금지** 권장. |

**응답 `facilities[]` 확장 (제안):**

| 필드            | 타입   | 설명                                                                                |
| --------------- | ------ | ----------------------------------------------------------------------------------- |
| `name`          | string | 표시 상호                                                                           |
| `address`       | string |                                                                                     |
| `lat`, `lng`    | float? |                                                                                     |
| `distance_m`    | int    | 사용자 기준                                                                         |
| `mention_count` | int    | §2.4; **항상 포함**, 실패·무멘션 시 **`0`**                                         |
| `mention_score` | float  | **범위 `[0.0, 1.0]`**; 멘션 신호 없음·블로그 실패 시 **`0.0`**, **필드 생략 금지**. |
| `source`        | string | 예: `public` \| `kakao` \| `public+kakao`                                           |
| `score`         | float  | **최종 랭킹 점수, 범위 `[0.0, 1.0]`**; **필드 생략 금지** (동일 정책).              |

**`trends[]`:** 기존 Redis 키워드 유지 가능. 그루밍 MVP에서 **블로그 언급 파이프**와 역할이 겹치지 않게 할지(트렌드만 카피용 vs 멘션은 시설 전용) **한 줄로 정책 고정** 추천.

**`recommendation`:** 규칙 기반 한 줄 + 선택적 LLM “상위 3개 이유” — Petory는 **LLM 없을 때**도 UI 깨지지 않게.

### Petory 체크리스트 (백엔드·프론트)

- [ ] `Facility`/`Recommend` DTO에 `mention_count`(**non-null, 0 허용**), `source`, `mention_score`·`score`(**0~1**), 루트 `recommend_version` 반영.
- [ ] 지도·카드에 **출처 표시** (공공 vs 카카오 보강).
- [ ] `recommendation`이 **짧은 규칙 문구**만 올 때 레이아웃.
- [ ] **`context !== grooming`**: MVP 기간 **기존 동작 유지** (§2.6).
- [ ] 서버 `GROOMING_MVP_ENABLED` off 시 그루밍도 **기존 응답 모양** 유지 (§1.1).
- [ ] **배포 전**: 확장 필드가 포함된 샘플 JSON으로 **`PetDataApiClient` 역직렬화** (Jackson unknown property 정책) 확인 (§3.1).

### 3.1 Petory 연동 — 단계적 적용 (크로스 레포)

**“설계대로 Python만 만들어도 된다”와 “Petory가 아무 수정 없이 당장 완전 호환”은 다르다.** 아래면 **충돌 없이 단계적으로** 맞출 수 있다.

| 전제                                                                     | 의미                                                                                                             |
| ------------------------------------------------------------------------ | ---------------------------------------------------------------------------------------------------------------- |
| **플래그 기본 off** (`GROOMING_MVP_ENABLED=false`)                       | Petory는 **당분간 안 건드려도** 되고, 회귀 위험이 작다 (§1.1).                                                   |
| **추가 필드 중심** (`recommend_version`, `mention_*`, `source`, `score`) | 업스트림이 먼저 내려도 다운스트림이 **아직 쓰지 않으면 대체로 무시** 가능하다.                                   |
| **기존 필드 유지**                                                       | `distance_m`, `generated_at` 등 **이미 쓰는 이름·타입**을 깨지 않으면, DTO 확장 전에도 동작할 **가능성이 높다**. |

**배포 전에 Petory에서 한 번 확인할 것**

- Java/Jackson 등에서 **알 수 없는 JSON 프로퍼티를 실패로 처리**하는 설정이면, 새 필드가 섞인 응답에서 **역직렬화 예외**가 날 수 있다. **`PetDataApiClient`(또는 동등) 응답 매핑·ObjectMapper 설정**을 스테이징에서 확인한다.
- `recommend_version`·시설의 `mention_count` / `mention_score` / `score` / `source`를 **실제로 쓰려면** DTO·프론트는 결국 §3 체크리스트대로 맞추는 것이 맞다.

**pet-data-api 쪽 권장**

- `context !== grooming` 일 때 응답이 **문서·기존 Petory 기대와 동일**한지 **테스트로 고정**해 두면, Petory는 나중에 **그루밍만** 확장하면 된다.

## 4. SLA·폴백·관측성

### 4.1 `/recommend` SLA 목표 (초안 — 배포 전 측정으로 조정)

| 구간                                      | 목표                                       |
| ----------------------------------------- | ------------------------------------------ |
| **전체**                                  | **p95 ≤ 1.5s** (프로덕션에서 RUM/APM 기준) |
| **LLM 제외** (공공+블로그+Kakao+랭킹까지) | **p95 ≤ 700ms**                            |

LLM은 타임아웃·서킷 브레이커를 두고 SLA에서 분리 측정한다.

### 4.2 실패 시 폴백 순서 (고정)

| 실패                                 | 폴백                                                                                                                                           |
| ------------------------------------ | ---------------------------------------------------------------------------------------------------------------------------------------------- |
| **Kakao 실패** (타임아웃, 5xx, 쿼터) | **공공 DB 반경 결과만** 반환 (멘션·카카오 출처 없음 또는 `source=public`만).                                                                   |
| **블로그 실패**                      | **거리·공공 기반만** 반환; 각 시설 **`mention_count = 0`**, `mention_score`·관련 필드는 §3 **0.0~1.0 규칙**에 맞게 **`0.0`** (필드 생략 금지). |
| **LLM 실패**                         | **규칙 기반 문구만** (`recommendation`); 시설 목록은 이미 결정된 대로 유지.                                                                    |

폴백 시에도 HTTP 200을 유지할지, 부분 실패를 헤더/바디 플래그로 줄지는 팀 합의 — MVP에서는 **200 + 완화된 payload**를 권장.

### 4.3 관측성 (로그/메트릭 필드)

| 필드                          | 의미                                                                   |
| ----------------------------- | ---------------------------------------------------------------------- |
| `candidate_count_raw`         | 블로그·룰에서 나온 상호 후보 수 (상한 적용 전)                         |
| `candidate_count_after_cap`   | 상한 적용 후 Kakao 단계 진입 수                                        |
| `candidate_count_after_match` | 공공 병합·반경 필터 후 최종 랭킹 후보 수                               |
| `kakao_call_count`            | 해당 요청에서 실제 Kakao HTTP 호출 수 (캐시 미스만 카운트하면 더 좋음) |
| `latency_ms`                  | 단계별 또는 전체 (예: `total`, `blog`, `kakao`, `rank`)                |

원문·전체 JSON 대신 **카운터·밀리초**로 운영 가능성을 확보한다.

### 4.4 요청 단위 추적 로그 (그루밍 MVP)

한 번의 `POST /recommend`(그루밍·플래그 on)마다 **`[req_id]`**(짧은 hex)로 같은 요청의 단계를 묶을 수 있다. 로거/메시지 접두:

| 접두              | 모듈                                      | 내용                                                            |
| ----------------- | ----------------------------------------- | --------------------------------------------------------------- |
| `grooming_pipe`   | `app.serving.api.recommend`               | 시작·공공 DB 건수·단계별 ms·요약·응답 크기                      |
| `grooming_blog`   | `app.ingestion.grooming_blog`             | 네이버 블로그 스텝 시작/종료(아이템 수·유니크 후보·소요 ms)     |
| `kakao_place`     | `app.ingestion.kakao`                     | 후보 수·**cache_hit / cache_miss / http_ok / http_err**·소요 ms |
| `grooming_ranker` | `app.serving.recommender.grooming_ranker` | 병합 후 후보 수·dedupe 후·출력 `source` 분포                    |
| `grooming_copy`   | `app.serving.recommender.builder`         | 규칙 문구 vs 트렌드 폴백                                        |

원문 스니펫·전체 페이로드는 로그에 넣지 않는다 (§2.7).

## 5. 구현 순서 제안 (pet-data-api)

1. `GROOMING_MVP_ENABLED`·§2.9 타임아웃·§2.2.1 캐시 키를 **설정 레이어에 반영**.
2. Kakao Local **키워드→장소** 클라이언트 + 단위 테스트(mock); **§2.2 캡·캐시·동시성** 포함.
3. 그루밍용 네이버 블로그 수집 → **상호 후보 추출**(룰+형태소) → `mention_count` 집계(§2.4, 요청 스코프만).
4. `pet_facilities` 반경 목록과 **§2.3 랭킹·dedupe**; **`score` / `mention_score` 0~1**.
5. `/recommend`에서 플래그·`context=grooming` 분기; 응답에 **`recommend_version: "grooming-mvp-v1"`** 및 §3 계약.
6. LLM은 마지막에, **상위 고정 후보만** 프롬프트에 넣기.
7. **§4** 메트릭·폴링 연동.
8. **`context !== grooming`** 및 **플래그 off + `context=grooming`** 응답이 기존 계약과 동일한지 **회귀 테스트**로 고정 (§3.1).
